# 模块规划与边界

## 1. 文档目标

定义仓库结构、后端模块、前端模块、组件化规则、依赖方向与大文件治理标准。

## 2. Workspace 结构

当前主结构已冻结为：

- `services/`：二进制入口
- `crates/`：后端分层能力
- `apps/`：Admin 与 Portal 产品
- `console/`：兼容与过渡外壳
- `bin/`：安装、构建、启动、停止、服务注册入口
- `scripts/`：开发辅助与专项脚本
- `docs/`：文档与参考

## 4. 服务边界

- `gateway-service`：只承担数据平面与兼容入口，不承担后台治理
- `admin-api-service`：只承担治理、配置、运营、诊断，不承担热路径推理
- `portal-api-service`：只承担客户自助与账户能力，不直连 Provider 执行细节
- `router-web-service`：只承担静态站点、公网入口与反向代理，不承载复杂业务规则
- `router-product-service`：只做产品装配与统一交付，不吞并内部边界

## 6. 组件化标准

- 页面容器负责装配，不直接写复杂业务规则
- API client、状态管理、展示组件必须分离
- 公共表格、过滤器、详情面板、状态标签、表单片段必须复用
- 业务模块按“一个工作区一个主职责”拆分，不把所有页面堆到同一包
- 网关路由、管理接口、门户接口都必须围绕能力家族拆分，而不是堆积成巨型入口文件

## 7. 依赖方向与治理规则

- `contract` 不依赖 `app`
- `domain` 不依赖 `interface`
- `app` 依赖 `domain` 与抽象契约
- `interface` 依赖 `app`
- `storage/provider/runtime` 依赖 `domain` 与契约，但不反向支配 `interface`
- `services` 只做装配

大文件治理规则：

- 路由文件、测试文件、页面文件超过可维护上限时，优先按能力家族拆分
- 新增能力不得把旧的“超大 `lib.rs` / 巨型 handler / 巨型 test”再做大

## 8. 评估标准

- 一个模块是否只有一个主职责
- 新增能力是否只影响有限边界
- 前端页面是否可通过包结构快速定位业务归属
- 后端代码是否仍保持 `interface -> app -> domain -> infra` 的单向理解路径
