# 高性能与容量规划设计

> 日期：2026-04-07
> 范围：Gateway 热路径、流式输出、缓存、存储、热点治理、压测门槛

## 1. 文档目标

本文档定义 `sdkwork-api-router` 的性能工程基线、容量规划原则、热点治理策略和发布门槛，确保平台能够支撑高并发 API Router 场景，而不是只在功能层面“可用”。

## 2. 当前可验证事实

### 2.1 当前已具备性能工程基础

当前仓库可验证的性能相关基础包括：

- Rust + Axum 作为数据平面主栈
- `stateful standalone` 与 `stateless runtime` 两种执行形态
- `deterministic_priority`、`weighted_random`、`slo_aware`、`geo_affinity` 等路由策略
- `route decision cache`、`capability catalog cache`
- `memory / Redis` 双缓存后端
- `ProviderHealthSnapshot` 与 recovery probe 协调
- 分布式锁与运行时状态协同
- request id、metrics、tracing、decision log 观测基线

### 2.2 当前性能口径必须拆分两类执行面

当前性能设计不能只写成一套统一结论，至少要拆成：

- `stateful standalone`
  关注鉴权、配额、路由、计费、决策日志、fallback、usage finalize 的平台开销
- `stateless runtime`
  关注 relay、协议转换、连接复用、背压和流式稳定性的轻量开销

这两类执行面共享代码基础，但不共享同一份容量假设。

## 3. 性能目标口径

平台性能目标必须拆成三层：

1. 网关自身开销
2. 端到端体验
3. 系统吞吐

### 3.1 网关自身开销

必须能单独回答：

- Admission 开销多大
- Routing Decision 开销多大
- 平台额外 TTFT 开销多大
- fallback 与 usage finalize 增加了多少尾延迟

### 3.2 端到端体验

必须关注：

- 非流式响应延迟
- TTFT
- 流式完成率
- 中途 stall / 中断误判率

### 3.3 系统吞吐

必须关注：

- 多租户并发
- 多 Provider 并发
- 多协议族并发
- 热点模型与热点项目的隔离能力

## 4. 建议核心指标

- `Admission p95`
- `Routing Decision p95`
- `Gateway Extra TTFT p95`
- `Streaming Completion Success Rate`
- `Provider Timeout Rate`
- `Fallback Success Rate`
- `Quota Check Latency`
- `Usage Finalization Latency`

## 5. 建议性能门槛

以下是 go-live 目标，不等于当前仓库已附带冻结压测报告：

| 指标 | 目标 |
|---|---|
| Admission + 鉴权 + 配额校验 p95 | 小于 `20ms` |
| Routing Decision p95 | 小于 `10ms` |
| 平台额外 TTFT 开销 p95 | 小于 `50ms` |
| 控制面关键读接口 p95 | 小于 `300ms` |
| 流式中途中断误判率 | 极低且可观测 |

## 6. 热路径设计原则

### 6.1 减少数据库放大

热路径必须避免：

- 重复加载相同配置
- 无缓存的全表查询
- 把报表型 SQL 放入请求主链

### 6.2 路由轻量化

路由主链必须优先使用：

- 预编译策略
- 健康快照
- 热点配置缓存

而不是对每个请求执行重型探测与重型拼装。

### 6.3 流式零拷贝倾向

流式链路必须尽量避免：

- 全量缓冲
- 二次序列化
- 大对象一次性入内存

### 6.4 异步与后台任务隔离

- 长耗时结算
- 对账
- 归档
- 大对象处理

不得侵入同步热路径。

## 7. 缓存与容量设计

### 7.1 第一优先级缓存

- 路由决策缓存
- 能力目录缓存
- Provider 健康快照缓存
- 分布式锁辅助状态

### 7.2 增强型缓存

以下属于下一阶段增强方向，不应误写为当前已全面上线：

- prompt cache
- 结果元数据缓存
- 热门模型 capability projection
- 热点项目配置缓存

## 8. 存储容量规划

### 8.1 PostgreSQL

主要承载：

- 配置真值
- usage / billing / audit
- route decision log
- runtime / rollout 状态

治理要求：

- 明细表与配置表分开看待
- 时间序列明细表要有归档与清理策略
- dashboard 聚合要预留离线聚合或物化视图路径

### 8.2 Redis

主要承载：

- 缓存
- 分布式锁
- 热点状态

治理要求：

- 以键空间与 TTL 为核心治理对象
- 避免把大对象长期塞入 Redis

## 9. 热点与多租户容量治理

### 9.1 配额保护

- project 级速率限制
- API key group 级并发限制
- provider / model 级熔断与降级

### 9.2 热点隔离

必须能识别：

- 热点模型
- 热点 Provider
- 热点项目
- 热点协议族

并能执行：

- 独立缓存
- 独立配额
- 独立权重
- 必要时独立 Cell

## 10. 压测与发布门槛

当前仓库中尚未附带统一冻结的性能基准报告，因此压测必须作为显式发布门槛，而不是被默认成“已完成”。

发布前压测至少要覆盖：

- `stateful standalone` 与 `stateless runtime` 分开压测
- 非流式文本请求
- 流式文本请求
- 图片 / 音频 / 视频类路由
- 高并发鉴权与配额校验
- 失败、超时与 fallback 场景

## 11. 设计红线

- 禁止把 dashboard 查询放进数据平面热路径。
- 禁止无上限缓冲流式结果。
- 禁止用“加机器”代替边界治理。
- 禁止在没有基线指标时宣称高性能。
- 禁止把 stateful 的容量数据直接外推到 stateless。

## 12. 评估标准

### 12.1 性能评估

- 网关纯开销是否在目标范围内。
- 流式与非流式是否都具备稳定表现。
- 是否分别给出 `stateful standalone` 与 `stateless runtime` 的结论。

### 12.2 容量评估

- 是否识别并隔离热点租户、热点模型、热点 Provider。
- 是否具备数据库、缓存、网关三层容量规划。
- 是否把控制平面依赖与轻量 relay 路径分开建模。

### 12.3 发布评估

- 每次发布是否都有压测对比与性能退化分析。
