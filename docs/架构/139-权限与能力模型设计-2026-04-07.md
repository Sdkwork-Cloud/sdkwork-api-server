# 权限与能力模型设计

> 日期：2026-04-07
> 范围：主体、资源、动作、能力命名、默认角色、解析顺序、紧急控制

## 1. 文档目标

本文档定义 `sdkwork-api-router` 的统一授权语言，确保 Admin、Portal、Gateway、Runtime 共用同一套“主体-资源-动作-能力”模型，避免出现页面、接口、运行时各自维护一套权限语义。

## 2. 当前可验证事实

### 2.1 当前已存在三条主身份边界

当前可验证的主身份边界包括：

- Admin JWT
- Portal JWT
- Gateway API Key

### 2.2 当前已存在运行时与服务主体

结合扩展运行时、系统服务和脚本托管路径，当前权限模型至少还需要覆盖：

- extension runtime
- service subject

这不是额外设计，而是由扩展安装、实例、runtime status、reload、rollout 等治理对象倒逼出来的现实需求。

## 3. 主体模型

建议固定以下主体族：

- `platform-admin`
- `ops-admin`
- `tenant-owner`
- `project-operator`
- `portal-developer`
- `gateway-api-key`
- `extension-runtime`
- `service-subject`

主体模型要求：

- 主体身份来源清晰
- 主体归属的 tenant / project 边界清晰
- 高风险主体可被紧急禁用

## 4. 资源模型

资源至少分为四类：

### 4.1 身份与工作区

- `tenant`
- `project`
- `user`
- `api-key`
- `api-key-group`

### 4.2 目录与路由

- `provider`
- `channel`
- `model`
- `routing-policy`
- `routing-profile`
- `routing-preference`

### 4.3 商业化

- `quota`
- `billing-ledger`
- `settlement`
- `order`
- `payment`
- `pricing-plan`

### 4.4 平台治理

- `extension`
- `runtime`
- `release`
- `diagnostic`
- `secret`
- `feature-flag`

## 5. 能力命名标准

能力建议采用 `domain.resource.action` 口径，例如：

- `gateway.chat.generate`
- `gateway.image.generate`
- `gateway.audio.transcribe`
- `gateway.video.generate`
- `gateway.music.generate`
- `routing.read`
- `routing.write`
- `billing.read`
- `billing.write`
- `extension.install`
- `extension.execute`
- `release.manage`
- `diagnostic.read`

说明：

- 能力命名必须稳定、可审计、可映射到角色。
- 协议差异不得改变能力语义。
- 文本、图片、语音、视频、音乐等能力家族都要纳入命名体系；具体执行真值仍以兼容矩阵和实际实现为准。

## 6. 默认角色关系

- `platform-admin`：平台级全局治理
- `ops-admin`：运行、诊断、发布、回滚、容量治理
- `tenant-owner`：租户级目录、账务、成员、项目治理
- `project-operator`：项目级路由、模型、配额、密钥治理
- `portal-developer`：自助使用、密钥与工作区查看
- `gateway-api-key`：只允许执行被授予的网关能力
- `extension-runtime`：只允许访问被授权的执行与健康检查接口
- `service-subject`：只允许执行系统内部编排职责

## 7. 授权解析顺序

授权解析顺序必须固定为：

1. 识别主体
2. 识别 tenant / project 边界
3. 识别资源归属
4. 识别动作与能力
5. 叠加角色、策略与显式禁用项
6. 形成授权结果并审计

## 8. 高风险能力与紧急控制

以下能力默认为高风险能力：

- 发布与回滚
- 密钥查看与切换
- 扩展安装与执行
- 生产级诊断
- 定价、账务、结算变更
- kill switch 与大范围流量切换

要求：

- 高风险能力必须支持 kill switch
- 高风险能力必须纳入审计
- 高风险能力不得只靠前端按钮控制

## 9. 演进规则

- 新能力先进入能力注册表，再开放给角色。
- 新协议族只能扩展能力家族，不能私下创造第二套权限命名。
- 新运行形态必须复用同一套授权语言，不能用“本地例外”“脚本例外”“扩展例外”绕过模型。

## 10. 设计红线

- 禁止页面、接口、运行时各自维护一套权限语义。
- 禁止把 tenant / project 边界留给前端自行解释。
- 禁止用资源显示过滤替代服务端授权。
- 禁止在没有 kill switch 和审计时开放高风险能力。

## 11. 评估标准

- 是否能用统一能力语言覆盖网关、控制面、门户、运行时。
- 是否能避免“页面能点、接口不该放”的越权问题。
- 是否能支持新协议、新能力家族、新角色的平滑演进。
