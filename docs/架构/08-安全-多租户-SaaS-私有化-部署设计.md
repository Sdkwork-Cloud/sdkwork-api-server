# 安全-多租户-SaaS-私有化-部署设计

## 1. 文档目标

定义认证、授权、密钥保护、多租户隔离、SaaS 与私有化部署安全标准。

## 2. 安全总原则

- 默认最小权限
- 默认租户隔离
- 默认密钥不明文暴露
- 默认审计可追踪
- 默认发布可回滚

## 3. 身份与认证模型

当前主体至少包括：

- `AdminUser`
- `PortalUser`
- `GatewayApiKey`
- `Service/Runtime Subject`

当前认证边界至少包括：

- Admin JWT
- Portal JWT
- Gateway Bearer Token
- 协议兼容输入：`x-api-key`、`x-goog-api-key`、`?key=`

## 4. 授权模型

授权必须同时基于：

- 角色
- 能力
- 租户边界
- 项目边界
- 资源归属

更细能力模型见 [139-权限与能力模型设计-2026-04-07](./139-权限与能力模型设计-2026-04-07.md)。

## 5. 密钥与凭据保护

- Provider 凭据与系统密钥必须走 Secret backend
- 页面、日志、错误对象不得直接暴露密钥原文
- 凭据轮换、停用、审计必须独立于业务页面实现

## 6. 多租户隔离

- `tenant` 是顶层边界
- `project` 是执行边界
- `api key`、`routing`、`quota`、`billing`、`settlement` 都必须保留租户与项目归属
- 任何共享缓存都不得破坏租户边界

## 7. SaaS 部署标准

- 支持共享 Cell 与 Dedicated Cell
- 数据平面与控制平面可独立扩缩容
- 共享环境必须强化租户隔离、速率限制、审计与故障域控制

## 8. 私有化部署标准

- 支持浏览器优先与产品宿主两类交付
- 支持 SQLite 轻量交付与 PostgreSQL/Redis 生产交付
- 支持离线或受限网络环境下的安装、升级、回滚
- 私有化环境必须保留与 SaaS 一致的身份、审计、发布门禁逻辑

## 9. 备份、恢复与运维安全

- 配置、账本、结算、凭据、扩展安装信息都必须具备备份与恢复路径
- 管理接口、运维接口、发布接口必须具备更严格的权限与审计
- 诊断能力必须最小暴露，不能为了排障而扩大长期风险面

## 10. 设计红线

- 禁止跨租户共享密钥与账本事实
- 禁止在前端、日志、错误消息中输出凭据原文
- 禁止把发布权限、配置权限、生产读写权限混为一体
- 禁止私有化交付绕过同一套权限和审计标准

## 11. 评估标准

- 是否能清楚回答“谁能看、谁能改、谁能发、谁能回滚”
- 是否能同时满足共享 SaaS 与私有化交付的安全约束
- 是否能在不牺牲热路径性能的前提下保证最小权限与可追责
