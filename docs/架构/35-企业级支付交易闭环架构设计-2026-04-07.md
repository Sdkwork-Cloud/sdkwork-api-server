# 企业级支付交易闭环架构设计

> 日期：2026-04-07
> 范围：`sdkwork-api-router`、`sdkwork-router-admin`
> 目标：在现有 commerce、billing、marketing 基础上补齐真实支付接入、支付方式配置、交易内核、webhook 安全、退款与对账，形成可部署的交易闭环。

## 1. 目标与边界

本次改造不重做现有 commerce 域模型，而是在以下既有实体基础上补齐闭环：

- `CommerceOrderRecord`
- `PaymentMethodRecord`
- `PaymentMethodCredentialBindingRecord`
- `CommercePaymentAttemptRecord`
- `CommercePaymentEventRecord`
- `CommerceWebhookInboxRecord`
- `CommerceWebhookDeliveryAttemptRecord`
- `CommerceRefundRecord`
- `CommerceReconciliationRunRecord`
- `CommerceReconciliationItemRecord`

本次交付覆盖：

- 支付方式管理与配置
- Portal 发起真实支付尝试
- Stripe 真实渠道接入
- webhook 验签、幂等、重放防护、失败重试
- 订单结算状态推进
- 退款与部分退款
- 对账与财务收口
- `sdkwork-router-admin` 的支付运维管理界面

本次不覆盖：

- 支付宝、微信支付的真实对接
- 自动化跑批调度器
- 多币种汇率结算

## 2. 核心原则

### 2.1 高内聚低耦合

交易链路拆分为独立模块：

- `payment_method`: 支付方式与凭据绑定治理
- `payment_attempt`: 发起支付尝试、幂等键、checkout handoff
- `provider/stripe`: Stripe API 适配
- `webhook`: provider webhook 解析、验签、重放防护、入箱与重试
- `refund`: 主动退款、部分退款、退款状态推进
- `reconciliation`: 对账运行与差异项生成
- `admin_finance`: 管理后台读取 payment attempts、refunds、reconciliation、webhook inbox

### 2.2 订单与支付尝试分离

订单是业务意图，支付尝试是支付执行事实。

- 一个订单可对应多个 `PaymentAttempt`
- 订单保存 `latest_payment_attempt_id`
- 订单只保存最终业务状态与结算状态
- provider 侧细节落到 `PaymentAttempt` 和 `WebhookInbox`

### 2.3 webhook 先入箱再处理

所有 provider 回调都先进入 `ai_commerce_webhook_inbox`，再进入处理器：

- 防止直接在 HTTP handler 中做不可恢复的状态修改
- 提供失败重放与死信重试能力
- 可被 admin 后台审计

### 2.4 结算与账户收口分层

- commerce 负责订单、支付、退款、对账
- billing 负责账户余额、权益、请求结算、账本收口
- commerce 通过 `CommercialBillingAdminKernel` 与 billing 同步

## 3. 产品与运营流程

### 3.1 下单

1. 用户在 portal 选择订阅 / 充值 / 自定义充值 / 优惠券兑换
2. `preview quote` 生成价格快照
3. `submit order` 写入订单与价格快照
4. 0 元或优惠券兑换订单直接履约
5. 需支付订单进入 `pending_payment`

### 3.2 发起支付

1. 前端读取可用支付方式
2. 选择支付方式
3. 创建 `PaymentAttempt`
4. 调用 provider 创建 Stripe Checkout Session
5. 返回 checkout URL 或二维码

### 3.3 支付成功

1. Stripe webhook 回调
2. 签名验证与时间窗校验
3. 按 `provider_event_id` 和 `dedupe_key` 入箱
4. 处理器将 webhook 映射为 `PaymentEvent`
5. 更新 `PaymentAttempt`
6. 推进订单 `settlement_status`
7. 触发 billing 收口与 reconciliation checkpoint

### 3.4 退款

1. admin 发起退款请求
2. 校验订单可退余额
3. 创建 `CommerceRefundRecord`
4. 调用 Stripe Refund API
5. webhook 或同步响应推进退款状态
6. 更新订单 `refunded_amount_minor` / `refundable_amount_minor`
7. 触发 billing 退款收口

### 3.5 对账

1. admin 发起对账运行
2. 按 provider + payment method + 时间窗口扫描本地 payment attempts / refunds
3. 汇总 provider 参考数据
4. 生成 reconciliation run 和 items
5. 差异项由运营处理

## 4. 领域模型设计

### 4.1 PaymentMethod

`PaymentMethodRecord` 负责表达一个可被选中的支付方式：

- 基本标识：`payment_method_id`、`display_name`
- 渠道信息：`provider`、`channel`、`mode`
- 治理信息：`enabled`、`sort_order`
- 能力：`capability_codes`
- 适用范围：`supported_currency_codes`、`supported_country_codes`、`supported_order_kinds`
- 回调策略：`callback_strategy`、`webhook_path`
- 安全参数：`webhook_tolerance_seconds`、`replay_window_seconds`、`max_retry_count`
- 运行配置：`config_json`

`config_json` 使用 JSON schema 风格的动态配置，首版重点支持 Stripe：

```json
{
  "checkout_mode": "payment",
  "success_url": "https://portal.example.com/payments/success?session_id={CHECKOUT_SESSION_ID}",
  "cancel_url": "https://portal.example.com/payments/cancel",
  "statement_descriptor": "SDKWORK",
  "allow_promotion_codes": false,
  "capture_method": "automatic",
  "webhook_event_types": [
    "checkout.session.completed",
    "checkout.session.expired",
    "payment_intent.payment_failed",
    "charge.refunded"
  ]
}
```

### 4.2 PaymentMethodCredentialBinding

一个支付方式可以绑定多种用途的密钥：

- `api_secret`
- `publishable_key`
- `webhook_secret`
- `reconciliation_secret`

凭据不直接明文落支付模块，而是引用加密凭据中心：

- `credential_tenant_id`
- `credential_provider_id`
- `credential_key_reference`

### 4.3 Order

订单新增并统一使用以下交易字段：

- `currency_code`
- `pricing_plan_id`
- `pricing_plan_version`
- `pricing_snapshot_json`
- `payment_method_id`
- `latest_payment_attempt_id`
- `settlement_status`
- `refundable_amount_minor`
- `refunded_amount_minor`

### 4.4 PaymentAttempt 状态机

`CommercePaymentAttemptRecord.status` 规范为：

- `created`
- `provider_pending`
- `requires_action`
- `processing`
- `succeeded`
- `failed`
- `canceled`
- `expired`
- `partially_refunded`
- `refunded`

### 4.5 Settlement 状态机

`CommerceOrderRecord.settlement_status` 规范为：

- `pending`
- `requires_action`
- `authorized`
- `settled`
- `failed`
- `canceled`
- `refunded`
- `partially_refunded`
- `not_required`

## 6. Gateway / Portal API 设计

### 6.1 Portal API

新增：

- `POST /portal/commerce/orders/{order_id}/payment-attempts`
  - 作用：创建真实支付尝试

## 7. Stripe 适配设计

### 7.1 选择 Stripe 的原因

- 已存在 `stripe` provider 常量
- 国际化能力成熟
- webhook 与退款闭环标准化
- 文档稳定，便于扩展

### 7.2 Stripe Credential 绑定

要求在 payment method 上至少绑定：

- `api_secret`
- `webhook_secret`

可选：

- `publishable_key`

### 7.3 创建支付尝试

首版使用 Stripe Checkout Session：

- 模式：`payment`
- 货币：从订单 `currency_code`
- 金额：从订单 `payable_price_cents`
- metadata：
  - `order_id`
  - `project_id`
  - `user_id`
  - `payment_attempt_id`
  - `payment_method_id`

### 7.4 webhook 事件映射

映射规则：

- `checkout.session.completed` -> `settled`
- `checkout.session.expired` -> `failed`
- `payment_intent.payment_failed` -> `failed`
- `charge.refunded` -> `refunded`

### 7.5 退款

首版调用 Stripe Refund API：

- 支持全额退款
- 支持部分退款
- 幂等键使用 `refund_id` 或业务退款幂等键

## 8. webhook 安全设计

### 8.1 验签

使用 Stripe `Stripe-Signature`：

- 提取 `t`
- 提取多个 `v1`
- 使用 endpoint secret 计算 HMAC-SHA256
- 使用常量时间比较

### 8.2 时间容忍

校验：

- `abs(now - t) <= webhook_tolerance_seconds`

### 8.3 重放防护

使用：

- `provider_event_id`
- `dedupe_key = provider:event_id`
- `replay_window_seconds`

同一个 `dedupe_key` 进入 inbox 只更新接收时间，不重复推进业务状态。

### 8.4 幂等

幂等边界：

- webhook 入箱幂等
- payment event 幂等
- settlement 状态推进幂等
- refund 落库与 billing 收口幂等

### 8.5 失败重试与死信

当 webhook 处理失败时：

- retry_count + 1
- 计算 `next_retry_at_ms`
- 写 `CommerceWebhookDeliveryAttemptRecord`
- 超过 `max_retry_count` 标记为 `dead_letter`

## 9. 缓存设计

首版采用轻量缓存，避免交易主路径依赖不一致缓存：

- 支付方式列表缓存 30 秒
- PaymentMethod 配置缓存 30 秒
- 凭据绑定缓存 30 秒
- webhook 幂等不依赖缓存，只依赖数据库唯一键

缓存 key：

- `commerce:payment_methods:all`
- `commerce:payment_method:{payment_method_id}`
- `commerce:payment_bindings:{payment_method_id}`

## 10. 财务收口设计

### 10.1 订单结算成功

- 更新 order / attempt / payment event
- 如为充值类订单，调用 billing 发放权益
- 更新 reconciliation checkpoint

### 10.2 退款成功

- 更新 refund / attempt / order
- 调用 billing 退款逻辑
- 更新 reconciliation checkpoint

### 10.3 差异对账

对账按支付与退款双轨比较：

- 本地有、provider 无
- provider 有、本地无
- 金额不一致
- 状态不一致

## 13. 部署要求

环境变量至少包括：

- `SDKWORK_PAYMENT_STRIPE_API_BASE_URL`
- `SDKWORK_PORTAL_BASE_URL`
- `SDKWORK_ADMIN_BASE_URL`
- `SDKWORK_CREDENTIAL_MASTER_KEY`

支付方式必须完成：

- payment method record 配置
- `api_secret` credential binding
- `webhook_secret` credential binding
- Stripe dashboard webhook endpoint 配置

## 14. 完成标准

满足以下条件视为交易闭环完成：

- portal 能创建真实 PaymentAttempt
- Stripe Checkout 能真实跳转支付
- webhook 能验签并推进订单结算
- 订单、attempt、payment event、refund 状态一致
- admin 能管理 payment methods 与 credentials
- admin 能查看 attempts / webhook / refund / reconciliation
- Postgres 与 SQLite 都支持完整交易存储模型
