# 多Cell多Region与灾备演进设计

> 日期：2026-04-07
> 范围：Cell、Region、放置策略、升级回滚、灾备、RTO/RPO

## 1. 文档目标

本文档定义 `sdkwork-api-router` 从单机与单 Cell 形态，演进到多 Cell、多 Region 与灾备治理的目标路径。

本文档强调：当前可验证事实主要集中在单机、本地治理与单 Cell 级能力；多 Cell、多 Region、灾备目标是演进方向，不应误写成当前已全面落地。

## 2. 当前可验证事实

当前可验证事实包括：

- 当前主交付形态仍以单机、本地与单 Cell 组织为主
- `stateful standalone` 与 `stateless runtime` 已形成明确区分
- 路由策略已包含 `geo_affinity`
- 运行时已具备 rollout、health snapshot、runtime status 等协同基础

这些事实说明平台已具备“继续向多 Cell 演进”的结构基础，但还不能直接宣称已经完成多 Region 灾备产品化。

## 3. 演进路径

平台演进顺序固定为：

1. 单机与本地可运行
2. 单 Cell 生产可运行
3. 多 Cell 可隔离
4. 多 Region 可扩展
5. 灾备与跨 Region 升级可控

## 4. Cell模型

`Cell` 是最小交付、故障和回滚单元，至少应包含：

- gateway
- admin / portal API
- web / product host
- 主存储与缓存
- secret backend
- 发布与回滚资产

建议 Cell 形态：

- `shared cell`
- `dedicated cell`
- `private cell`

## 5. Region模型

`Region` 负责地理扩展与灾备，不应打破 Cell 边界。原则是：

- 先保证 Cell 内一致性
- 再做 Region 间复制、切换与流量编排
- 控制面、数据平面、对象存储和审计证据分别规划

## 6. 数据与服务放置原则

- gateway 可按 Region 水平扩展
- 控制面可按 Cell 独立部署，也可集中治理后分发
- 账本、结算、审计证据必须明确主写区域与灾备策略
- 大对象与媒体优先走对象存储分层

## 7. 升级、回滚与故障切换

- 升级必须按 Cell 分批灰度，不得全网一次切换
- 回滚必须优先保护账本、结算、配置、密钥边界
- 长连接、流式任务、异步任务必须定义 drain 与中断策略
- Region 切换必须保证协议、路由、计费、审计语义不失真

## 8. RTO / RPO口径

以下指标是目标建议，不应直接写成当前已验证承诺：

- 共享 SaaS：`RTO <= 30 分钟`、`RPO <= 5 分钟`
- Dedicated / Private：按合同与交付等级提高要求

## 9. 设计红线

- 禁止把单 Cell 成功经验直接外推成多 Region 成熟结论。
- 禁止以“入口可用”替代“账务、审计、配置真值可恢复”。
- 禁止跳过 Cell 粒度的灰度与回滚边界，直接做全局切换。
- 禁止在没有主写区域和复制策略的前提下宣称灾备达标。

## 10. 评估标准

- 是否能把 Cell 当作独立交付、故障、回滚单元。
- 是否能在多 Region 下保持协议、路由、计费、审计语义一致。
- 是否能在故障切换时保护关键业务事实，而不只是恢复 API 入口。
