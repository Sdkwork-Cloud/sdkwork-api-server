# 产品设计与需求范围

## 1. 文档目标

定义 `sdkwork-api-router` 的产品定位、用户对象、能力边界、当前事实、目标能力与验收标准。

## 2. 产品定位

`sdkwork-api-router` 是企业级 AI API Router 平台，不是单厂商代理，也不是只做 OpenAI 兼容的薄转发层。它同时承担四类职责：

- 对外提供多协议、多模态 API 入口
- 对内统一承载路由、配额、计费、审计与扩展运行时
- 对运维提供 Admin 控制平面
- 对客户与开发者提供 Portal 自助门户与商业化入口

行业对标对象是 `OpenRouter`、`new-api` 一类平台，但目标标准更高：治理更强、边界更清晰、发布更可控、私有化更可交付。

## 3. 目标用户与场景

- 开发者：希望通过统一入口接入 OpenAI、Anthropic、Gemini 及后续更多协议与模型生态
- 平台团队：希望统一治理 `provider / channel / model / routing / quota / usage / billing`
- 运营团队：希望通过 Admin 处理上游接入、容量、价格、故障、活动、订单、结算
- 企业客户：希望通过 Portal 完成注册、登录、密钥管理、用量查看、充值、结算、模型与路由偏好管理

核心场景包括：

- 通用 LLM 文本生成与工具调用
- 图像、音频、视频、文件、向量、评测、实时会话等多模态 API
- 多租户 SaaS、自托管私有化、浏览器交付、桌面交付
- 高并发 gateway server 与对应 admin、portal、trade 子域协同

## 4. 当前已验证的产品事实

- 后端运行面已包含 `gateway-service`、`admin-api-service`、`portal-api-service`、`router-web-service`、`router-product-service`
- 前端运行面已包含 `sdkwork-router-admin`、`sdkwork-router-portal`；`console/` 为兼容与过渡外壳
- Gateway 已形成 `stateful standalone` 与 `stateless runtime` 双运行模式
- 协议兼容面已覆盖 `OpenAI Compatible`、`Anthropic Messages Compatible`、`Gemini Compatible`
- 数据平面已覆盖 chat、responses、embeddings、files、uploads、audio、images、assistants、threads、conversations、vector stores、batches、evals、videos、realtime sessions、containers 等高价值家族
- `music` 已按统一资源化 API 家族口径纳入兼容文档；执行真值以兼容矩阵与参考文档为准
- 内建 Provider 已明确包含 `openai`、`openrouter`、`ollama`
- 扩展运行时已形成 `builtin / connector / native-dynamic` 三类接入模型
- 控制平面已覆盖 tenants、projects、api keys、api-key-groups、providers、channels、models、credentials、extensions、routing、usage、billing、quota 等对象
- Portal 工作区已覆盖 workspace、dashboard、usage、billing、api keys、routing、credits、recharge、settlements、models、docs、downloads 等入口
- 存储已覆盖 `SQLite`、`PostgreSQL`；缓存已覆盖 `memory`、`Redis`；secret backend 已覆盖本地与系统密钥后端

## 5. 目标能力边界

目标态下，平台必须同时满足以下边界：

- 统一接入更多上游协议与模型生态，但不破坏统一路由和计费主链路
- 统一支持 LLM、图像、音频、视频、音乐与未来新增能力族
- 统一支持高并发 gateway、运营后台、客户门户、交易系统与私有化交付
- 统一支持 SaaS、共享 Cell、Dedicated Cell、离线交付、桌面与浏览器混合场景
- 统一支持商业化闭环：价格、配额、账本、订单、支付、结算、对账、退款、审计

## 6. 非目标与红线

- 不自研通用推理引擎
- 不为每个协议或每个 Provider 再造一套独立网关
- 不允许把业务规则散落在 HTTP handler、前端页面或脚本里
- 不允许依赖人工运维动作维持生产发布
- 不允许以大文件堆积替代模块边界设计

## 7. 成功标准

- 一个统一产品面同时支持 `gateway / admin / portal / web host / product host`
- 多协议入口共享同一条身份、路由、配额、计费、审计主链
- 平台可在本地、测试、生产、私有化四类环境稳定交付
- 新增 Provider、协议、能力家族时，改动受限在明确边界内
- 商业化域与网关热路径分层清楚，既可独立演进，又能形成闭环
- 文档、兼容矩阵、代码结构、发布流程长期保持一致

## 8. 评估标准

### 8.1 产品完整性

- 是否完整覆盖开发者入口、运维入口、客户入口、商业化入口
- 是否同时覆盖文本、多模态、交易、私有化四类核心场景

### 8.2 架构适配性

- 是否能同时支持高并发网关与后台治理，而不是偏科到单一方向
- 是否能在不破坏现有主链路的前提下扩展更多协议与能力

### 8.3 可交付性

- 是否具备清晰的安装、部署、发布、回滚与验收标准
- 是否能把当前事实与目标态明确分开，避免误导实现与运营
