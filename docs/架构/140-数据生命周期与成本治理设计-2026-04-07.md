# 数据生命周期与成本治理设计

> 日期：2026-04-07
> 范围：主数据、执行事实、媒体对象、运行时数据、缓存、归档、删除、成本

## 1. 文档目标

本文档定义 `sdkwork-api-router` 的数据分类、保留周期、冷热分层、归档、删除和成本治理边界，确保平台在多模态扩展后不会因为日志、媒体、结算和审计对象持续膨胀而失控。

## 2. 当前可验证事实

当前仓库中已经可以验证以下高价值数据对象：

- 工作区与身份对象：tenant、project、API key、API key group
- 路由与配置对象：routing policy、routing profile、routing preference
- 商业化对象：billing events、request settlements、pricing plans、pricing rates
- 路由证据对象：`RoutingDecisionLog`、`ProviderHealthSnapshot`
- 运行时治理对象：extension installations、instances、runtime rollouts、config rollouts

这说明平台当前并不是只有“请求转发数据”，而是已经同时拥有主数据、执行事实、治理事实和商业事实。

## 3. 数据分类

平台数据建议固定分为五类：

### 3.1 主数据

- tenant
- project
- user
- API key / API key group
- provider / model / routing / pricing 真值

### 3.2 执行事实

- request
- usage
- billing event
- settlement
- route decision
- health snapshot

### 3.3 媒体与大对象

- 文件
- 上传对象
- 音频
- 视频
- 音乐
- 导出产物

### 3.4 运行时数据

- extension installation
- extension instance
- runtime status
- rollout record
- release record

### 3.5 视图与缓存

- dashboard 视图
- 聚合表
- 热点缓存
- 临时任务中间态

## 4. 生命周期分层

- `Hot`：热查询、热路由、近期账务、近期审计
- `Warm`：运营回溯、客户账单、结算追踪
- `Cold`：长期合规保留、争议处理、审计取证
- `Archive`：低频访问但不可直接删除的数据
- `Ephemeral`：缓存、临时上传片段、任务中间态

## 5. 治理原则

### 5.1 主数据

- 长期保留
- 强审计
- 变更必须可追溯

### 5.2 执行事实

- 按时间、租户、项目、Provider 分区
- 明细保留与聚合保留分离
- 回溯价值与成本价值要分开评估

### 5.3 大对象

- 优先落对象存储或冷层
- 不得长期占用热数据库
- 必须有到期清理和引用追踪策略

### 5.4 视图与缓存

- 可重建
- 不承担长期保留义务
- 优先服务读性能，不承担合规真值

## 6. 成本控制手段

- 分区表与归档表
- TTL 与定时回收
- 冷热分层存储
- 明细保留与聚合保留分离
- 诊断与日志采样、压缩、限流
- 把可重建视图优先做成派生资产

## 7. 删除与合规

删除必须区分：

- 逻辑删除
- 业务失效
- 归档保留
- 法务保留

默认不应直接物理删除的对象包括：

- 账本
- 结算
- 核心审计证据

默认必须具备到期清理策略的对象包括：

- 临时媒体对象
- 缓存
- 中间态任务数据

## 8. 成本度量标准

成本治理至少要持续回答：

- 哪类数据增长最快
- 哪类数据最贵
- 哪类数据必须保留
- 哪类数据可聚合替代明细
- 哪类数据可转冷层或归档

## 9. 设计红线

- 禁止把可重建视图当成长期合规真值。
- 禁止把大对象长期留在热数据库。
- 禁止没有清理策略的临时对象持续堆积。
- 禁止账务、结算、审计对象在无保留规则下直接物理删除。

## 10. 评估标准

- 是否能同时满足性能、成本、审计、客户查询四类目标。
- 是否能明确回答每类数据保留多久、放在哪里、如何恢复。
- 是否能在多模态扩展后避免成本随对象体积失控。
