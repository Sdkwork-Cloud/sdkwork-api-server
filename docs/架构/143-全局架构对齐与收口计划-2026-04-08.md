# 全局架构对齐与收口计划

> 日期：2026-04-08
> 目标：把 `sdkwork-api-router` 的“当前事实、可交付能力、目标态能力”继续收口，避免文档、脚本、实现三者漂移。

## 1. 当前已验证事实

- 平台已具备多运行面：`gateway / admin / portal / web / product runtime`。
- 发布工作流已覆盖 `Windows / Linux / macOS` 与 `x64 / arm64` 目标矩阵。
- 工作区启动、进程监管、发布治理、Portal 商业台账契约测试本轮均通过。
- Windows Rust workspace 校验链路已恢复，默认 target-dir 政策已统一到仓库内受管目录。

## 2. 当前不能夸大的地方

- 多 Cell / 多 Region / 灾备：仍是演进设计，不是当前生产事实。
- SLO 治理：当前有观测资产，不等于已形成发布门禁级 SLO 治理。
- 数据策略：当前仍以 Provider 级事实为主，平台级统一策略未完全落地。
- 三平台安装产物：当前有工作流和脚本契约，不等于已完成全平台实机安装验收。

## 3. 收口原则

1. 当前事实必须能由代码、脚本、测试、命令直接验证。
2. 目标态必须单列，不得混写成“已完成”。
3. 先收口发布、启动、验证链路，再扩大功能面。
4. 先补门禁和证据，再宣称企业级能力。

## 4. 本轮收口项

### 4.1 Windows Rust 目标目录统一

- 策略：`CARGO_TARGET_DIR` 显式传入时尊重；未显式传入时，Windows 默认使用仓库内 `bin/.sdkwork-target-vs2022`。
- 原因：避免受限环境写入 `USERPROFILE` 失败，统一 `check-rust-verification-matrix` 与 `start-dev.ps1` 的行为。
- 验收：`node scripts/check-rust-verification-matrix.mjs --group workspace` 通过。

### 4.2 运行时工具链测试基线与执行环境边界统一

- 测试基线：`bin/tests/router-runtime-tooling.test.mjs` 的 Windows release/install 断言已同步到仓库内受管 target-dir。
- 宿主边界：Node 直接拉起 `powershell.exe` 受宿主能力约束；当前环境不可拉起时，相关 smoke test 应跳过而不是误报产品失败。
- 发布门禁：`scripts/release/run-release-governance-checks.mjs` 已纳入 `release-runtime-tooling-test`，并在 child exec 受限时走契约 fallback。
- 验收：`node --test --experimental-test-isolation=none bin/tests/router-runtime-tooling.test.mjs` 通过。

## 5. 下一阶段对齐顺序

### P0 发布与运行链路

- 为安装产物补 Linux/macOS smoke test。
- 在具备 PowerShell 子进程能力的 Windows 车道补跑被跳过的 PowerShell smoke。
- 将 Release 依赖边界、本地 sibling 依赖阻断、安装后健康检查纳入固定门禁。
- 统一所有 Windows Rust/Tauri/脚本 target-dir 策略。

### P1 治理与合规

- 落平台级数据策略：保留期、缓存、脱敏、审计、租户覆盖。
- 落控制面配置版本化、变更审批、Kill Switch、回滚证据。
- 补高价值审计对象的结构化最小字段要求。

### P1 可观测性与容量

- 把请求链、路由链、运行时链、商业链统一到一套 SLO 口径。
- 为热点租户、配额保护、缓存命中、队列积压建立发布阈值。
- 把 burn-rate 与发布阻断打通。

### P2 高可用与灾备

- 从单 Cell 稳态验收起步。
- 再补 Cell 切片、Region 放置、故障切换、drain、RTO/RPO 演练。
- 演练未稳定前，不对外宣称“多 Region 生产可用”。

## 6. 验收口径

- 代码：关键路径无构建失败、无脚本策略冲突。
- 测试：启动、发布、治理、Portal 契约、Rust workspace 校验全绿。
- 文档：现状与目标态分离，不夸大。
- 交付：至少一轮真实安装/启动/回滚烟测可复现。
