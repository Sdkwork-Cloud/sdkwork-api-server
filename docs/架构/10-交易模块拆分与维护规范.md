# 交易模块拆分与维护规范

## 1. 目标

本规范用于约束 `sdkwork-api-router` 交易体系代码的模块边界，解决当前交易相关实现长期集中在超大 `lib.rs` / `index.tsx` / `index.ts` 文件中的问题，确保后续支付方式、支付尝试、Webhook、退款、对账、安全策略与管理端能力可以持续演进，而不会继续形成“大泥球”。

目标有四个：

1. 高内聚：每个模块只负责一个稳定职责。
2. 低耦合：跨模块通过明确 DTO / trait / facade 交互，避免相互穿透。
3. 易定位：出现问题时可以快速定位到“订单”“支付方式”“Webhook”“退款”“对账”等明确模块。
4. 易扩展：后续新增 Stripe 之外的支付渠道时，不再修改多个巨型文件。

## 2. 当前交易相关大文件现状

交易链路主要存在以下高风险大文件：

1. `crates/sdkwork-api-app-commerce/src/lib.rs`
2. `crates/sdkwork-api-interface-admin/src/lib.rs`
3. `crates/sdkwork-api-interface-portal/src/lib.rs`
4. `crates/sdkwork-api-storage-sqlite/src/lib.rs`
5. `crates/sdkwork-api-storage-postgres/src/lib.rs`
6. `apps/sdkwork-router-admin/packages/sdkwork-router-admin-commercial/src/index.tsx`
7. `apps/sdkwork-router-admin/packages/sdkwork-router-admin-admin-api/src/index.ts`
8. `apps/sdkwork-router-admin/packages/sdkwork-router-admin-types/src/index.ts`

其中 `sdkwork-api-domain-commerce` 已先完成第一轮拆分，作为交易内核模型层的基线示范。

## 3. 分层拆分原则

### 3.1 Domain 层

Domain 只表达交易事实与状态，不承担 IO、鉴权、HTTP、加密、外部渠道调用逻辑。

推荐模块：

1. `order`
2. `payment_event`
3. `payment_method`
4. `webhook`
5. `refund`
6. `membership`

约束：

1. Domain 结构体允许提供 `new` 与 `with_*` builder。
2. Domain 不依赖存储层和接口层。
3. Domain 中的 JSON 字段只做快照承载，不在此层解析外部渠道协议细节。

### 3.2 App 层

App 层是交易业务编排层，负责规则、渠道接入、幂等、防重放、账务推进。

`sdkwork-api-app-commerce` 必须拆分为以下子模块：

1. `error`
   负责 `CommerceError`、错误映射与公共 Result。
2. `constants`
   负责 provider、channel、usage_kind、状态常量。
3. `types`
   负责 portal/admin 公开 DTO，不放业务流程函数。
4. `catalog`
   负责商品目录、报价、价格快照生成。
5. `coupon`
   负责交易中的营销券解析、预占、确认、回滚与补偿。
6. `order`
   负责下单、订单加载、订单状态推进。
7. `settlement`
   负责订单履约、账务推进、会员与额度侧效果。
8. `payment_method`
   负责支付方式可用性过滤、能力匹配、凭据绑定解析。
9. `payment_attempt`
   负责支付尝试创建、状态推进、幂等重试。
10. `providers/stripe`
    负责 Stripe Checkout / Refund / Reconciliation / Webhook 签名验证。
11. `webhook`
    负责入站 webhook inbox、重放防护、死信重试、事件分发。
12. `refund`
    负责退款规则、退款记录、部分退款策略与安全约束。
13. `reconciliation`
    负责差异对账、结果归档、财务收口状态。

约束：

1. `lib.rs` 仅作为 facade，负责 `mod`、`pub use`、少量组合入口。
2. Stripe 以 provider adapter 实现，不能把 HTTP 调用散落在多个模块。
3. 下单与支付尝试分离，不允许“创建订单时直接混入渠道 IO”。

### 3.3 Storage 层

Storage 层按“领域子域”拆分实现，而不是继续在单个超大存储文件中追加新 SQL。

推荐结构：

1. `commerce_order_store`
2. `commerce_payment_method_store`
3. `commerce_payment_attempt_store`
4. `commerce_webhook_store`
5. `commerce_refund_store`
6. `commerce_reconciliation_store`

约束：

1. 每个模块只维护一组相邻表和 mapper。
2. `run_migrations` 可以保留在主文件，但每个子域的建表语句必须抽成独立函数。
3. `impl SqliteAdminStore` / `impl PostgresAdminStore` 可以分散在多个模块中实现，避免单文件继续膨胀。

### 3.4 Interface 层

接口层按“身份域 + 业务域”拆分，不再把全部 DTO、OpenAPI、handler 放在一个文件。

`sdkwork-api-interface-admin` 拆分建议：

1. `auth`
2. `common`
3. `commerce`
4. `billing`
5. `marketing`
6. `catalog`
7. `routing`
8. `runtime`

`sdkwork-api-interface-portal` 拆分建议：

1. `auth`
2. `workspace`
3. `commerce`
4. `billing`
5. `routing`
6. `jobs`

其中 `commerce` 再拆为：

1. `catalog_routes`
2. `order_routes`
3. `payment_routes`
4. `webhook_routes`
5. `refund_routes`

### 3.5 Admin 前端层

`sdkwork-router-admin-commercial` 必须从单页巨型组件拆成页面容器 + 区块组件 + 表单组件：

1. `CommercialOverviewPage`
2. `PaymentMethodRegistrySection`
3. `PaymentMethodEditorDialog`
4. `PaymentCredentialBindingsCard`
5. `PaymentAttemptAuditSection`
6. `RefundAuditSection`
7. `WebhookInboxSection`
8. `ReconciliationRunSection`
9. `CommercialOrderAuditSection`
10. `shared.ts`

`sdkwork-router-admin-admin-api` 中与 commerce 相关的调用拆到：

1. `commerce.ts`
2. `billing.ts`
3. `marketing.ts`

`sdkwork-router-admin-types` 中与 commerce 相关的类型拆到：

1. `commerce.ts`
2. `billing.ts`
3. `marketing.ts`
4. `index.ts` 仅做 re-export

## 4. 交易主链路的推荐依赖方向

必须遵循以下方向：

1. `domain-commerce` <- `storage-*`
2. `domain-commerce` <- `app-commerce`
3. `app-commerce` <- `interface-admin` / `interface-portal`
4. `admin types` <- `admin api` <- `admin page`

禁止以下反向依赖：

1. interface 反向引用 storage SQL 细节
2. admin page 直接拼装后端 URL 字符串和 DTO
3. provider adapter 直接写到 interface handler 中
4. webhook 验签逻辑散落在 route handler 中

## 5. 交易对象与模块归属

### 5.1 订单域

归属模块：

1. `domain-commerce/order`
2. `app-commerce/order`
3. `storage-*/commerce_order_store`

负责：

1. 订单价格快照
2. 订单支付方式绑定
3. 结算状态推进
4. 可退款金额管理

### 5.2 支付方式域

归属模块：

1. `domain-commerce/payment_method`
2. `app-commerce/payment_method`
3. `storage-*/commerce_payment_method_store`
4. `interface-admin/commerce/payment_method_routes`
5. `admin-commercial/PaymentMethodRegistrySection`

负责：

1. 启停
2. 能力声明
3. 币种/国家/订单类型匹配
4. 回调策略
5. 加密凭据绑定

### 5.3 支付尝试域

归属模块：

1. `domain-commerce/payment_method`
2. `app-commerce/payment_attempt`
3. `app-commerce/providers/stripe`
4. `storage-*/commerce_payment_attempt_store`

负责：

1. 幂等键
2. checkout / payment intent 引用
3. customer action 状态
4. captured / refunded 金额追踪

### 5.4 Webhook 域

归属模块：

1. `domain-commerce/webhook`
2. `app-commerce/webhook`
3. `storage-*/commerce_webhook_store`
4. `interface-portal/commerce/webhook_routes`

负责：

1. provider webhook 入站
2. 验签
3. 幂等
4. 重放防护
5. 死信重试

### 5.5 退款与对账域

归属模块：

1. `domain-commerce/refund`
2. `app-commerce/refund`
3. `app-commerce/reconciliation`
4. `storage-*/commerce_refund_store`
5. `storage-*/commerce_reconciliation_store`

负责：

1. 退款规则
2. 全额退款
3. 部分退款限制与安全策略
4. 差异对账
5. 财务收口状态

## 6. 当前实施顺序

本轮重构和实现严格按以下顺序推进：

1. 完成 `domain-commerce` 拆分
2. 完成 `app-commerce` 模块化，至少把新支付链路从 `lib.rs` 中剥离
3. 在 `storage-sqlite` / `storage-postgres` 中把交易新增对象抽成 commerce 子模块
4. 在 `interface-admin` / `interface-portal` 中抽出 `commerce` 子模块
5. 在 `sdkwork-router-admin-commercial` 中拆出支付方式管理相关组件
6. 继续完成支付闭环实现与统一回归测试

## 7. 验收标准

满足以下标准才算拆分合格：

1. 新增支付逻辑不得继续放回原超大文件。
2. 每个新模块文件长度优先控制在 200-500 行内。
3. 单个模块只承担一种稳定职责。
4. 交易新增功能的入口、存储、接口、前端都能按模块名直达定位。
5. 交易相关跨层依赖保持单向，不出现“接口层持有 SQL 细节”“前端绕过 API client”等反模式。
