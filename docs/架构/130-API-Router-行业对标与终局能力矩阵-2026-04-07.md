# API Router 行业对标与终局能力矩阵

> 日期：2026-04-07
> 范围：`sdkwork-api-router`
> 对标对象：`OpenRouter`、`new-api`

## 1. 文档目标

本文档用于把 `sdkwork-api-router` 放到行业竞争语境中，明确三件事：

1. 当前仓库已经可验证落地了哪些能力。
2. 对标产品在公开层面体现了哪些行业基线。
3. 从“可用的 API 聚合器”演进到“行业领先的 API Router 平台”还需要补齐哪些能力。

本文档口径遵循 [README](./README.md)：

- “当前已具备”仅指仓库、接口文档和服务实现可直接验证的事实。
- “目标状态”仅指下一阶段必须达到的平台标准，不等同于当前已全部生产化完成。

## 2. 行业公开基线

### 2.1 OpenRouter 的公开信号

从公开资料看，OpenRouter 代表的不是“多接几个模型”，而是：

- 统一入口访问多家大模型 Provider
- 请求级 Provider 排序、优先级、fallback 与参数约束
- 成本、延迟、稳定性与数据策略共同参与路由
- `zdr`、data policy、prompt caching、sticky routing 等高阶治理能力

结论：行业领先 Router 的竞争力不只在协议覆盖，更在“数据策略感知、成本感知、性能感知、命中率优化”。

### 2.2 new-api 的公开信号

从公开资料看，`new-api` 更强调“可交付、可运营、可部署”：

- 控制台、统计分析、用户与令牌治理
- 多协议兼容与快速部署
- weighted random、重试、限流、用户级约束
- SQLite / MySQL / PostgreSQL / Docker 友好路径
- 充值、计费、应用接入等平台化交付能力

结论：另一条行业基线是“能交付、能运维、能卖、能管”。

## 3. 当前可验证定位

当前仓库更接近“平台化 API Router”，而不是“轻量兼容代理”。可验证依据包括：

- 多后端服务：`gateway-service`、`admin-api-service`、`portal-api-service`、`router-web-service`、`router-product-service`
- 多前端宿主：`apps/sdkwork-router-admin`、`apps/sdkwork-router-portal`、`console/`
- 双运行形态：`stateful standalone` 与 `stateless runtime`
- 多协议入口：OpenAI、Anthropic、Gemini
- 控制平面与自助门户同时存在

这说明当前主线已经具备平台骨架，但“骨架已形成”不等于“所有治理能力已达到行业终局”。

## 4. 当前已形成的能力骨架

### 4.1 多运行面骨架

当前可直接验证的平台运行面包括：

- 数据平面：Gateway 与协议兼容入口
- 控制平面：Admin API + Admin App
- 自助平面：Portal API + Portal App
- Web/Product 宿主层
- 兼容保留入口：`console/`

这是平台骨架事实，不应夸大为“所有形态都已同成熟度”。

### 4.2 协议与能力覆盖面

当前可验证的协议面包括：

- OpenAI Compatible `/v1/*`
- Anthropic `/v1/messages`、`/v1/messages/count_tokens`
- Gemini `generateContent`、`streamGenerateContent`、`countTokens`

当前可验证的能力家族包括：

- chat / responses / completions / embeddings / moderations
- images / audio / files / uploads / containers
- assistants / threads / conversations / vector stores
- batches / fine tuning / webhooks / realtime / evals
- videos

补充说明：

- `music` 已进入统一资源化能力口径，且账本存在 `music_seconds` 维度
- 但 `music` 的实际上游兼容范围与执行真值仍以兼容矩阵和参考文档为准，不应仅凭账本维度推断其已完成全量生产化覆盖

### 4.3 双运行模式

当前必须严格区分两种形态：

- `stateful standalone`：连接 Admin Store，执行本地鉴权、租户/项目解析、路由、配额、用量与计费收口
- `stateless runtime`：承担轻量 relay / compatibility / runtime embedding，不等同于完整本地控制平面

这是一项结构性优势，但所有成熟度判断都必须拆开评估，不能把 stateful 的能力错误外推到 stateless。

### 4.4 路由内核资产

当前可验证的路由资产包括：

- `RoutingPolicy`
- `RoutingProfileRecord`
- `ProjectRoutingPreferences`
- `ProviderHealthSnapshot`
- `RoutingDecisionLog`
- route decision cache
- capability catalog cache
- distributed lock store

当前可验证的策略插件包括：

- `deterministic_priority`
- `weighted_random`
- `slo_aware`
- `geo_affinity`

结论：当前已具备企业级调度内核骨架，但仍不应把“路由骨架存在”写成“终局策略全部完成”。

### 4.5 Provider 与扩展治理

当前仓库已具备三类统一扩展入口：

1. `builtin`
2. `connector`
3. `native_dynamic`

当前可验证的内建 Provider 至少包括：

- `openai`
- `openrouter`
- `ollama`

`openrouter` 适配器体现了正确工程方向：

- 复用 OpenAI 主契约和主适配层
- 仅在 Provider 特有位置注入偏好字段
- 当前可验证偏好字段包括 `order`、`allow_fallbacks`、`require_parameters`、`data_collection`、`zdr`

### 4.6 控制平面与自助门户

当前 `admin-api` 已可验证的高价值能力包括：

当前 `portal-api` 已可验证的高价值能力包括：

- 工作区级 API key 生命周期
- 工作区级 API key group 生命周期
- `routing/summary`
- `routing/profiles`

结论：当前平台已不再等同于“发 Key 页面”，但“能力面已存在”仍不等于“操作体验与工作流已达到行业最优”。

## 5. 当前明确差距

### 5.1 数据策略仍以 Provider 级事实为主

当前可验证事实：

- OpenRouter 适配器已支持 `data_collection` 与 `zdr` 偏好注入

当前仍不能写成“已全部具备”的部分：

- 租户级 / 项目级 / API Key Group 级统一数据策略产品化
- 跨 Provider 可比较的隐私策略总线
- “高敏请求仅允许命中满足 no-log / region restriction 的候选集”这类完整策略编排界面

### 5.2 prompt cache / sticky routing 仍是目标能力

当前正确表述应为：

- 仓库已具备 route / cache / lock 基础设施
- 当前不应宣称已经具备完整的 prompt cache 产品化、sticky routing 产品化与命中效果运营闭环

### 5.3 控制面能力面强，但工作流仍需打磨

当前 Admin / Portal 更像“治理能力已进入真实产品边界”，但要达到行业领先，还需要继续产品化：

- Provider onboarding 工作流
- 模型目录发布工作流
- 路由仿真与灰度放量工作流
- 计费、定价、配额、告警联动工作流
- 扩展安装、实例绑定、状态检查、reload、rollout 的一体化工作台

### 5.4 go-live 标准仍需硬化

当前仓库已具备较强安装与发布骨架，但达到行业领先仍需要：

- 稳定容量压测基线
- SLO 与告警分级标准
- 灰度、回滚、演练验收标准
- stateful 与 stateless 分别定义的上线门槛

## 6. 能力矩阵

| 维度 | OpenRouter 公开信号 | new-api 公开信号 | 当前 `sdkwork-api-router` | 当前判断 | 目标标准 |
|---|---|---|---|---|---|
| 协议兼容 | 强统一入口 | 强协议覆盖 | OpenAI + Anthropic + Gemini + 大量 `/v1/*` 家族 | 当前覆盖已广 | 形成协议 profile 与版本治理体系 |
| 多模态 | 强模型与 Provider 聚合 | 强应用接入面 | 文本、图像、音频、视频、文件、向量、批处理、评测、Realtime；`music` 以兼容矩阵为准 | 当前覆盖已广 | 多模态目录、差异约束与定价统一 |
| 路由内核 | 数据策略、缓存、性能感知 | weighted random、限流、重试 | policy / profile / health / decision log / snapshot / geo_affinity / slo_aware | 当前骨架已成 | 成本、延迟、吞吐、隐私统一评分 |
| 运行形态 | 平台服务形态突出 | 部署友好 | `stateful standalone` + `stateless runtime` | 当前双形态已成 | 两种形态均有清晰标准与验收 |
| 控制平面 | 偏路由治理 | 偏运营后台 | Admin API + Admin App 能力面较深 | 当前治理面较深 | 完整治理工作台与灰度编排 |
| 门户与商业化 | 相对轻 | 相对强 | Portal 已具 billing、account、pricing、routing、coupon | 当前入口已成体系 | 完整自助运营与企业结算体系 |
| Provider 生态 | 强聚合深度 | 强渠道广度 | builtin + connector + native_dynamic | 当前扩展面已成 | 标准化 Provider Registry 与治理基线 |
| 数据策略 | ZDR / data policy 明确 | 运维治理友好 | OpenRouter 级偏好已支持，平台级策略仍在补强 | 当前平台级治理待补 | 租户 / 项目 / 分组级策略一等化 |
| 可观测性 | 文档清晰 | 运维入口友好 | `/metrics`、`x-request-id`、decision log、health snapshot、rollout/runtime status | 当前观测资产已成 | SLO 驱动运营闭环 |
| 部署与发布 | 平台托管为主 | Docker / 多库友好 | build/install/start/stop + service descriptors + release manifest | 当前交付骨架已成 | SaaS / 私有化 / air-gapped 一体标准 |

## 7. 本轮架构裁决

1. 目标不是复制 OpenRouter，也不是复制 new-api。
2. 正确方向是吸收 OpenRouter 的路由与数据策略深度，再结合 new-api 的交付与运营友好性。
3. 当前仓库最有价值的差异化优势，是已形成多运行面骨架、双运行模式、扩展运行时和较深控制平面。
4. 当前最需要补强的，不是“继续堆协议”，而是把数据策略、缓存命中、灰度治理、SLO 与 go-live 标准做深。

## 8. 评估标准

### 8.1 行业竞争力评估

- 是否同时具备协议兼容、路由治理、控制平面、自助门户、交付能力五类能力
- 是否在至少三个关键维度上明显优于“纯转发网关”

### 8.2 当前态真实性评估

- 标记为“当前已具备”的能力，是否都能在代码、接口文档或服务实现中找到证据
- 是否避免把 Provider 级能力误写成平台级产品化能力

### 8.3 stateful / stateless 分离评估

- 是否分别评估 `stateful standalone` 与 `stateless runtime` 的成熟度
- 是否避免把 stateful 的本地治理能力错误套用到 stateless

### 8.4 终局对齐评估

- 是否已具备数据策略、缓存、灰度发布、SLO、回滚和容量规划等行业领先能力
- 若未达到，是否在文档中明确列出差距而不是含糊处理
