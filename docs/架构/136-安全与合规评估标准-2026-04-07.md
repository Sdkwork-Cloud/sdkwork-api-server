# 安全与合规评估标准

> 日期：2026-04-07
> 范围：身份、权限、密钥、数据策略、多租户、审计、发布安全

## 1. 文档目标

本文档把 `sdkwork-api-router` 的安全与合规要求固化为可评估、可验收、可交付的标准，而不是停留在原则口号。

本文档只把仓库、接口、配置与治理对象中已验证存在的内容写成事实；平台级统一数据策略台、细粒度权限体系、跨 Region 合规编排等内容只作为目标状态表述。

## 2. 当前可验证事实

### 2.1 当前已存在三条清晰身份边界

当前仓库可验证的主身份边界包括：

- Admin JWT
- Portal JWT
- Gateway API Key

这说明运维面、门户面和数据面已经不是一套混用身份。

### 2.2 当前兼容协议认证仍收敛到统一网关边界

当前可验证事实包括：

- Anthropic 兼容入口接受 `Authorization` 与 `x-api-key`
- Gemini 兼容入口接受 `Authorization`、`x-goog-api-key` 与 `?key=`
- 兼容协议差异没有形成绕开统一网关身份解析的第二信任边界

### 2.3 当前已形成 Secret Backend 体系

当前已验证的 Secret Backend 包括：

- `database_encrypted`
- `local_encrypted_file`
- `os_keyring`

当前配置中还能验证以下关键字段：

- `secret_backend`
- `secret_local_file`
- `secret_keyring_service`

### 2.4 当前多租户边界已进入真实业务对象

当前可验证事实包括：

- API Key 与 API Key Group 绑定 `tenant_id + project_id + environment`
- Portal 侧路由、计费、API Key Group、Pricing 等查询以工作区边界过滤
- `billing/events/summary` 支持按 project、API key group、capability、accounting mode 聚合

### 2.5 当前已存在 Provider 级数据策略落点

当前可验证事实包括：

- OpenRouter 适配器支持 `data_collection`
- OpenRouter 适配器支持 `zdr`

这代表 Provider 级数据策略偏好已经有真实落点，但不等于平台级统一数据策略治理已闭环。

## 3. 安全评估维度

安全评估统一采用八个维度：

1. 身份认证
2. 权限与越权防护
3. 密钥与凭据保护
4. 数据最小暴露
5. 多租户隔离
6. 审计与追责
7. 发布与运维安全
8. 合规与数据策略

## 4. 身份认证标准

### 4.1 通过标准

- Admin JWT、Portal JWT、Gateway API Key 三套边界清晰。
- 兼容协议认证最终收敛到统一网关身份解析。
- 高价值接口默认拒绝匿名访问。

### 4.2 不通过信号

- Portal 身份可直接访问 Admin 控制面能力。
- 请求体中的 tenant / project 被直接信任为权威身份。
- 兼容协议绕开统一网关中间件。

## 5. 权限与越权防护标准

### 5.1 通过标准

- Admin 关键写操作具备服务端权限校验。
- Portal 只能访问调用方自己的工作区范围。
- API key group、routing profile、pricing、credential 等对象不得跨工作区随意绑定。

### 5.2 不通过信号

- 只做前端按钮控制，没有服务端校验。
- 查询接口缺少 tenant / project 过滤。
- 允许引用其他租户或项目的 group、profile、credential。

## 6. 密钥与凭据保护标准

### 6.1 通过标准

- Provider secret 不以明文落表返回。
- 密钥后端、密钥定位信息和业务配置清晰分离。
- 日志、错误响应、脚本输出不泄露敏感值。
- 凭据轮换必须具备文档、验证和回滚方案。

### 6.2 当前可验证强化点

当前可以写成事实的是：

- Secret backend 已进入配置与运行时治理范围。
- 历史凭据可读性依赖的定位信息与主密钥元数据已被显式建模。

当前不能直接写成事实的是：

- 全平台已经具备统一密钥轮换编排台。
- 所有凭据都已形成自动轮换闭环。

### 6.3 不通过信号

- 明文 API key / provider key 出现在数据库、日志或标准脚本输出中。
- 凭据与普通配置混存。
- 切换密钥后无法兼容读取旧凭据。

## 7. 数据最小暴露与数据策略标准

### 7.1 通过标准

- 高价值日志默认不落敏感载荷。
- Provider 级数据策略偏好可被表达和传递。
- 不同风险等级请求可应用不同数据策略。

### 7.2 当前事实与目标边界

当前可以写成事实的是：

- OpenRouter Provider 级 `data_collection` / `zdr` 注入。

当前只能写成目标的是：

- 租户级、项目级、API key group 级统一数据策略治理。
- 可比较、可校验、可强制执行的跨 Provider 数据策略总线。
- no-log、region restricted、zero retention 的统一控制面工作台。

## 8. 多租户隔离标准

### 8.1 通过标准

- 鉴权后的权威上下文必须包含 tenant / project 归属。
- usage、billing、routing decision 必须按租户边界回溯。
- Portal 视图不得泄露其他工作区的 group、pricing、billing event。

### 8.2 不通过信号

- 关键明细缺少 tenant / project 维度。
- 跨租户导出依赖前端过滤而不是服务端过滤。
- API key group、pricing 或 routing profile 可跨租户误引用。

## 9. 审计与追责标准

### 9.1 高价值审计对象

至少应把以下动作视为高价值审计对象：

- 登录与失败登录
- API key / API key group 生命周期变更
- Provider / channel / model / credential 变更
- routing policy / profile / preference 变更
- extension install / instance / reload / rollout 变更
- pricing 与 billing 关键治理动作
- release、rollback、kill switch 等生产高风险动作

### 9.2 审计记录最低字段

- actor
- target
- action
- request id
- timestamp
- result
- before / after 摘要

## 10. 发布与运维安全标准

### 10.1 通过标准

- build、install、start、stop、release 资产可审计。
- 服务注册与发布描述文件可检查。
- 配置变更、reload、rollout、rollback 边界清晰。

### 10.2 当前可验证资产

- `bin/build.*`
- `bin/install.*`
- `bin/start.*`
- `bin/stop.*`
- `service/systemd/*`
- `service/launchd/*`
- `service/windows-task/*`
- `release-manifest.json`

## 11. 合规成熟度等级

### 11.1 L1 基线

- 身份边界清晰。
- 凭据不明文裸奔。
- 工作区边界可验证。

### 11.2 L2 生产

- 权限、租户隔离、审计、发布安全形成最小闭环。
- 高价值写操作和工作区对象都由服务端硬校验。

### 11.3 L3 企业级目标

- 数据策略、Region 策略、细粒度权限、发布治理一体化。
- 扩展生态、热更新和跨节点 rollout 进入严格安全编排。

## 12. 评分卡

| 维度 | 权重 | 通过条件 |
|---|---|---|
| 身份认证 | 15 | 三套身份边界清晰且不可互穿 |
| 权限控制 | 15 | 服务端权限校验完整 |
| 密钥保护 | 15 | 无明文敏感信息泄露，secret backend 可治理 |
| 数据策略 | 10 | 敏感请求最小暴露，事实与目标边界清晰 |
| 多租户隔离 | 15 | 查询、配置、计费、路由隔离可验证 |
| 审计 | 10 | 高价值动作可追责 |
| 运维安全 | 10 | 发布、reload、rollout、rollback 资产可审计 |
| 合规扩展 | 10 | 具备继续提升为企业级治理的结构基础 |

总分低于 `80` 不得宣称达到企业级生产安全标准。

## 13. 设计红线

- 禁止把目标状态的统一数据策略台写成当前已完成事实。
- 禁止把兼容协议外观差异扩展成第二套安全边界。
- 禁止把多租户隔离退化为前端展示过滤。
- 禁止在未定义审计和回滚边界的情况下开放高风险生产操作。

## 14. 评估标准

### 14.1 当前事实真实性

- 是否准确写明 OpenRouter 的 Provider 级数据策略支持。
- 是否避免把平台级统一数据策略工作台误写成当前已完成。

### 14.2 发布门槛

- 关键安全维度是否全部达到最小上线标准。
- 是否杜绝“先上线后补安全”的口径。

### 14.3 持续治理

- 重大协议、路由、扩展、发布能力变更后，是否重新执行本评分卡。
