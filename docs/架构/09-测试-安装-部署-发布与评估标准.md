# 测试-安装-部署-发布与评估标准

## 1. 文档目标

定义质量保障、安装与启动、部署形态、发布门禁、回滚与阶段性验收标准。

## 2. 测试矩阵

最低测试矩阵必须覆盖：

- 后端：领域测试、应用测试、接口测试、存储测试、兼容协议测试
- 前端：类型检查、构建验证、关键工作区交互验证
- 平台：性能压测、安全验证、发布验证、回滚验证
- 文档：兼容矩阵、构建与安装说明一致性验证

推荐验证基线以 `docs/reference/build-and-tooling.md` 与仓库 `README.md` 为准，至少应覆盖：

- `cargo test --workspace -q -j 1`
- `pnpm --dir apps/sdkwork-router-admin typecheck && pnpm --dir apps/sdkwork-router-admin build`
- `pnpm --dir apps/sdkwork-router-portal typecheck && pnpm --dir apps/sdkwork-router-portal build`
- `pnpm --dir docs build`

## 3. 安装与启动标准

当前主入口分两层：

- 开发入口：`scripts/dev/*`
- 交付入口：`bin/build.*`、`bin/install.*`、`bin/start.*`、`bin/stop.*`

要求：

- 本地开发、预览、桌面模式、发布模式都必须有标准脚本
- 默认端口、默认配置根、默认数据库路径必须可解释、可覆盖
- 安装后的运行时必须自包含，不能依赖源码工作区隐式文件

## 4. 部署拓扑标准

标准部署形态至少包括：

- 本地单机开发
- 共享 SaaS Cell
- Dedicated Cell
- 私有化部署
- `router-product-service` 一体化部署
- `gateway / admin / portal / web` 独立服务部署

## 5. 发布门禁

发布前必须同时满足：

- 代码、文档、兼容矩阵、构建脚本一致
- 版本、变更说明、验收清单齐全
- 构建产物可安装、可启动、可健康检查、可停止
- 发布后有明确验证清单，发布失败有明确回滚清单

依赖治理门禁：

- 源码工作区允许使用 sibling repo 相对路径依赖
- Release 与 GitHub 发布必须切换为 GitHub 不可变依赖地址
- `sdkwork-core`、`sdkwork-ui`、`sdkwork-appbase`、`sdkwork-im-sdk` 等独立仓库必须先做本地与远端版本一致性校验
- 远端不可验证、本地仓库脏、标签或提交不一致时，不得提交、push、release

变更说明门禁：

- GitHub Release 说明必须基于 `/docs/release` 与版本历史生成
- 之前未成功发布的版本变更，必须并入当前 Release
- `/docs/release` 不完整时，必须从上一次成功发布到当前代码变更补全 changelog

## 6. 回滚标准

回滚必须满足：

- 有上一个可安装版本包
- 有可恢复配置与服务注册脚本
- 有回滚前、回滚后验证清单
- 有对账、账本、结算与扩展安装状态的恢复边界说明

## 7. 阶段性验收标准

一期上线至少满足：

- 协议兼容、路由、配额、计费、审计主链路闭环
- Admin 对目录、路由、扩展、账务、诊断可治理
- Portal 对注册、登录、工作区、密钥、用量、账务可自助
- 本地、测试、生产三类环境都有标准启动与发布路径

二期增强重点：

- 商业化闭环深化
- 多 Cell / 多 Region 演进
- 更强配置治理、灾备与容量治理

## 8. 评估标准

- 是否存在清晰的发布前检查、发布后验证、回滚清单
- 是否能在脱离本地工作区的 Release 环境中稳定运行
- 是否能把测试、安装、部署、发布、回滚纳入同一套工程标准，而不是依赖人工经验
