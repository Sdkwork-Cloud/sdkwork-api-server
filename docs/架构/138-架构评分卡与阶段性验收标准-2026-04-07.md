# 架构评分卡与阶段性验收标准

> 日期：2026-04-07
> 范围：产品、架构、性能、安全、测试、交付、运维

## 1. 文档目标

本文档把 `sdkwork-api-router` 的主线能力量化为评分卡和阶段性验收标准，用于统一回答两个问题：

1. 当前版本到底处于什么成熟度阶段。
2. 距离领先级目标还差哪些硬门槛。

本文档不直接把“领先级目标”写成“当前事实得分”。

## 2. 评分原则

### 2.1 必须拆开评估两类运行形态

评分时必须分别评估：

- `stateful standalone`
- `stateless runtime`

原因是：

- `stateful standalone` 已承载本地治理、路由证据、计费与控制面联动。
- `stateless runtime` 当前更接近轻量 relay / runtime 嵌入形态。

不拆开评估，就会把 stateful 的能力误外推给 stateless。

### 2.2 必须区分“当前事实分”和“目标分”

- 当前事实分：只按当前仓库可验证证据评分。
- 目标分：按领先级标准衡量，可包含尚未全部落地项。

## 5. 维度评分标准

### 5.1 产品与协议完备度

看点：

- 是否覆盖 OpenAI / Anthropic / Gemini 等高价值协议面。
- 是否覆盖文本、图片、语音、视频、音乐等能力家族。
- 是否同时具备 gateway / admin / portal 三个产品面。

### 5.2 架构与模块边界

看点：

- 是否保持 interface / app / domain / storage-provider-runtime 分层。
- 是否具备 Provider 可插拔、扩展运行时与运行模式分层。
- 是否把控制平面与热路径清晰隔离。

### 5.3 路由与多模态能力

看点：

- 是否具备 policy、profile、preferences、health snapshot、decision log。
- 是否具备 geo affinity、weighted random、slo aware 等策略基础。
- 多模态链路是否共享统一路由主链。

### 5.4 性能与容量

看点：

- 是否定义平台自身开销口径。
- 是否定义 stateful 与 stateless 两类执行面的不同门槛。
- 是否具备容量规划、热点治理与发布前压测要求。

### 5.5 安全与合规

看点：

- 身份边界是否清晰。
- 多租户隔离是否可验证。
- Secret backend、数据策略、审计与发布安全是否达标。

### 5.6 测试与质量保障

看点：

- 是否覆盖单元、契约、集成、运行时与关键前端路径。
- 是否覆盖 stateful 与 stateless 的关键差异。
- 是否覆盖 Provider、routing、billing、extension runtime 等主链路。

### 5.7 安装部署发布回滚

看点：

- 是否具备 build / install / start / stop / release 全链路。
- 是否具备服务注册资产与 release manifest。
- 是否具备 reload / rollout / rollback 边界。
- 是否具备 SDK 远程一致性与 GitHub Release 依赖治理。

### 5.8 可观测性与运维

看点：

- 是否具备 health、metrics、tracing、`x-request-id`。
- 是否具备 `RoutingDecisionLog`、`ProviderHealthSnapshot` 与 runtime status。
- 是否具备 SLO、告警与统一诊断路径。

## 6. 等级定义

### 6.1 L1 可运行

- 本地可启动
- 主接口可访问
- 基本文档可读

### 6.2 L2 可开发

- 模块边界基本清晰
- 主路径具备自动化验证
- gateway / admin / portal 已联通

### 6.3 L3 可生产

- PostgreSQL / Redis 主链路稳定
- 发布、回滚、诊断边界清晰
- 安全、可观测性、商业化与路由主链路达到生产基线

### 6.4 L4 企业级

- 多租户治理成熟
- 数据策略与发布治理成熟
- 扩展运行时、容量规划、SLO 进入稳定运营态

### 6.5 L5 领先级目标

- 协议兼容与多模态覆盖领先
- 路由、缓存、数据策略、商业化、交付与灾备全部达到平台级成熟度
- `stateful standalone` 与 `stateless runtime` 都拥有独立、可信的领先级验收结论

## 7. 当前阶段判断

### 7.1 对 `stateful standalone` 的当前判断

基于当前可验证事实，更稳妥的表述应为：

- 已完成 L2 基线
- 具备明显的 L3 候选特征
- 尚不应直接宣称达到 L4 或 L5

支撑证据包括：

- 完整的 gateway / admin / portal / web 产品骨架
- OpenAI / Anthropic / Gemini 兼容面
- policy / profile / health snapshot / decision log / simulation
- billing events / account / pricing / API key group
- extension installation / instance / runtime status / reload / rollout
- build / install / service descriptor / release manifest

尚未直接宣称更高等级的原因包括：

- 平台级统一数据策略仍在继续收口
- prompt cache、sticky routing、容量基线与 SLO 证据仍需持续做实
- 多 Cell / 多 Region / 灾备门槛尚不能写成当前事实

### 7.2 对 `stateless runtime` 的当前判断

基于当前可验证事实，更稳妥的表述应为：

- 已完成 L1 基线
- 具备 L2 候选特征
- 不应被误写为与 standalone 对等的控制平面产品

支撑证据包括：

- 主协议兼容与 relay / transcoding 基础
- 单上游 runtime 嵌入路径
- 与主代码库共享的大量协议处理逻辑

当前不能直接宣称具备的是：

- 与 standalone 同级的本地商业化治理
- 与 standalone 同级的本地审计、计费、发布证据
- 与 standalone 完全对等的上线标准

## 8. 发布门槛

### 8.1 预发门槛

- 总分不低于 `75`
- 安全维度不低于 `12/15`
- 测试维度不低于 `7/10`
- `stateful standalone` 与 `stateless runtime` 都必须给出独立结论

### 8.2 正式生产门槛

- 总分不低于 `85`
- 安全维度不低于 `13/15`
- 安装部署发布回滚维度不低于 `8/10`
- 可观测性与运维维度不低于 `8/10`
- 不得存在“当前事实不明确”的一级维度空白项

### 8.3 领先级门槛

- 总分不低于 `92`
- 任一一级维度不低于 `80%`
- 数据策略、缓存命中、SLO、灰度与回滚都必须有稳定证据
- 不允许仅靠 stateful 得分掩盖 stateless 的成熟度缺口

## 9. 阶段性验收清单

### 9.1 架构验收

- 主干架构文档已冻结
- 当前事实与目标状态边界清晰
- stateful / stateless 口径一致

### 9.2 功能验收

- 关键协议面可用
- Admin / Portal 核心工作区可用
- Provider、routing、billing、extension runtime 主链路可用

### 9.3 交付验收

- 可构建
- 可安装
- 可启动
- 可停止
- 可回滚

### 9.4 运维验收

- health / metrics / logs / tracing 可用
- `x-request-id` 可追踪
- decision log / health snapshot / runtime status 可查询
- 关键 billing evidence 可回溯

## 10. 设计红线

- 禁止用“支持很多 API”替代平台级评分。
- 禁止把目标分数写成当前事实分数。
- 禁止把 stateful 的得分直接挪给 stateless。
- 禁止在安全、可观测性、回滚三项不达标时宣称“可生产”。

## 11. 结论口径

当前版本更准确的结论应表述为：

- `sdkwork-api-router` 在 `stateful standalone` 形态下已具备平台级骨架和较高成熟度，当前最稳妥的判断是“L2 已完成，L3 候选”。
- `sdkwork-api-router` 在 `stateless runtime` 形态下已具备高价值兼容与嵌入能力，但不应被误写成与 standalone 对等的控制平面产品。
- 下一阶段的主攻方向不是继续堆接口，而是继续把数据策略、缓存、容量、SLO、灰度、回滚和灾备标准做实。
