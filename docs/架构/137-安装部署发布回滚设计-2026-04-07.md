# 安装部署发布回滚设计

> 日期：2026-04-07
> 范围：工作区启动、发布安装、服务注册、GitHub Release、回滚

## 1. 文档目标

本文档定义 `sdkwork-api-router` 从源码运行到生产安装、从本地工作区到 GitHub Release 的完整交付路径，确保安装、部署、发布、回滚行为可重复、可审计、可自动化。

## 2. 当前可验证事实

### 2.1 当前已具备完整交付资产

当前仓库已具备以下交付资产：

- `scripts/dev/*`
- `bin/build.*`
- `bin/install.*`
- `bin/start.*`
- `bin/stop.*`
- `service/systemd/*`
- `service/launchd/*`
- `service/windows-task/*`
- `release-manifest.json`

### 2.2 当前已存在多种启动形态

当前文档、脚本与运行时可共同验证以下形态：

- 原生独立服务启动
- 工作区源码启动
- 托管开发态启动
- 托管发布态启动

### 2.3 当前已存在双口径端口体系

当前可验证事实是：

- 原生独立服务文档以 `127.0.0.1:8080 / 8081 / 8082` 为默认独立服务入口口径。
- 工作区与脚本管理形态以 `127.0.0.1:9980 / 9981 / 9982 / 9983` 为主要产品化口径。

这两套端口不是冲突，而是两种交付面：独立服务调试面与脚本托管产品面。

## 3. 安装模型

### 3.1 源码安装

适用：

- 开发
- 调试
- 回归测试

要求：

- Rust / Node / pnpm 依赖明确
- 一条命令可启动工作区
- 允许保留相对路径的 sibling SDK 依赖，服务于本地联调

### 3.2 发布安装

适用：

- 预发
- 生产
- 私有化交付
- GitHub Release 资产安装

要求：

- build 产物与 install 产物边界清晰
- 安装目录只保留运行时必需资产
- 站点目录、配置目录、数据目录、日志目录分离
- 发布安装不得依赖本地 sibling 相对路径 SDK

## 4. 启动与部署模型

### 4.1 原生独立服务

适用：

- 单服务调试
- 进程级故障定位
- 直接验证 `gateway-service / admin-api-service / portal-api-service`

### 4.2 工作区模式

适用：

- 本地前后端联调
- Admin / Portal / Console 开发
- 桌面端与浏览器双形态调试

### 4.3 托管发布模式

适用：

- 预发与生产
- systemd / launchd / Windows Task Scheduler 托管
- 私有化标准交付

### 4.4 拓扑模型

当前交付至少支持三类拓扑：

- 一体化部署
- 分体部署
- 按 `gateway / admin / portal / web` 角色切片部署

## 5. 发布流水线标准

标准发布流水线必须至少包括：

1. 构建 Rust release 二进制与前端静态站点。
2. 生成安装目录、配置模板、服务描述文件和 `release-manifest.json`。
3. 执行安装验收、健康检查与关键入口验收。
4. 生成 GitHub Release 资产与发布说明。
5. 保存可回滚版本和可审计元数据。

## 6. SDK依赖治理

### 6.1 工作区规则

工作区源码形态允许使用 sibling 相对路径 SDK 依赖，用于本地联调和减少不必要的 npm 中央仓库发布频次。

### 6.2 发布规则

Release 环境与 GitHub Release 资产必须使用不可变的 GitHub 地址或 Git 提交引用，不得继续引用本地相对路径依赖。

### 6.3 一致性校验规则

以下仓库在正式发布前必须完成本地与远程一致性校验：

- `sdkwork-core`
- `sdkwork-ui`
- `sdkwork-appbase`
- `sdkwork-craw-chat-sdk`（发布 Git 根：`apps/craw-chat`）

发布前必须确认：

- 本地仓库 HEAD 与计划引用的远程提交、Tag 或版本信息一致。
- Release 清单中的依赖地址与远程可访问地址一致。
- 如本地版本与远程 Git 状态不一致，必须阻断提交、push、发布与 GitHub Release。

这条规则的目的不是增加流程，而是防止发布产物引用一个本地存在、远程不存在、或远程版本不匹配的 SDK 真值。

## 7. GitHub Release标准

GitHub Release 必须至少包含：

- 可安装产物
- `release-manifest.json`
- 版本号与构建来源
- 变更说明
- 风险说明
- 回滚说明

Release 说明必须遵循：

- 以最近一次成功发布为基线汇总变更
- 之前未成功发布版本的变更必须并入当前成功发布说明
- 若 `/docs/release` 信息不完整，则回溯成功发布记录到当前代码差异重新汇总

## 8. 配置治理

发布安装必须清晰区分：

- 内建默认值
- 安装后环境变量
- 持久化业务配置
- 临时运行态衍生视图

不得把本地开发期配置、发布期配置和控制面持久化配置混为一谈。

## 9. 回滚模型

### 9.1 代码回滚

- 保留上一个稳定版本产物
- 发布目录具备版本化策略

### 9.2 配置回滚

- 配置文件必须可版本管理
- 环境变量变更必须可追踪
- reload / rollout 必须具备失败回退边界

### 9.3 数据回滚

- 迁移脚本必须考虑向后兼容与恢复路径
- 不可恢复迁移不得在无保护条件下直接进生产

### 9.4 流量回滚

- 按 Cell、按角色、按实例的灰度与回退边界必须清晰
- 长连接、流式请求、异步任务要定义中断与 drain 策略

## 10. 私有化交付标准

私有化交付必须提供：

- 安装说明
- 配置说明
- 启动说明
- 停止说明
- 升级说明
- 回滚说明
- 健康检查与诊断说明
- 依赖来源说明

依赖来源说明必须显式区分：

- 本地联调用相对路径依赖
- Release 环境用 GitHub 不可变依赖

## 11. 设计红线

- 禁止在 Release 产物中保留本地 sibling 相对路径 SDK 依赖。
- 禁止在未完成 SDK 本地/远程一致性校验时提交、push 或发布。
- 禁止在没有安装验收、健康验收和回滚路径时创建 GitHub Release。
- 禁止让发布脚本默认输出敏感环境变量或密钥。
- 禁止把原生独立服务口径与托管产品口径混成同一套运维说明。

## 12. 评估标准

### 12.1 安装评估

- 是否能在无隐式经验条件下完成安装。
- 是否能区分源码运行和发布运行。
- 是否能准确区分 `808x` 独立服务口径与 `998x` 产品化口径。

### 12.2 发布评估

- 是否具备构建、安装、启动、验收、发布说明全链路。
- 是否能提供 GitHub Release 所需全部产物与说明。
- 是否能追踪 Release 依赖指向的 GitHub 真值。

### 12.3 依赖评估

- 是否完成 `sdkwork-core / sdkwork-ui / sdkwork-appbase / sdkwork-craw-chat-sdk`（发布 Git 根：`apps/craw-chat`）的本地与远程一致性比对。
- 发现不一致时是否自动阻断发布。

### 12.4 回滚评估

- 是否具备代码、配置、数据、流量四条回滚路径。
- rollout / rollback 是否可观察、可审计、可分批执行。
