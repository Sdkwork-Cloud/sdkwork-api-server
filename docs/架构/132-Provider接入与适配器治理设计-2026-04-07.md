# Provider接入与适配器治理设计

> 日期：2026-04-07
> 范围：Provider、协议兼容层、适配器、扩展运行时、能力目录

## 1. 文档目标

本文档定义 `sdkwork-api-router` 的 Provider 接入与适配器治理标准，目标是让平台在持续引入新 Provider、新协议族和新运行时形态时，仍保持：

- 高内聚
- 低耦合
- 可验证
- 可治理
- 可回滚

本文档严格区分“当前可验证事实”和“目标状态”，避免把规划中的统一能力目录、统一治理台和统一运行时误写成已全部落地。

## 2. 当前可验证事实

### 2.1 当前已存在三类运行时入口

当前仓库、Admin 控制面与运行时治理对象可以共同验证三类入口：

- `builtin`
- `connector`
- `native_dynamic`

这说明 Provider 接入并不是单一路径，而是已经具备“内建实现、外部连接器、原生动态运行时”三种装配面。

### 2.2 当前已知内建 Provider

当前仓库中可直接验证的内建 Provider 至少包括：

- `openai`
- `openrouter`
- `ollama`

### 2.3 当前已存在扩展治理对象

当前 Admin 控制面已暴露并可验证以下治理对象：

- `extensions/installations`
- `extensions/packages`
- `extensions/instances`
- `extensions/runtime-statuses`
- `extensions/runtime-reloads`
- `extensions/runtime-rollouts`
- `runtime-config/rollouts`

这表明平台当前不是“只能加载插件”，而是已经进入“安装、实例化、状态观测、reload、rollout、配置传播”一体化治理阶段。

### 2.4 当前已具备扩展持久化与协同基础

当前可验证事实包括：

- SQLite / PostgreSQL 已持久化扩展安装与实例对象。
- 扩展安装对象与扩展实例对象是分离建模的。
- rollout 与 config rollout 已具备 durable participant record 一类的协同记录模型。

### 2.5 当前已存在 Provider 级差异注入样例

`openrouter` 当前已验证支持以下 Provider 级差异字段：

- `order`
- `allow_fallbacks`
- `require_parameters`
- `data_collection`
- `zdr`

它证明当前仓库已经采用“复用共享协议主契约，仅在局部注入 Provider 差异”的正确方向。

## 3. 架构判断

### 3.1 接入主线

Provider 接入的主线应始终是：

1. 复用共享协议与共享请求模型。
2. 在适配层表达 Provider 差异。
3. 在能力层表达可路由真值。
4. 在治理层表达安装、实例、状态、灰度与回滚。

### 3.2 当前与目标的边界

当前可以写成事实的是：

- 已有多类运行时入口。
- 已有扩展治理对象。
- 已有 Provider 级局部差异注入样例。

当前只能写成目标的是：

- 全平台统一 capability registry 已彻底收口为单一权威真值。
- 任意新 Provider 都能以固定模板、固定工期、固定回归矩阵完成接入。

## 4. Provider标准模型

每个 Provider 都必须显式给出以下四类信息。

### 4.1 身份模型

- provider id
- display name
- vendor
- runtime kind
- extension id
- trust / rollout posture

`extension_id`、`installation_id`、`instance_id` 必须严格区分：

- `extension_id`：谁提供了这类扩展或内建能力。
- `installation_id`：哪一次安装记录生成了该运行时来源。
- `instance_id`：哪一个真实可路由节点参与请求执行。

### 4.2 能力模型

- protocol support
- modality support
- model family support
- parameter support
- streaming support
- tool support
- caching support
- data policy support

当前仓库已经围绕 capability 和 routing assessment 建模，但统一能力目录仍应继续收口为单一真值源。

### 4.3 凭据模型

- 需要何种 credential
- credential 如何引用
- 是否允许实例级覆盖 base URL
- 是否存在 Provider 特有认证头、查询参数或签名方式

### 4.4 运维模型

- 健康探测方式
- timeout 建议
- retry 建议
- fallback 姿态
- rollout 风险等级
- 是否需要区域隔离或数据策略约束

## 5. 适配器分层标准

### 5.1 ProviderAdapter

职责：

- 提供统一身份、能力与元数据。
- 声明该 Provider 暴露什么能力，不直接承担请求执行。

### 5.2 ProviderExecutionAdapter

职责：

- 接收统一 `ProviderRequest`。
- 输出统一 `ProviderOutput` 或 `ProviderStreamOutput`。
- 屏蔽上游差异，不污染领域模型。

### 5.3 Provider-Specific Mapping

职责：

- 仅在局部注入 Provider 专有 header、body 字段和偏好项。
- 严禁把上游私有语义扩散为平台级公共字段。

### 5.4 Compatibility Bridge

职责：

- 承接 Anthropic、Gemini 等兼容面与 Provider 执行层之间的协议差异。
- 严禁在兼容层再长出第二套路由系统或第二套业务语义。

## 6. 扩展运行时治理模型

### 6.1 Installation、Instance、Runtime Status 的边界

最小边界必须保持为：

- `installation`：安装来源、版本、启用状态、静态配置。
- `instance`：面向路由的具体执行实体。
- `runtime status`：该实例当前是否健康、可达、可接流量。

当前代码和控制面已经体现这三层对象，后续文档和接口不得再把三者混写成同一个概念。

### 6.2 三类运行时的治理要求

`builtin`

- 进入主回归矩阵。
- 进入主发布链路。
- 默认承担最高质量要求。

`connector`

- 必须纳入运行时监督。
- 必须具备健康检查、超时、重载和下线治理。

`native_dynamic`

- 必须声明 ABI / runtime 兼容边界。
- 必须具备加载、卸载、drain、回滚语义。
- 没有来源和信任治理时，不得进入生产流量。

## 7. Provider接入流程标准

新 Provider 接入至少要完成以下动作：

1. 定义身份、能力、凭据、运维元数据。
2. 复用共享协议模型，局部补齐 Provider 差异。
3. 完成非流式、流式、错误映射和超时策略。
4. 接入健康探测、健康快照和回退策略。
5. 接入 usage / billing 事实映射。
6. 接入 Admin 治理对象、状态可视化和 rollout。
7. 完成契约测试、运行时测试和回滚验证。

## 8. 设计红线

- 禁止在业务代码中散落手写上游 HTTP 请求并绕过统一适配层。
- 禁止把 Provider 私有字段扩散为全局领域字段。
- 禁止没有 capability metadata 的 Provider 进入正式治理对象。
- 禁止没有健康探测、状态可见性和回滚语义的扩展进入生产流量。
- 禁止把 installation、instance、runtime status 混写成单一对象。

## 9. 评估标准

### 9.1 当前事实真实性

- 文档是否只把已验证对象写成事实。
- 是否避免把统一 capability registry 写成“当前已完全成型”。

### 9.2 接入效率

- 新 Provider 是否主要通过共享契约和局部映射完成接入。
- 是否能把改动限制在有限文件边界内。

### 9.3 治理完备度

- Admin 是否可管理安装、实例、状态、reload、rollout。
- 运行时是否具备观察、隔离、回退和下线能力。

### 9.4 风险隔离

- Provider 差异是否被收束在适配层。
- 兼容协议层是否保持单一职责，没有长出第二套业务主链。
