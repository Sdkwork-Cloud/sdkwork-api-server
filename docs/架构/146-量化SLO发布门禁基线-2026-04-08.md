# 量化SLO发布门禁基线

> 日期：2026-04-08
> 目标：把 `/docs/架构/135` 与 `/docs/架构/143` 中已定义的 SLO / burn-rate 要求收敛成一份可执行、可测试、可进入 release-governance 的量化基线。

## 1. 当前已验证事实

- 仓库已具备 observability contract gate，见 `scripts/release/observability-contracts.mjs`。
- 仓库已具备路由层 SLO 相关领域模型与测试，例如 `max_cost`、`max_latency_ms`、`slo_applied`、`slo_degraded`。
- 发布治理入口 `scripts/release/run-release-governance-checks.mjs` 已经是统一 release-truth 入口。
- 现状缺口不是“没有观测资产”，而是“没有平台级量化 SLO 门禁基线”。

## 2. 本轮收口目标

- 新增一份机器可读的量化 SLO 基线。
- 把数据面、控制面、商业链三类 SLO 统一进同一份发布口径。
- 把 burn-rate 窗口写成可执行规则，而不是只保留在文档叙述层。
- 只落地“基线 + evaluator + test gate”，不伪造“live telemetry 已接通”的现状。

## 3. 基线设计

### 3.1 三个治理平面

- `data-plane`
  - `gateway-availability`
  - `gateway-non-streaming-success-rate`
  - `gateway-streaming-completion-success-rate`
  - `gateway-fallback-success-rate`
  - `gateway-provider-timeout-budget`
- `control-plane`
  - `admin-api-availability`
  - `portal-api-availability`
  - `routing-simulation-p95-latency`
  - `api-key-issuance-success-rate`
  - `runtime-rollout-success-rate`
- `commercial-plane`
  - `billing-event-write-success-rate`
  - `account-hold-creation-success-rate`
  - `request-settlement-finalize-success-rate`
  - `pricing-lifecycle-synchronize-success-rate`

### 3.2 三种指标类型

- `ratio_min`
  - 用于可用性和成功率，要求 `actual >= objective`
- `ratio_max`
  - 用于预算上限，例如 timeout budget，要求 `actual <= objective`
- `value_max`
  - 用于时延上限，要求 `actual <= objective`

### 3.3 burn-rate 规则

- 每个 target 固定携带两档窗口：
  - `1h`
  - `6h`
- 每个窗口都带 `maxBurnRate`
- 当前统一写入基线，是为了先把“有无门禁”收口，再为后续 live telemetry 对接提供稳定契约

### 3.4 证据来源

每个 target 必须声明至少一个 evidence source，优先指向现有资产：

- `/health`
- `/metrics`
- `RoutingDecisionLog`
- `ProviderHealthSnapshot`
- `runtime-config/rollouts`
- `billing/events`
- `billing/account-holds`
- `billing/request-settlements`

## 4. 代码落点

- `scripts/release/slo-governance.mjs`
  - 基线定义
  - 量化 evidence evaluator
  - `evidence-missing` blocked 语义
- `scripts/release/slo-governance-contracts.mjs`
  - 基线结构 contract
- `scripts/release/tests/release-slo-governance.test.mjs`
  - baseline / pass / fail / blocked / collect contract
- `scripts/release/run-release-governance-checks.mjs`
  - 固定纳入 `release-slo-governance-test`

## 5. 当前边界

- 已完成：
  - 量化基线存在
  - evaluator 存在
  - release-governance test lane 存在
  - child exec 受限时有 fallback
- 未完成：
  - live telemetry 证据文件未产出
  - live SLO lane 还未并入 release-governance
  - burn-rate 阻断还未接入真实 release / CI 工件流

## 6. 下一步

1. 固定 live evidence artifact 路径，例如 `docs/release/slo-governance-latest.json`。
2. 由 release / CI 产线真实生成该 artifact。
3. 在 evidence 路径稳定后，把 evaluator 从 test lane 升级为 live lane。
4. 再将平台级数据策略与 SLO 门禁并入同一 release-truth 入口。
