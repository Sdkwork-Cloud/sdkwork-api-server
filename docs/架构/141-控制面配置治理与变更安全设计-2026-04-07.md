# 控制面配置治理与变更安全设计

> 日期：2026-04-07
> 范围：配置权威源、分层、变更流程、Flag、Kill Switch、漂移治理、发布依赖

## 1. 文档目标

本文档定义 `sdkwork-api-router` 的配置真值、变更流程、传播路径和回退边界，防止系统出现“本地能跑、发布漂移、控制面与运行时不一致”的配置失控问题。

## 2. 当前可验证事实

当前可验证的控制面配置与观测对象包括：

- `routing/snapshots`
- `routing/preview`
- `routing/decision-logs`
- `runtime-config/rollouts`
- `extensions/runtime-rollouts`
- Secret backend 相关配置项

这些对象说明当前平台已经存在“配置真值、模拟、传播、观测、回滚”五个环节的现实基础。

## 3. 配置权威源

配置来源固定分为：

- 代码默认值
- 本地配置文件
- 环境变量
- 控制面持久化配置
- 运行时缓存与派生视图

权威原则：

- 控制面持久化配置是业务真值。
- 环境变量和本地文件服务于部署引导。
- 缓存和派生视图不得成为唯一事实来源。

## 4. 配置分层

### 4.1 平台级

- 服务绑定
- 存储后端
- Secret backend
- 扩展搜索路径
- 全局限流与熔断阈值

### 4.2 租户级

- 套餐
- 限额
- 默认价格策略
- 默认数据策略

### 4.3 项目级

- 模型偏好
- 路由偏好
- API key group
- 配额与计费姿态

### 4.4 运行级

- feature flag
- rollout
- kill switch
- 容量阈值

## 5. 变更流程

标准流程应为：

1. 草案
2. 审核
3. 应用
4. 传播
5. 观测
6. 固化或回滚

要求：

- 高风险变更必须留痕
- 变更后必须有观测与验收
- 配置发布不得跳过回滚路径

## 6. Feature Flag 与 Kill Switch

- 新协议、新 Provider、新能力家族默认通过 Flag 受控开放
- 高风险能力必须具备快速关闭能力
- Kill Switch 必须独立于业务页面，不依赖临时人工脚本

## 7. 漂移治理

必须持续对齐以下四类来源：

- 文档
- 控制面持久化配置
- 环境变量 / 本地配置
- 部署脚本与 Release 清单

## 8. 工作区与发布环境边界

### 8.1 工作区边界

源码工作区允许使用 sibling 相对路径 SDK 依赖，服务于本地联调和跨仓库协作。

### 8.2 发布边界

Release 环境与 GitHub Release 清单必须使用不可变 GitHub 地址，不得沿用本地相对路径依赖。

### 8.3 发布前一致性校验

发布前必须显式比对：

- `sdkwork-core`
- `sdkwork-ui`
- `sdkwork-appbase`
- `sdkwork-im-sdk`

校验要求：

- 本地仓库 HEAD 与远程计划引用版本一致
- Release 清单中的依赖地址与远程 Git 真值一致
- 如不一致，必须阻断提交、push 和发布

## 9. 设计红线

- 禁止把缓存或派生视图当作唯一真值。
- 禁止高风险配置变更无审核、无观测、无回滚。
- 禁止工作区相对路径依赖进入 Release 清单。
- 禁止控制面配置、环境变量和部署脚本长期漂移不对齐。

## 10. 评估标准

- 是否能明确配置从哪里来、谁能改、何时生效、如何回滚。
- 是否能避免“本地能跑、发布即漂移”的配置事故。
- 是否能在多租户、多环境、多 Cell 下维持同一套治理语言。
