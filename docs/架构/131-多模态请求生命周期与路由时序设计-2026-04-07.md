# 多模态请求生命周期与路由时序设计

> 日期：2026-04-07
> 范围：gateway、compatibility、provider execution、usage / billing

## 1. 文档目标

本文档用于冻结 `sdkwork-api-router` 当前网关的数据平面时序模型，重点回答四个问题：

1. 不同协议家族如何汇聚到统一执行主链路。
2. `stateful` 与 `stateless` 两种运行形态在生命周期上的本质差异。
3. 多模态、流式、长任务、资源型接口分别如何收口。
4. 用量、计费、诊断、回退在什么阶段产生和结束。

本文档只描述当前仓库可验证的主链路设计，并把目标标准明确标为“目标状态”。

## 2. 统一主链路

统一请求生命周期分为八段：

1. `Ingress`
2. `Protocol Normalization`
3. `Identity Resolution`
4. `Policy / Quota Admission`
5. `Routing Decision`
6. `Provider Execution`
7. `Usage / Billing Finalization`
8. `Response / Stream Completion`

无论入口是 OpenAI、Anthropic 还是 Gemini，最终都必须回收到这条统一主链路，而不能形成多套彼此脱节的执行系统。

## 3. 两条必须明确区分的主路径

### 3.1 Stateful Standalone Gateway

独立服务形态下，当前网关主线是 `stateful`：

- 接收 Gateway API Key
- 连接本地 Admin Store
- 执行本地鉴权、租户/项目解析、配额与路由决策
- 产生日志、快照、决策证据
- 在本地完成 usage / billing 收口

这是当前“完整平台治理链路”的主形态。

### 3.2 Stateless Runtime

当前仓库也支持 `stateless` Runtime / library 形态：

- 侧重 relay / compatibility / lightweight runtime embedding
- 可以转译或透传到单上游运行时
- 不应被描述为已具备与 standalone 同等级的本地控制平面、商业化与审计治理能力

因此，以下描述都必须优先以 `stateful standalone` 为准，不能默认外推到 `stateless`：

- 本地配额
- 本地 billing events
- 本地 canonical account
- 本地 routing evidence

## 4. 协议汇聚模型

### 4.1 OpenAI Compatible 主链路

当前主入口包括：

- `/v1/chat/completions`
- `/v1/responses`
- `/v1/completions`
- `/v1/embeddings`
- `/v1/moderations`
- 大量资源型 `/v1/*` 家族

这些接口是数据平面的主入口，也是其他兼容协议最终汇聚的目标语义。

### 4.2 Anthropic Messages 汇聚模型

当前可验证兼容面包括：

- `/v1/messages`
- `/v1/messages/count_tokens`

当前设计原则：

- 兼容入口只承担协议翻译，不拥有独立路由系统
- `Authorization`、`x-api-key`、`anthropic-version`、`anthropic-beta` 等字段在入口层解析并保留必要上下文
- 请求最终汇聚到共享文本执行链路

### 4.3 Gemini 汇聚模型

当前可验证兼容面包括：

- `/v1beta/models/{model}:generateContent`
- `/v1beta/models/{model}:streamGenerateContent`
- `/v1beta/models/{model}:countTokens`

当前设计原则：

- `Authorization`、`x-goog-api-key` 与 `?key=` 等认证入口在兼容层处理
- Gemini 请求最终回收到共享执行与路由链路
- SSE 输出在兼容层重新封装为 Gemini 兼容事件帧

## 5. 当前主时序设计

### 5.1 文本生成类请求

以 `/v1/chat/completions` 与 `/v1/responses` 为代表，当前主路径是：

1. 入口路由接收请求
2. 解析模型、流式标记、协议上下文与请求选项
3. 解析调用身份
4. 执行 admission
5. 构造候选集并应用路由策略
6. 选定 Provider 并执行请求
7. 非流式请求在响应前完成 usage / billing 收口
8. 流式请求在流结束或明确中断后完成 usage / billing / trace 收口

其中身份解析与 admission 的口径必须区分运行形态：

- `stateful`：解析 Gateway API Key，并绑定租户 / 项目 / 环境
- `stateless`：仅解析 relay 所需上下文

### 5.2 资源型与创建型请求

当前 Gateway 已覆盖大量资源型接口，例如：

- files / uploads / containers
- assistants / threads / conversations
- vector stores / batches / fine tuning / evals
- videos

当前正确设计口径是：

- 资源标识符可以是上游返回的对象 ID
- 在 `stateful` 模式下，路由与计费不一定跟随上游资源 ID 变化，而是按请求模型、路由键或创建语义收口

这在当前 [Gateway API](/api-reference/gateway-api) 中已体现为：

- generation-style routes 保持按请求模型计费
- create-like routes 可以保留 route-key 型 Provider 选择，同时记录创建资源的 downstream reference

## 6. 多模态差异化时序

### 6.1 图像

图像链路的关键差异在于：

- 可能处理 multipart 输入
- 可能返回 URL、base64 或文件对象
- 计量维度可能是 `image_count` 或固定单价，而不是 token

当前 Billing 2.0 事件账本已支持图像维度计量，这是当前事实。

### 6.2 音频

当前音频家族包括：

- transcription
- translation
- speech
- voices / voice_consents

音频链路的关键差异在于：

- 输入或输出可能是二进制流
- speech 更接近内容流返回
- 需要更严格的 body size、content type 和 timeout 治理

### 6.3 视频

视频链路当前更接近“长任务 + 资源对象”模型：

- create
- retrieve
- content
- remix / edits / extend / character

视频链路的关键差异在于：

- 大对象
- 长耗时
- 更适合对象化与状态查询

### 6.4 音乐

音乐链路当前按资源型接口家族建模：

- create
- retrieve
- content
- lyrics

当前事实只有两层：

- 账本已支持 `music_seconds` 等维度
- 架构口径已将其纳入统一资源化能力家族

但更细的上游兼容范围、执行成熟度与计量细节，仍必须以兼容矩阵与参考文档为准，不能把“账本已识别”写成“全链路已成熟”。

## 7. 上传、异步任务与长作业

### 7.1 上传生命周期

当前上传主链路包括：

1. `POST /uploads`
2. `POST /uploads/{upload_id}/parts`
3. `POST /uploads/{upload_id}/complete`
4. `POST /uploads/{upload_id}/cancel`

上传本身不等于最终业务处理。上传完成后，还可能进入后续文件、向量、微调或视频处理链路。

### 7.2 长任务统一原则

对以下对象，应统一使用“任务 / 资源 / 状态查询”模型：

- batches
- fine tuning
- evals
- videos
- 部分文件与上传后处理流程

目标状态下，应继续强化统一异步语义；当前事实是资源化接口骨架已经存在。

## 8. 路由时序中的关键决策点

### 8.1 Identity Resolution

当前身份解析必须做到：

- 兼容协议不绕开统一身份入口
- Gateway 记录 `x-request-id` 关联信息
- 可选接收 `x-sdkwork-region` 作为地理亲和路由提示

### 8.2 Admission

在真正访问 Provider 之前，必须完成：

- 鉴权
- 权限
- rate limit
- quota
- 关键请求合法性校验

### 8.3 Candidate Selection

当前候选选择至少考虑：

- 模型支持性
- Provider 健康度
- 参数兼容性
- 区域提示与 geo-affinity
- 策略指定的优先级与排序

目标状态下，还应继续强化：

- 成本约束
- 延迟预测
- 数据策略过滤
- 缓存命中倾向

### 8.4 Fallback

当前 fallback 必须是可解释的，而不是隐式切换。至少需要能够回答：

- 为什么切换
- 从哪个候选切换到哪个候选
- 最终是否成功

这正是 `RoutingDecisionLog` 与 `ProviderHealthSnapshot` 的价值所在。

## 9. 流式收口模型

### 9.1 流式完成

流式响应完成时，必须依次收口：

- stream closed
- usage finalized
- billing finalized
- trace finalized

### 9.2 流式中断

流式中断至少要区分：

- 客户端取消
- Provider 中断
- 网关超时
- 内部错误

这些差异必须在日志、指标和诊断中可见，否则后续无法判断是上游问题、网关问题还是客户端问题。

## 10. 当前时序设计中的证据资产

当前可直接支撑生命周期解释的证据资产包括：

- `x-request-id`
- `RoutingDecisionLog`
- `ProviderHealthSnapshot`
- `/metrics`
- `billing/events`
- `billing/events/summary`
- `routing/snapshots`
- `routing/decision-logs`

这说明当前架构已经不是“请求进来后黑盒转发”，而是具备可解释生命周期证据链。

## 11. 性能预算与时序预算标准

| 阶段 | 当前要求 | 说明 |
|---|---|---|
| 身份解析 | 极短路径 | 避免为每次请求引入不必要的数据放大 |
| admission | 毫秒级内完成 | 不得成为流式首字节主要瓶颈 |
| 路由决策 | 子毫秒到毫秒级 | 允许依赖快照、缓存与健康视图 |
| Provider 首字节 | 持续追踪，不做主观承诺 | 受上游影响，但必须可观测 |
| usage / billing 收口 | 不得显著阻塞响应 | 尤其不得阻塞流式输出结束 |

## 12. 评估标准

### 12.1 主链路一致性评估

- 不同协议面是否最终汇聚到统一执行主链路
- 是否避免兼容协议拥有独立身份、独立配额、独立路由逻辑

### 12.2 stateful / stateless 准确性评估

- 是否明确区分 stateful 的本地治理链路与 stateless 的轻量 relay 链路
- 是否避免把 stateful 的 billing / quota / route evidence 误写成 stateless 的当前事实

### 12.3 多模态一致性评估

- 图像、音频、视频、音乐、文件、上传是否都遵循统一的 admission、routing、execution、finalization 模型
- 长任务是否具备清晰的资源化和状态查询语义

### 12.4 可运维性评估

- 是否能用 `x-request-id` 回溯完整生命周期
- 是否能定位 admission、routing、provider execution、stream finalize 四个阶段中的故障点
