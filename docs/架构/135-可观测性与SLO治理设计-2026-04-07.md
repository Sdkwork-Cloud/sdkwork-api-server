# 可观测性与SLO治理设计

> 日期：2026-04-07
> 范围：health、metrics、tracing、logs、decision log、runtime status、SLO、告警

## 1. 文档目标

本文档冻结 `sdkwork-api-router` 的可观测性与 SLO 治理标准，回答三个问题：

1. 当前仓库里哪些观测资产已经可以被验证。
2. 下一阶段必须执行的 SLO 口径是什么。
3. 如何把请求链、路由链、运行时链、商业链收敛成一条统一诊断闭环。

## 2. 当前可验证事实

### 2.1 当前已具备健康与指标入口

当前可验证事实包括：

- Gateway 与控制面服务已纳入 `health / metrics` 暴露体系
- 运维文档已明确把指标、健康检查和请求追踪作为标准运维入口

### 2.2 当前已具备请求关联能力

当前可验证事实包括：

- `x-request-id` 被作为主关联键
- 调用方传入的 request id 会被保留；未传入时由平台生成并回传

### 2.3 当前已具备路由证据资产

当前可验证事实包括：

- `RoutingDecisionLog`
- `ProviderHealthSnapshot`
- `routing/snapshots`
- `routing/decision-logs`

这意味着当前平台已经不只是记录 HTTP 访问日志，而是能解释为什么选了某个候选、候选健康如何、是否发生 fallback。

### 2.4 当前已具备运行时观测资产

当前 Admin API 已暴露：

- `extensions/runtime-statuses`
- `extensions/runtime-reloads`
- `extensions/runtime-rollouts`
- `runtime-config/rollouts`

### 2.5 当前已具备商业链观测资产

当前可验证事实包括：

- `billing/events`
- `billing/events/summary`
- `billing/accounts`
- `billing/account-holds`
- `billing/request-settlements`

这说明当前平台已经能把技术链路和商业链路建立事实关联。

## 3. 统一观测模型

平台可观测性必须同时覆盖四条主链：

1. 请求链
2. 路由链
3. 运行时链
4. 商业链

### 3.1 请求链

最小闭环：

- ingress
- auth
- quota
- route decision
- provider execution
- response / stream finalize

### 3.2 路由链

最小闭环：

- 候选集构造
- 健康快照
- 策略命中
- 选中候选
- fallback
- 最终执行结果

### 3.3 运行时链

最小闭环：

- extension installation
- extension instance
- runtime status
- reload
- rollout
- config rollout

### 3.4 商业链

最小闭环：

- usage 记录
- billing events
- hold / settlement
- pricing posture
- API key group / routing profile 影响面

## 4. 指标标准

### 4.1 数据平面指标

必须覆盖：

- 请求量
- 成功率 / 错误率
- 延迟分布
- TTFT
- 流式完成率
- Provider 调用量
- Provider 错误率
- Provider 超时率
- fallback 次数与成功率

### 4.2 控制平面指标

必须覆盖：

- Admin API 可用性
- Portal API 可用性
- 关键写接口成功率
- routing simulation 延迟
- API key / API key group 生命周期操作成功率
- runtime reload / rollout 成功率

### 4.3 商业指标

必须覆盖：

- usage 量级
- billing event 量级
- hold / settlement 结果分布
- pricing 发布情况
- 工作区与 API key group 的账务异常

## 5. tracing 与日志标准

### 5.1 高价值请求最低字段

每个高价值请求至少要能关联到：

- `x-request-id`
- protocol family
- tenant / project
- model / route key
- selected provider
- fallback provider
- error kind

### 5.2 当前优先依赖的结构化证据

当前应优先依赖以下结构化对象，而不是只依赖文本日志：

- `RoutingDecisionLog`
- `ProviderHealthSnapshot`
- runtime status
- billing events

## 6. SLO口径

### 6.1 SLO 的正确含义

本文档中的 SLO 是：

- 下一阶段必须执行的运营门槛
- 发布与容量规划门槛

而不是：

- 当前已对外承诺并长期稳定达成的 SLA

### 6.2 数据平面 SLO

建议至少定义：

- gateway availability
- non-streaming success rate
- streaming completion success rate
- fallback success rate
- provider timeout budget

### 6.3 控制平面 SLO

建议至少定义：

- admin API availability
- portal API availability
- routing simulation response time
- API key issuance success rate
- runtime reload / rollout success rate

### 6.4 商业链 SLO

建议至少定义：

- billing event write success rate
- account hold creation success rate
- request settlement finalize success rate
- pricing lifecycle synchronize success rate

## 7. 告警分级标准

`P0`

- gateway 全局不可用
- admin / portal 全局不可用
- usage / billing 收口大面积失败
- rollout 造成核心服务不可用

`P1`

- 单 Provider 大面积故障
- fallback 大量发生且成功率下降
- PostgreSQL / Redis 明显异常
- runtime reload / rollout 连续失败

`P2`

- 单模型异常
- 单租户热点
- 单工作区账务异常
- 单 Region / geo-affinity 偏斜

## 8. 标准诊断路径

标准诊断顺序应为：

1. 看 `health`
2. 看 `metrics`
3. 看 `logs`
4. 看 `trace`
5. 看 `RoutingDecisionLog`
6. 看 `ProviderHealthSnapshot`
7. 看 runtime status / rollout status
8. 看 billing events / account evidence

这条路径的目标是把技术故障与商业收口故障放在同一条可操作链路中。

## 9. Dashboard标准

### 9.1 平台总览

至少展示：

- QPS
- 延迟
- 错误率
- 流式完成率
- 热门模型
- 协议族分布

### 9.2 路由总览

至少展示：

- 策略命中分布
- fallback 分布
- Provider 健康状态
- Region 分布
- group / profile 影响面

### 9.3 商业总览

至少展示：

- usage
- billing events
- hold / settlement 概览
- pricing 版本与生效状态
- credits / balance / recharge 关键指标

## 10. 设计红线

- 禁止高价值请求没有 `x-request-id` 就进入不可回溯状态。
- 禁止只有 HTTP 访问日志而没有路由决策证据。
- 禁止只有错误计数而没有错误分类。
- 禁止把 SLO 目标直接写成当前稳定生产结论。

## 11. 评估标准

### 11.1 观测完备度

- 是否能同时观测请求链、路由链、运行时链、商业链。
- 是否能把技术异常与 billing / usage 异常关联起来。

### 11.2 可运维性

- 是否能通过 `x-request-id` 快速回溯链路。
- 是否能通过结构化证据而不是文本猜测完成诊断。

### 11.3 SLO治理成熟度

- 是否已定义并执行数据平面、控制平面、商业链三类 SLO。
- 是否已形成发布门槛、容量门槛和告警门槛。
