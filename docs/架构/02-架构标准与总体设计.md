# 架构标准与总体设计

## 1. 文档目标

定义 `sdkwork-api-router` 的总体架构模型、运行面、分层标准、状态真值模型、扩展路径与设计红线。

## 2. 总体架构结论

系统必须被理解为“多运行面平台”，而不是“一个网关可执行文件”。当前权威运行面为：

- `gateway-service`：数据平面入口
- `admin-api-service`：控制平面入口
- `portal-api-service`：自助门户入口
- `router-web-service`：公网静态站点与统一入口宿主
- `router-product-service`：一体化产品宿主
- `sdkwork-router-admin`、`sdkwork-router-portal`：主线前端产品
- `console/`：兼容与过渡外壳

## 3. Plane 模型

系统按六个主 Plane 和两个横切 Plane 设计：

- `Gateway Plane`：统一数据平面与 `/v1/*` 家族入口
- `Compatibility Plane`：Anthropic、Gemini 及未来协议兼容层
- `Control Plane`：Admin 治理、配置、运营与运维
- `Portal Plane`：客户自助、账户、密钥、账务与工作区
- `Provider Execution Plane`：Provider 适配器、扩展运行时、连接器监督
- `State Plane`：配置、存储、缓存、密钥、审计、用量、计费、结算

横切 Plane：

- `Security Plane`
- `Observability / Ops Plane`

## 4. 分层标准

代码必须长期维持以下分层：

- `interface`：协议入口、参数校验、认证边界、响应整形
- `app`：用例编排、跨域流程、工作流
- `domain`：规则、不变量、策略、能力模型
- `infra`：storage、provider、runtime、secret、cache、host

约束：

- `domain` 不依赖 `interface`
- `app` 不拥有传输协议细节
- `interface` 不拥有业务真值
- `infra` 不反向定义产品规则

## 5. 运行模式标准

系统当前必须同时容纳两类维度：

### 5.1 服务部署维度

- `standalone services`：`gateway/admin/portal/web` 独立部署，适合生产与弹性扩缩容
- `product host`：由 `router-product-service` 统一装配，适合本地、边缘与简化私有化交付

### 5.2 执行语义维度

- `stateful standalone`：依赖本地控制面真值执行租户、项目、路由、配额、计费、审计链路
- `stateless runtime`：围绕已配置上游运行时做轻量 relay 与兼容执行，适合 bootstrap、兼容代理、轻部署

## 6. 状态真值模型

系统必须区分三种真值：

- `Durable Truth`：租户、项目、Provider、模型、策略、配额、账本、订单、结算、扩展安装等持久化真值
- `Cache Truth`：能力缓存、路由热缓存、健康快照辅助状态、分布式锁；可失效、可重建、不可当唯一事实来源
- `Query Truth`：Dashboard、汇总报表、运营视图；允许延迟一致，但必须可追溯

## 7. 目标部署演进

目标态必须支持：

- 本地单机开发
- 共享 SaaS Cell
- Dedicated Cell
- 私有化 Cell
- 多 Region 演进

扩展顺序必须是：先稳定单 Cell 架构，再做多 Cell；先稳定数据与配置治理，再做跨 Region。

## 8. 设计红线

- 禁止每个协议各自维护独立路由与计费内核
- 禁止缓存成为唯一真值来源
- 禁止控制平面逻辑侵入网关热路径
- 禁止把 Provider 细节写散到页面、handler、脚本与 SQL 层
- 禁止一体化产品宿主吞并所有内部边界

## 9. 评估标准

### 9.1 完整性

- 是否覆盖数据平面、控制平面、门户、自助、执行平面与状态平面

### 9.2 清晰度

- 是否能明确回答“请求如何进来、如何路由、如何计费、如何审计、如何发布”

### 9.3 可扩展性

- 新增协议、Provider、存储后端、部署形态时是否只影响局部边界

### 9.4 可交付性

- 是否能被 `bin/*`、`scripts/dev/*`、部署文档与兼容矩阵共同支撑，而不是只停留在理念层
