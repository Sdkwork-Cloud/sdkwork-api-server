# 技术选型与可插拔策略

## 1. 文档目标

说明当前技术栈、选型理由、替代方案取舍、可插拔机制与发布依赖治理标准。

## 2. 选型原则

- 热路径优先：先保障并发、稳定性、内存安全与可观测性
- 边界优先：技术栈必须服务于分层与模块边界，而不是反过来
- 交付优先：本地开发、私有化安装、生产发布必须共用一套工程模型
- 可插拔优先：Provider、存储、缓存、Secret、扩展运行时必须能局部替换

## 3. 后端核心技术栈

- `Rust + Cargo workspace`：支撑高并发、强类型、长生命周期演进
- `Axum`：作为 HTTP 与 API 接口层的主框架
- `Tokio`：承担异步执行、流式链路与后台任务
- `Pingora`：承担公网静态资源与统一入口宿主
- `OpenAPI` 契约：承担接口暴露与文档一致性

取舍结论：

- 不选动态语言作为网关热路径主实现
- 不把控制面、门户、Provider 执行与网关全部塞进一个单体框架

## 4. 前端、文档与桌面技术栈

- `React + TypeScript + pnpm`：Admin 与 Portal 主栈
- `Tauri`：桌面宿主与本地交付能力
- `VitePress`：文档站点

取舍结论：

- 浏览器应用是主交付面
- 桌面宿主服务于增强交付，不取代主线浏览器产品

## 5. 存储、缓存与密钥技术栈

- `SQLite`：本地开发、轻部署、单机私有化
- `PostgreSQL`：生产与多租户主持久化
- `memory cache`：本地与开发默认缓存
- `Redis`：生产缓存、热配置、并发协调辅助设施
- Secret backend：本地加密文件、系统密钥后端、数据库加密策略

取舍结论：

- 缓存只做加速和并发协调，不做唯一真值
- `SQLite` 与 `PostgreSQL` 必须维持双轨可运行，而不是只保留一种环境

## 6. Provider 与扩展运行时策略

标准接入模型固定为：

- `builtin`：内建 Provider，适合主线稳定能力
- `connector`：外部进程或外部服务接入，适合异构生态
- `native-dynamic`：受信动态库扩展，适合高性能与受控私有扩展

标准 Provider 当前基线为：

- `openai`
- `openrouter`
- `ollama`

取舍结论：

- 禁止业务代码手写零散 HTTP 调用接入上游
- 新增 Provider 必须通过适配器、注册、能力声明、运行时治理进入系统

## 7. 替代方案与取舍

- 单体 Node/Java 后台：上手快，但不利于热路径性能与边界治理
- 每个协议单独网关：短期快，长期会造成协议、路由、计费三套分叉
- 只做 OpenAI 兼容：实现轻，但无法支撑 Claude Code、Gemini CLI 与后续生态

最终取舍是：用 Rust 统一热路径与治理内核，用 React 系产品化界面，用扩展运行时吸收生态差异。

## 8. 发布依赖治理

源代码工作区与 Release 环境必须分开对待：

- 源代码工作区允许使用 sibling repo 的相对路径依赖，提高联调效率
- Release 与 GitHub 发布环境必须把外部 SDK 依赖固化为 GitHub 不可变地址，而不是继续引用本地相对路径
- `sdkwork-core`、`sdkwork-ui`、`sdkwork-appbase`、`sdkwork-im-sdk` 等独立仓库若本地状态脏、远端不可验证、版本或提交不一致，发布必须阻断
- 这样做的目标是减少对 npm 中央仓库的频繁发布依赖，同时保证 release 可复现

## 9. 评估标准

- 技术栈是否服务于高并发、低耦合、易扩展目标
- 新增 Provider、缓存后端、Secret 后端、部署形态时是否无需推倒重来
- Release 是否能够在脱离本地工作区后仍然稳定构建与运行
