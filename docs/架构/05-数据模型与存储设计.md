# 数据模型与存储设计

## 1. 文档目标

定义核心实体、真值分层、存储职责、多租户隔离、生命周期边界与评估标准。

## 2. 核心实体分层

### 2.1 身份与工作区

- `Tenant`
- `Project`
- `AdminUser`
- `PortalUser`
- `GatewayApiKey`
- `ApiKeyGroup`

### 2.2 Provider、目录与路由

- `ProxyProvider`

- `RoutingPolicy`
- `RoutingProfileRecord`
- `ProjectRoutingPreferences`

### 2.3 用量、配额、计费

- `UsageRecord`

### 2.4 商业化与交易

- 订单、充值、支付、退款、账本、结算、对账对象

### 2.5 扩展与运行时

- `ExtensionInstallation`
- `ExtensionInstance`
- `ExtensionRuntimeStatusRecord`

## 3. 真值模型

- `Durable Truth`：身份、目录、路由、配额、账本、订单、扩展安装等持久化对象
- `Cache Truth`：能力缓存、热点目录、健康快照辅助状态、分布式锁
- `Query Truth`：Dashboard、汇总报表、运营视图、结算视图

原则：

- Durable Truth 可审计
- Cache Truth 可失效
- Query Truth 可重建

## 4. 存储职责

- `SQLite`：本地、开发、轻量部署、单机私有化
- `PostgreSQL`：生产、共享 SaaS、多租户、复杂查询
- `memory cache`：本地默认缓存
- `Redis`：生产缓存与并发协调
- Secret backend：密钥与凭据隔离存储

## 5. 多租户与一致性

- 所有工作区真值都必须以 `tenant`、`project` 为一等边界
- `api key`、`routing`、`quota`、`usage`、`billing`、`settlement` 必须保留租户与项目归属
- 路由健康、计费账本、结算证据必须可按租户、项目、Provider、时间窗口查询
- 汇总视图允许延迟一致，但账本与结算证据不能丢失

## 6. 生命周期与成本边界

- 配置与主数据：长期保留，强审计
- 用量、计费、结算：长期保留，支持分区、归档、追溯
- 媒体、文件、上传、视频、音乐等大对象：必须支持冷热分层与对象存储迁移
- 缓存与临时任务状态：允许 TTL 与回收

更细的保留、归档、压缩、删除策略见 [140-数据生命周期与成本治理设计-2026-04-07](./140-数据生命周期与成本治理设计-2026-04-07.md)。

## 7. 设计红线

- 禁止把缓存当数据库用
- 禁止把报表视图当权威账本
- 禁止跨租户共享未显式隔离的密钥、账本与结算对象
- 禁止把媒体大对象长期塞进热数据库主表

## 8. 评估标准

- 数据模型是否覆盖身份、路由、用量、计费、交易、扩展六类核心域
- 存储设计是否同时适配本地、生产、私有化三种环境
- 生命周期策略是否兼顾性能、成本、审计与合规
