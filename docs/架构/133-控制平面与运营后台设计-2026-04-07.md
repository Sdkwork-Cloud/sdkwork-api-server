# 控制平面与运营后台设计

> 日期：2026-04-07
> 范围：admin-api-service、portal-api-service、admin app、portal app

## 1. 文档目标

本文档用于冻结 `sdkwork-api-router` 当前控制平面与自助门户的真实边界，确保文档能准确回答：

1. 当前 Admin 和 Portal 分别负责什么。
2. 当前控制平面已具备哪些真实治理能力。
3. 哪些能力已进入商业化与运营工作流。
4. 哪些部分属于下一阶段的体验深化，而不是当前缺失的基础设施。

## 2. 控制平面定位

控制平面不是“附属管理页”，而是整个平台的治理大脑。它统一治理：

- tenant / project / environment
- API key / API key group
- provider / channel / model / credential
- routing policy / routing profile / routing preference / snapshot / simulation
- usage / billing / pricing / quota / commercial account
- marketing / coupon / campaign
- extension installation / instance / runtime status / reload / rollout
- runtime config rollout

这说明当前产品已不再等同于“轻量代理 + 设置页”。

## 5. 关键运营流程

### 5.1 Provider 上线流程

1. 创建 channel / provider
2. 绑定 model 能力
3. 绑定 credential
4. 创建或更新 extension installation / instance
5. 查看 runtime status 与 health snapshot
6. 调整 routing policy / profile / preference
7. 通过 simulation 验证
8. 执行 reload / rollout

### 5.2 API Key Group 路由发布流程

1. 创建 routing profile
2. 将 profile 绑定到 API key group
3. 在 Portal 或 Admin 中查看 compiled snapshot
4. 使用 preview / simulation 验证效果
5. 通过 decision log 验证真实流量结果

### 5.3 定价发布流程

1. 创建 pricing plan
2. 维护 pricing rate
3. clone 下一版本
4. schedule 生效时间
5. publish 或 retire
6. Portal 侧读取当前工作区可见定价姿态

### 5.4 营销与优惠流程

1. 创建 coupon template
2. 绑定 campaign / budget / code
3. Portal 侧执行 validation / reservation / confirm / rollback
4. 通过 reward-history / redemptions / rollbacks 完成回溯

### 5.5 扩展运行时发布流程

1. 安装 extension package
2. 创建 installation
3. 创建 instance
4. 查看 runtime status
5. 局部 runtime reload
6. 跨节点 extension runtime rollout
7. 必要时执行 config rollout 或回滚

## 6. 组件化与边界设计原则

### 6.1 Admin 侧原则

- 工作区容器、表单、详情面板、日志视图、操作条分离
- API client 与 UI 表现逻辑分离
- 高价值治理域以独立业务包承载

### 6.2 Portal 侧原则

- 工作区业务域按 `routing / billing / api-keys / account / marketing` 拆分
- 页面层主要做编排，不做跨域硬编码
- 后端工作区边界优先于前端展示边界

### 6.3 权威边界原则

- Admin 是系统真值写入口
- Portal 是工作区自助入口
- Gateway 不直接承载控制平面写逻辑
- `console/` 不再定义主线产品边界

## 7. 下一阶段重点优化方向

下一阶段要补的是“产品体验深度”，不是“有没有控制平面”：

- Provider onboarding 体验
- 定价、账务、路由、告警之间的联动闭环
- 扩展运行时与灰度发布的一体化操作台
- API key group、routing profile、pricing posture 的跨页面协同体验

## 8. 评估标准

### 8.1 完备度评估

- Admin 是否覆盖目录、路由、商业、扩展、发布五类核心治理域
- Portal 是否覆盖工作区路由、账务、营销、API 资产四类核心自助域

### 8.2 边界清晰度评估

- Admin / Portal / Gateway / Console 四者的责任是否清晰
- 是否避免把 Portal 误写成只负责发 Key

### 8.3 运营效率评估

- 是否能通过控制平面完成 Provider 上线、路由发布、定价发布、扩展发布
- 是否能通过工作区入口完成自助 key、group、routing、billing 与 coupon 管理
