# 支付管理后台与交易运维实现设计

## 1. 目标

在既有《35-企业级支付交易闭环架构设计-2026-04-07》基础上，将支付闭环补齐到可运营、可审计、可对账、可退款、可接真实渠道的交付级形态，重点覆盖以下落地面：

1. `sdkwork-api-interface-admin` 提供支付方式管理、支付尝试、退款、Webhook Inbox、对账运行的真实管理接口。
2. `sdkwork-router-admin` 增加支付方式管理、凭据绑定、订单操作、Webhook 处理观测、对账运维等可视化能力。
3. 前端实现遵循高内聚低耦合拆分，不再把支付闭环继续堆进单文件页面。

## 2. 运营闭环范围

### 2.1 支付方式管理

- 管理支付方式基础元数据：
  - `payment_method_id`
  - `display_name`
  - `provider`
  - `channel`
  - `mode`
  - `enabled`
- 管理支付策略：
  - `capability_codes`
  - `supported_currency_codes`
  - `supported_country_codes`
  - `supported_order_kinds`
  - `callback_strategy`
  - `webhook_path`
  - `webhook_tolerance_seconds`
  - `replay_window_seconds`
  - `max_retry_count`
- 管理非密钥配置：
  - `config_json` 仅保存非敏感 provider 配置
  - 密钥必须通过 `payment_method_credential_bindings` 绑定到加密凭据系统

### 2.2 订单与支付尝试运营

- 基于 `CommerceOrderRecord` 展示最新订单。
- 对单个订单加载：
  - `CommercePaymentAttemptRecord[]`
  - `CommerceRefundRecord[]`
- 对运营人员暴露：
  - 订单状态
  - 结算状态
  - 最新支付方式
  - 可退金额
  - 支付尝试的 provider reference / session / capture / refund 进度

### 2.3 Webhook 安全运营

- 展示 webhook inbox 原始记录：
  - provider
  - payment_method_id
  - provider_event_id
  - dedupe_key
  - processing_status
  - retry_count / max_retry_count
  - next_retry_at_ms
- 展示 delivery attempts：
  - response_code
  - error_message
  - started_at_ms / finished_at_ms
- 支持快速识别：
  - 重放攻击窗口内重复事件
  - 验签失败
  - 死信重试积压

### 2.4 退款与财务收口

- 对单个订单发起真实退款。
- 支持：
  - 全额退款
  - 部分退款
  - 指定 `payment_attempt_id`
  - 自定义 `idempotency_key`
- 退款必须驱动：
  - provider refund record
  - order `refundable_amount_minor`
  - order `refunded_amount_minor`
  - settlement 状态向 `partially_refunded` / `refunded` 推进

### 2.5 对账运维

- 以 provider 和 payment_method 为维度手工触发 reconciliation run。
- 定义时间窗：
  - `scope_started_at_ms`
  - `scope_ended_at_ms`
- 产出：
  - `CommerceReconciliationRunRecord`
  - `CommerceReconciliationItemRecord`
- 管理端可看到：
  - mismatch / missing / amount mismatch 等差异
  - 本地预期金额与 provider 金额
  - 差异明细 JSON

## 3. Admin API 设计

### 3.1 新增路由

- `GET /admin/commerce/orders/{order_id}/payment-attempts`
- `GET /admin/commerce/orders/{order_id}/refunds`
- `POST /admin/commerce/orders/{order_id}/refunds`
- `GET /admin/commerce/webhook-inbox`
- `GET /admin/commerce/webhook-inbox/{webhook_inbox_id}/delivery-attempts`
- `GET /admin/commerce/reconciliation-runs`
- `POST /admin/commerce/reconciliation-runs`
- `GET /admin/commerce/reconciliation-runs/{reconciliation_run_id}/items`

### 3.2 设计原则

- admin 侧不重新实现支付逻辑，只复用 `sdkwork-api-app-commerce` 的应用服务。
- 所有错误统一走 `admin_commerce_error_response`，避免出现支付域错误泄漏成不一致的 HTTP 表达。
- OpenAPI 与运行时路由保持同源定义，避免文档漂移。

## 4. 管理端前端模块拆分

### 4.1 页面容器

- `sdkwork-router-admin-commercial/src/index.tsx`
  - 负责总览数据组合
  - 管理订单审计抽屉
  - 加载支付方式列表
  - 拼装补充运营区块

### 4.2 支付闭环区块

- `paymentMethodManagerSection.tsx`
  - 支付方式管理表格
  - 新建/编辑
  - 删除
  - 打开凭据绑定
- `paymentMethodDialog.tsx`
  - 支付方式主配置编辑
- `paymentCredentialBindingsDialog.tsx`
  - 绑定到加密凭据
  - 区分 `usage_kind`
- `paymentOrderOperationsSection.tsx`
  - 订单操作与退款
  - 查看 attempts / refunds
- `paymentRefundDialog.tsx`
  - 发起退款
- `paymentWebhookInboxSection.tsx`
  - webhook inbox 与 delivery attempts
- `paymentReconciliationSection.tsx`
  - 对账运行发起与差异明细查看
- `paymentShared.tsx`
  - 支付模块通用对话框字段、确认框、状态映射、列表解析工具

### 4.3 拆分原则

- 页面容器不承载具体业务表单细节。
- 每个 section 对应单一运维主题。
- 每个 dialog 对应一个原子运营动作。
- 通用字段与通用工具抽到共享文件，避免复制粘贴。

## 5. 性能与可维护性约束

### 5.1 加载策略

- 支付方式使用页面局部拉取，不进入全局 workbench 快照，避免 admin 首页全局载荷继续膨胀。
- attempts / refunds / delivery attempts / reconciliation items 采用选中后按需加载。
- 对账 items 不预取，防止大窗口扫描后前端一次装载过多差异行。

### 5.2 边界控制

- 全局快照继续承担“总览型”数据。
- 支付运维子模块承担“局部详情型”数据。
- 通过局部 fetch 降低模块耦合，避免任何支付子表变化迫使整个管理台刷新。

## 6. 安全约束

- `config_json` 不允许承载密钥。
- 密钥只通过 `credential bindings` 间接引用加密凭据。
- webhook 安全观测必须保留：
  - signature header
  - dedupe key
  - retry count
  - processing status
- 退款入口必须支持幂等键，避免重复退款。

## 7. 部署落地要求

- 生产环境至少配置一条真实支付方式，例如 Stripe Checkout。
- 必须完成：
  - API 凭据录入
  - payment method 与 credential binding 绑定
  - webhook 回调地址公开暴露
  - webhook secret 绑定
  - 退款密钥绑定
- 上线前统一执行：
  - Rust 后端编译与集成验证
  - Admin 前端 typecheck
  - 支付尝试、webhook、退款、对账回归
