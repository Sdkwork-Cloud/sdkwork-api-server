# SDKWork API Router 架构标准

## 1. 文档目标

本目录用于冻结 `sdkwork-api-router` 的当前事实、目标架构、实施红线与评估标准，覆盖：

- 需求分析、产品设计、功能边界与业务流程
- 技术架构、模块规划、组件化设计与实现逻辑
- 性能、安全、测试、安装、部署、发布、回滚与验收

本体系同时满足两件事：

- 精确描述仓库已存在、可由代码与参考文档直接验证的事实
- 定义对标 `OpenRouter`、`new-api` 后的目标标准，但不把目标态误写成现状

## 2. 当前产品定义

`sdkwork-api-router` 不是单厂商代理，也不是单一 OpenAI 兼容转发层，而是面向企业与开发者的多协议、多模态 AI API Router 平台。当前冻结口径如下：

- 后端运行面：`gateway-service`、`admin-api-service`、`portal-api-service`、`router-web-service`、`router-product-service`
- 前端运行面：`apps/sdkwork-router-admin`、`apps/sdkwork-router-portal`、`console/`
- 网关运行模式：`stateful standalone` 与 `stateless runtime`
- 协议兼容面：`OpenAI Compatible`、`Anthropic Messages Compatible`、`Gemini Compatible`
- 能力家族：文本、图像、音频、视频、文件、上传、向量、批处理、评测、实时会话、Agents 类 API；`music` 采用统一资源化能力口径，执行真值以兼容矩阵与参考文档为准
- 控制面对象：`tenant / project / api key / api key group / provider / channel / model / routing policy / quota / usage / billing / extension runtime`
- Provider 模型：`builtin / connector / native-dynamic`

## 3. 事实冻结规则

- 文档中的“当前”“已落地”“已具备”，仅指可由 `Cargo.toml`、`services/*`、`apps/*/packages`、`README*`、`docs/reference/repository-layout.md`、`docs/api/compatibility-matrix.md`、`docs/reference/api-compatibility.md`、`bin/*`、`scripts/dev/*` 直接验证的事实。

## 5. 当前架构结论

- 平台已形成“高并发网关 + 控制平面 + 自助门户 + 统一产品宿主 + 公网交付宿主”的多运行面结构，而不是单体转发脚本。
- 协议兼容层必须汇聚到同一条身份、路由、配额、计费、审计主链路，禁止每种协议形成第二套路由内核。
- Rust workspace 分层、React 模块化前端、Tauri 宿主、Pingora 公网宿主与 `bin/*` 生命周期脚本已经构成当前交付底座。
- 存储与缓存已形成 `SQLite / PostgreSQL / memory / Redis / secret backend` 双轨工程模式，但不同部署形态的成熟度仍需分开评估。
- Admin、Portal、Commerce/Billing/Settlement 子域已进入真实产品边界，下一阶段重点不是补“有没有”，而是补“治理深度、操作闭环与发布标准”。
- 发布体系必须区分“源码工作区的相对路径依赖”和“Release 环境的 GitHub 不可变依赖”；任一 sibling SDK 仓库与远端版本不一致、工作区脏状态未清理、远端不可验证时，不应放行发布。

## 6. 文档质量要求

- 用词尽量短，但不牺牲边界、例外、约束和评估标准。
- 每篇文档都必须同时回答：系统是什么、现在做到什么、目标做到什么、为什么这样设计、如何判断达标。
- 对当前事实与目标态必须显式区分，避免“空洞愿景文档”和“把现状吹成目标态”两类错误。
