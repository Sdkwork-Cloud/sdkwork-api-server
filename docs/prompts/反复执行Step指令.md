# SDKWork API Router 反复执行 Step 指令

## 使用说明

- 每次继续推进 `sdkwork-api-router` 时，直接把下面整段 `Prompt 正文` 原样输入给执行模型。
- 这是可重复输入的连续执行母提示词。重复输入同一份内容时，模型必须基于当前仓库真实状态续跑，而不是重置。
- 目标是短、硬、可续跑：持续把 `docs/step/00-13` 全部做完、验完、回写完、发布说明更完，最终交付可完整商业化的应用。

## Prompt 正文

```md
你现在负责持续推进 `sdkwork-api-router`。你的任务不是给建议，而是基于当前仓库真实状态持续执行，直到：

1. `docs/step/00-13` 全部闭环。
2. 每个 step 的 `8.3`、`8.6`、`91`、`95`、`97`、`98` 都完成。
3. `docs/架构/130-142` 对应能力被真实实现或有明确发布限制说明。
4. `docs/review/`、`docs/release/`、`docs/step/`、`docs/架构/` 一致。
5. 应用达到可商业化交付、可部署、可回滚、可运维、可持续优化的标准。

## 一、最高依据

每次先读取并对齐：

- `docs/step/README.md`
- `docs/step/00-13`
- `docs/step/90-98`
- `docs/架构/README.md`
- `docs/架构/01-09`
- `docs/架构/130-142`
- `docs/review/` 下最新执行卡、复盘、审计、波次验收、架构兑现文档
- `docs/release/README.md`
- `docs/release/CHANGELOG.md`
- `docs/release/` 下最新 `Unreleased` 与版本日志
- 当前代码、测试、脚本、`bin/`、`scripts/`、`services/`、`apps/`、`crates/`、`git status`

若文档、代码、review、release 记录冲突：

- 先区分“当前事实”和“目标状态”
- 不允许把目标状态写成已落地事实
- 必须在本轮修正到一致
- 若 `docs/step`、`docs/架构`、`docs/review`、`docs/release` 本身已不足以支撑事实推进，先修文档系统，再继续实施

## 二、串并行规则

- step 级必须串行：`00 -> 01 -> 02 -> 03 -> 04`
- `13` 必须作为最终总收口 step
- `04` 未闭环前，不允许大规模展开 `05-12`
- 波次未通过 `93`，不得进入下一波次
- 可并行的仅限：当前已解锁 step 内车道、当前波次已满足依赖的切片、验证车道、文档回写车道、`Release-Truth Lane`
- 不得并行的包括：协议权威字段、路由主时序、权限模型、发布结论、release notes 基线

`11-13` 必须单独设置 `Release-Truth Lane`，专门核对：

- `/docs/release`
- `sdkwork-core`
- `sdkwork-ui`
- `sdkwork-appbase`
- `sdkwork-im-sdk`
- GitHub 不可变依赖
- release notes 基线

任一真值不一致，按 `B3` 处理。

## 三、每轮固定循环

每轮都按以下顺序执行：

1. 恢复状态：判定当前波次 / step、完成度、双运行态成熟度、最大 `B2/B3`
2. 审计当前 step：核对目标、`8.3`、`8.6`、`91`、`95`、`97`、`98`、下一步准入条件
3. 自我思考：至少给出 `Top 3` 候选动作，优先收口当前 step、清阻塞、提升商业化与可验证性，选择综合收益最高项
4. 若连续 `2` 轮没有显著减少未闭环能力、阻塞或验收缺口，自动转“阻塞清除模式”，优先修边界、依赖、验证资产
5. 批量实现：只在当前已解锁 `ready set` 内批量补代码、脚本、配置、文档骨架、测试资产
6. 批量自检：编译 / 构建 / smoke / 关键脚本 / 高风险回归 / `B2/B3`
7. 逐 step 深验收：对触达 step 逐个完成测试、验证、`91`、`95`、`97`、`98`
8. 波次验收：满足条件即执行 `93`，未通过则回修当前波次
9. release / changelog：只要有真实进展，立即更新 `docs/release/CHANGELOG.md` 与对应版本或 `Unreleased` 日志
10. 继续执行：`00-13` 未闭环就继续下一轮；全部闭环后转持续优化

补充规则：

- “先批量实现，再逐 step 深验收”只适用于当前已解锁 `ready set`
- 不允许为批量速度突破 step 依赖
- 不得把 changelog 留到最后统一补

## 四、step 硬完成定义

任一步只有同时满足以下条件，才能标记为完成：

- 实现已落地
- 测试已完成
- `8.3` 成立
- `8.6` 全部兑现
- `91` 达标
- `95` 通过
- `97` 完成
- `98` 完成
- 下一步准入条件成立

只要其中一项不满足，该 step 只能标记为“进行中”。

## 五、Change Log 与版本规则

每轮迭代后都必须更新 `/docs/release`。每次至少写明：

- 日期
- 版本号
- 当前波次 / step
- 本轮主执行模式
- Top 3 候选动作与最终选择理由
- 已兑现的架构能力
- 代码 / 测试 / 文档 / 脚本变更摘要
- 验证结果
- 已知风险与限制
- 下一轮入口

版本号建议：

- `Patch`：缺陷修复、测试补强、文档回写、脚本治理、局部收口
- `Minor`：完成重要 step 或完整波次
- `Major`：正式商业化大版本或重大协议 / 架构 / 交付模型变化

若当前版本无法证明已成功发布：

- 必须保留 `Unreleased`
- 未成功发布版本的变更必须并入下一次成功 release
- `/docs/release` 不完整时，必须从最近一次成功发布回溯 Git 历史补全 changelog

## 六、每轮输出必须包含

- 当前波次 / 当前 step
- 当前主执行模式
- 当前 step 完成度
- 当前波次完成度
- 本轮主动作与选择理由
- 本轮实际完成内容
- 改动的代码 / 测试 / 文档 / 脚本
- 已兑现 / 未兑现的 `8.6` 能力
- 是否完成 `docs/架构` 回写
- 是否完成 `docs/release` 更新
- 当前剩余 `B2/B3`
- 下一轮继续做什么

## 七、禁止事项

禁止：

- 停留在计划或空泛总结
- 跳过测试、review、架构回写、changelog
- 把局部完成当整体完成
- 把目标状态写成当前事实
- 为了并行速度破坏 step 依赖
- 在 release 产物中保留本地 sibling 相对路径依赖

## 八、最终停止条件

只有同时满足以下条件，才允许把主线视为完成：

- `docs/step/00-13` 全部闭环
- `docs/架构/130-142` 对应能力都已实现或有明确发布限制
- `docs/review/`、`docs/release/`、`docs/架构/`、`docs/step/` 已一致
- `stateful standalone` 与 `stateless runtime` 都有独立可信结论
- 应用达到可商业化交付标准

若不满足，则继续执行，不得停止。
```
