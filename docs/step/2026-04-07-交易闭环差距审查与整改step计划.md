## 3. 当前已确认落地能力

以下能力已经可以确认存在，不应在下一轮整改中被误判为“完全未实现”：

- 订单报价、下单、价格快照、优惠券预占与结算副作用快照已经存在，订单层已保存 `pricing_snapshot_json`。
- 真实支付渠道已接入 1 条：Stripe。`payment_attempt.rs` 已通过 Stripe Checkout Session 创建真实支付尝试，且保存 provider session / payment intent / checkout_url。
- `PaymentAttempt` 已落库，支持按订单查询、幂等键去重、尝试序号递增、支付成功/失败/过期状态更新。
- `refund.rs` 已支持退款发起、幂等键、全额/部分退款、Stripe Refund API、退款完成后的订单与尝试状态推进。
- `webhook.rs` 已支持 Stripe webhook 验签、dedupe inbox、delivery attempts、失败重试计划、dead-letter 状态记录。
- `reconciliation.rs` 已支持创建对账 run、扫描本地 attempts / refunds、调用 Stripe 查询源事实、写入 discrepancy items。
- Admin 后端已提供订单维度的 attempts / refunds、webhook inbox、delivery attempts、reconciliation runs / items。
- `sdkwork-router-admin` 已落地支付方式管理、密钥绑定、订单支付操作、webhook inbox、对账 runs 的分区式页面模块。
- 管理端当前对支付运维数据采用“商业页局部拉取，不塞进全局快照”的策略，这是一种当前阶段的性能折中，而不是单纯遗漏。

## 4. 核心审查结论

当前系统已经具备“交易闭环的骨架”，但距离“商业化运营标准”仍有明显差距，且存在至少 3 类高优先级问题：

1. **接口契约与架构文档不一致**：Admin API、Portal API 都存在缺口与旧路径残留，文档、OpenAPI、运行时路由还没有完全收口到同一真相源。
2. **生产流与模拟流没有完全隔离**：Portal 侧仍保留可直接推进订单结算的旧接口，真实支付闭环存在被绕过的风险。
3. **可靠性闭环未最终闭合**：webhook 重试仅完成了“计划落库”，但还没有看到统一 replay / retry 执行通路；统一编译与回归也尚未做，当前不能宣称稳定。

## 7. 分阶段整改 Step 计划

### Phase 0：冻结契约真相源与风险边界

**目标**

- 在进入下一轮实现前，冻结交易闭环的目标契约，明确哪些接口是生产正式接口，哪些接口是历史兼容或仅开发演练接口。
- 先把“不能继续保留的高风险入口”标出来，避免后续边做边偏。

**重点问题**

- `payment-events` / `settle` 仍能推进支付后订单结算。
- 架构文档、OpenAPI、运行时路由、前端 SDK 命名不一致。
- Portal 购买目录、充值目录、订单报价与价格快照仍未对齐 canonical pricing 控制面。

**入口文件**

- `docs/架构/06-gateway-api设计.md`
- `crates/sdkwork-api-app-commerce/src/order.rs`
- `crates/sdkwork-api-app-commerce/src/lib.rs`

### Phase 4：统一交易状态机、webhook 可靠性与财务收口

**目标**

- Portal `/marketing/my-coupons`、`/reward-history`、`/redemptions`、`/codes` 与 rollback 幂等路径当前仍依赖 `list_coupon_reservation_records()`、`list_coupon_redemption_records()`、`list_coupon_code_records()`、`list_coupon_rollback_records()` 全量读取后再按当前主体过滤，正式营销钱包与奖励历史入口在活动规模增长后存在热点慢查询与读取面扩大风险

## 8. 依赖关系

1. 先做 Phase 0，再做后续实现，避免边写边漂移。
2. Phase 1 和 Phase 2 是主干，必须先补接口，再补管理端 UI。

4. Phase 4 需要在 Phase 1 和 Phase 2 的契约基本稳定后统一收口，否则会反复重命名和返工。

Payment method 的 `replay_window_seconds` 当前仍停留在配置与持久化层，未进入 webhook anti-replay 的正式运行时边界：`docs/架构/35-企业级支付交易闭环架构设计-2026-04-07.md` 在 8.3 已明确把 `provider_event_id`、`dedupe_key = provider:event_id` 与 `replay_window_seconds` 一起定义为 webhook 防重放的正式参数，`docs/架构/36-支付管理后台与交易运维实现设计-2026-04-07.md` 也已把 `replay_window_seconds` 纳入 payment method 安全治理字段；`PaymentMethodRecord`、SQLite/Postgres `ai_payment_methods` 持久化链路同样都已经固化并落库该字段。 但 `crates/sdkwork-api-app-commerce/src/webhook.rs` 的 `process_portal_stripe_webhook(...)` 当前只把 `payment_method.webhook_tolerance_seconds` 传给 `stripe::verify_webhook_signature(...)`，随后仅依据 `dedupe_key` 命中 inbox 做 duplicate ignored 短路；`crates/sdkwork-api-app-commerce/src/payment_provider/stripe.rs` 的 `verify_webhook_signature(...)` 也只接收 `tolerance_seconds` 做签名时间戳容忍校验，并没有把 `replay_window_seconds` 接入“是否仍允许受理、是否仅记录重复投递、是否禁止人工 replay、是否进入 dead-letter/retry”这些正式运行时判定边界。 这会让 `replay_window_seconds` 虽然已经出现在 Admin 配置、领域模型和数据库里，但线上 webhook 仍处于“只验签、不校验业务 anti-replay 受理窗口”的半成品状态：重复回调会被简单按 `dedupe_key` 合并，过窗事件是否允许重放也没有统一规则，运维侧无法基于 payment method 配置判断 manual replay 资格。它不是 74“provider webhook accepted-then-async 缺口”的重复，因为 74 关注的是首包入箱与 ACK 边界；也不是 64/7.107“dedupe_key 命中后的 inbox 证据覆盖写语义”的重复，因为 64 关注的是 duplicate 写入如何污染原始收件证据。83 关注的是 `replay_window_seconds` 这个正式安全治理字段本身没有进入 webhook anti-replay 受理窗口与人工 replay 资格边界，必须纳入 Phase 4 的 webhook 可靠性与交易风控收口，以及 Phase 6 的部署验收门槛。

## 11. 当前 iteration 的执行结论

本轮审查已经确认：

- **本轮继续新增确认的支付方式治理契约漂移问题** 包括：文档要求 `POST /admin/commerce/payment-methods`、activate/deactivate 与 bindings CRUD，但当前 `sdkwork-api-interface-admin` 只暴露 `PUT/DELETE /payment-methods/{id}` 和 `GET/PUT /credential-bindings`；commerce kernel 仍是 upsert / delete / bindings replace 语义，Admin SDK/UI 也只能围绕这组兼容路由工作。这说明 payment method 管理面自身尚未收敛为正式、可审计的控制面动作模型。
- **本轮继续新增确认的退款能力硬约束缺口问题** 包括：架构文档已把 payment method 的 `supports_refund / supports_partial_refund` 作为正式能力语义，Admin 管理端也在统计 refund capable methods，但 `paymentOrderOperationsSection` 的退款入口仍只按 `refundable_amount_minor` 开门，`create_admin_commerce_refund(...)` 也不会校验 capability codes。这说明 payment method 的退款能力目前仍停留在展示层，没有真正进入 refund 主链路的 fail-closed 约束。
- **本轮继续新增确认的 refund secret 语义漂移问题** 包括：Admin 凭据绑定 UI 已经把 `refund_secret` 作为正式 `usage_kind` 暴露给运营，并承诺 refund execution 读取最小必要 secret；但 `PaymentMethodSecretBundle` 没有 `refund_secret` 字段，`resolve_payment_method_secret_bundle(...)` 也不解析它，`create_admin_commerce_refund(...)` 最终仍只消费 `api_secret`。这说明 refund-specific credential isolation 目前仍停留在 UI/文档层，没有真正进入退款执行边界。
- **本轮继续新增确认的 webhook path 伪配置问题** 包括：文档、领域模型、数据库和 Admin 配置界面都把 `webhook_path` 作为 payment method 正式治理字段暴露，但 Portal 运行时只注册固定 `POST /portal/commerce/webhooks/stripe/{payment_method_id}`，`stripe_webhook_handler(...)` 也不会读取或校验 `payment_method.webhook_path`。这说明 callback governance 当前仍停留在文档与配置层，没有真正进入运行时回调入口边界。
- **本轮继续新增确认的 payment method `webhook_path` 索引分叉问题** 包括：`docs/架构/35-企业级支付交易闭环架构设计-2026-04-07.md` 已要求支付方式表增加 `(webhook_path)` 索引，`docs/架构/36-支付管理后台与交易运维实现设计-2026-04-07.md` 也把 `webhook_path` 视为正式治理轴线；但 Postgres 迁移已创建 `idx_ai_payment_methods_webhook_path`，SQLite 迁移却只持久化 `webhook_path` 字段而没有同语义索引。这说明即便后续修复了 7.71 的运行时路由边界，支付方式回调路径治理在双存储部署下仍会保留正式索引契约分叉。
- **本轮继续新增确认的 payment method 主表契约分叉问题** 包括：`docs/架构/05-数据库设计.md` 仍把 `ai_payment_methods` 定义为 `provider_code/method_code/status/checkout_flow/webhook_strategy/supports_refund/supports_partial_refund` 模型，并要求 `idx_ai_payment_methods_status_sort`、`idx_ai_payment_methods_provider_code` 与 `(provider_code, method_code, mode)` 唯一索引；但 `PaymentMethodRecord`、SQLite/Postgres 迁移与存储读写链路实际已经固化为 `provider/channel/enabled/capability_codes/callback_strategy/webhook_path/webhook_tolerance_seconds/replay_window_seconds/max_retry_count` 模型，只保留 `idx_ai_payment_methods_provider_enabled_sort` 与 `(webhook_path)` 索引，也没有同义自然键约束。这说明 payment method 主表当前没有与架构文档共享单一 schema、唯一键和治理状态语义，即便后续补齐了 payment method API 动作模型和 webhook path 运行时入口，底层真相源仍然是分叉的。
- **本轮继续新增确认的 payment method 外部 DTO 契约分叉问题** 包括：`docs/架构/06-gateway-api设计.md` 仍把 Payment Method 返回规范写成 `provider_code/method_code/status/checkout_flow/currency_codes/.../supports_webhook/bindings` 模型，并把 credential bindings 视为 payment method 的内联字段；但 `sdkwork-api-interface-admin` 的 `list_payment_methods_handler(...)` / `put_payment_method_handler(...)` 实际直接返回 `PaymentMethodRecord`，`sdkwork-router-admin` 类型也已经固化为 `provider/channel/enabled/capability_codes/callback_strategy/webhook_path/webhook_tolerance_seconds/replay_window_seconds/max_retry_count/config_json`，bindings 另走独立 `PaymentMethodCredentialBindingRecord` 列表接口。这说明 payment method 对外 API/SDK/管理端字段语义当前没有与架构文档共享单一 DTO 真相源，即便底层主表和治理动作后续收口，外部消费契约仍然会保持分叉。
- **本轮继续新增确认的 payment attempt 外部 DTO 契约分叉问题** 包括：`docs/架构/06-gateway-api设计.md` 仍把 Payment Attempt 返回规范写成 `provider_checkout_url/provider_session_id/provider_payment_intent_id/amount_minor/expires_at_ms` 为主的旧字段模型；但 `sdkwork-api-interface-admin` 的 `list_commerce_payment_attempts_handler(...)` 实际直接返回 `CommercePaymentAttemptRecord`，`sdkwork-router-admin` 类型也已经固化为 `provider/channel/idempotency_key/attempt_sequence/captured_amount_minor/refunded_amount_minor/provider_checkout_session_id/provider_reference/checkout_url/request_payload_json/response_payload_json/error_code/error_message/initiated_at_ms/completed_at_ms/updated_at_ms` 模型。这说明 payment attempt 对外 API/SDK/管理端字段语义当前没有与架构文档共享单一 DTO 真相源，即便底层 attempt 主表后续收口，外部消费契约仍然会保持分叉。
- **本轮继续新增确认的 webhook/reconciliation 外部 DTO 真相源缺口问题** 包括：`docs/架构/06-gateway-api设计.md` 在 2.2 虽列出 `GET /admin/commerce/webhook-inbox`、`GET/POST /admin/commerce/reconciliation-runs`、`GET /admin/commerce/reconciliation-runs/{reconciliation_run_id}/items` 等运维接口，但 5. API 返回规范只定义了 Payment Method、Payment Attempt、Refund，完全没有 `CommerceWebhookInboxRecord`、`CommerceWebhookDeliveryAttemptRecord`、`CommerceReconciliationRunRecord`、`CommerceReconciliationItemRecord` 的正式返回模型；同时真实 Admin 路由/OpenAPI 还额外暴露了文档未列出的 `GET /admin/commerce/webhook-inbox/{webhook_inbox_id}/delivery-attempts`，`sdkwork-router-admin` 类型已固定消费这些记录。这说明 webhook/reconciliation 运维读模型当前没有与架构文档共享单一 DTO 真相源，即便后续补齐 replay 或 reconciliation 执行动作，读模型与接入文档仍会继续分叉。
- **本轮继续新增确认的 provider webhook 公共入口路径文档分叉问题** 包括：`docs/架构/06-gateway-api设计.md` 仍把 Stripe provider webhook 公共入口定义为 `/webhooks/commerce/stripe`，但 `docs/架构/35-企业级支付交易闭环架构设计-2026-04-07.md` 与真实 Portal 运行时已经固定为 `/portal/commerce/webhooks/stripe/{payment_method_id}`。这会让部署手册、第三方接入、OpenAPI 与支付方式管理后台文案继续建立在分叉的公开回调边界之上，即便后续补齐了 `webhook_path` 配置落地与 webhook/reconciliation DTO，provider webhook 入口本身仍然没有单一真相源。
- **本轮继续新增确认的交易域结构化日志缺口问题** 包括：`docs/架构/08-安全、风控与性能设计.md` 已正式要求 `order_id`、`payment_attempt_id`、`provider_code`、`provider_event_id`、`refund_id`、`reconciliation_run_id` 六个结构化日志字段；但当前 Portal/Admin 运行时只接入通用 HTTP `observe_http_metrics` / `observe_http_tracing`，`sdkwork-api-app-commerce` 的 `payment_attempt`、`webhook`、`refund`、`reconciliation` 主链里未见这些字段的结构化日志落点。这说明交易运行态目前仍缺少 machine-searchable 的上下文证据，dead-letter / retry / refund mismatch / reconciliation diff 的生产排障无法与指标告警、operator audit 收口为同一条正式证据链。
- **本轮继续新增确认的 provider webhook 首包 ACK 边界缺口问题** 包括：架构文档已要求 provider webhook 先进入 `ai_commerce_webhook_inbox`，再由消费任务 claim 后异步推进状态；但 `stripe_webhook_handler(...)` 仍直接等待 `process_portal_stripe_webhook(...)`，而后者会在同一个 HTTP 请求内同步执行 attempt 持久化、订单/账务副作用与退款完成副作用，并在失败时直接返回错误。这说明“已入箱”与“已 ACK”目前仍绑定在同步主链成功上，provider 首包 delivery 还没有形成正式的 accepted-then-async 边界。
- **本轮继续新增确认的 callback strategy 弱语义问题** 包括：文档已把 `callback_strategy` 定义成正式支付治理主轴，但 Admin 仍把它作为自由文本录入，运行时只做非空校验，并通过 `contains('webhook')` 推导 `supports_webhook`；同一个字段还被直接写进 checkout method 的 `webhook_verification` 展示位，`process_portal_stripe_webhook(...)` 也不会校验该 payment method 是否真的允许 webhook 模式。这说明 callback strategy 当前尚未形成正式、可验证、可执行的运行时策略边界。
- **本轮继续新增确认的 reconciliation secret 语义漂移问题** 包括：架构文档已把 `reconciliation_secret` 定义成正式 usage_kind，`PaymentMethodSecretBundle` 也已经建模该字段；但 Admin `paymentCredentialBindingsDialog.tsx` 的默认 usage kinds 与示例文案仍未把它作为一等 binding 暴露，`load_reconciliation_secret(...)` 又会在缺失时直接回退 `api_secret`。这说明 reconciliation 凭据隔离仍停留在文档和半截运行时模型层，对账入口尚未真正具备 fail-closed 的最小权限边界。
- **本轮继续新增确认的 Stripe refund provider identity 回退错误** 包括：架构和数据库都把 `provider_payment_intent_id`、`provider_checkout_session_id`、`provider_reference` 区分建模，并要求以 `provider_payment_intent_id` 作为 provider 交易真相源；但 `create_checkout_session(...)` 当前把 `provider_reference` 保存为 checkout session id，而 `stripe::create_refund(...)` 又会在缺失 `provider_payment_intent_id` 时回退使用该 `provider_reference` 作为 `payment_intent`。这说明退款执行对 provider transaction identity 还没有建立 typed、fail-closed 的正式边界。
- **本轮继续新增确认的 refund webhook 身份边界缺口** 包括：`create_refund(...)` 已把 `refund_id` 与 `payment_method_id` 写入 Stripe metadata，`parse_webhook_event(...)` 也能同时解析 `local_refund_id`、`payment_method_id` 与 `provider_refund_id`；但 `load_refund_for_webhook(...)` 只要命中 `metadata.refund_id` 就直接返回本地 refund，不校验既有 `provider_refund_id`、路由 payment method 或 metadata payment method 是否一致，随后 `RefundUpdated` 分支还会直接覆写 refund 并推进副作用。这说明退款回调主链尚未把 `refund_id + provider_refund_id + payment_method_id` 收口为单一身份边界。
- **本轮继续新增确认的 refund reconciliation 身份盲区** 包括：Stripe refund 创建时已经写入 `metadata[order_id]`、`metadata[refund_id]`、`metadata[payment_attempt_id]`、`metadata[payment_method_id]`，但 `retrieve_refund(...)` 只返回 amount/status/currency，`reconcile_refund(...)` 也只比较金额与状态，不校验 provider metadata 身份字段。这说明即便退款 webhook 已经把 provider refund 错绑到错误的本地 refund 上，当前 reconciliation 也可能因为金额/状态一致而完全发现不了问题。
