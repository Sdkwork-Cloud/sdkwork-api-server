# 96 - Step 并行执行周计划与排班建议

## 1. 文档目标

本文件用于把 `00-13` 的 step 体系转换成更接近项目管理层可执行的周计划和排班建议，适合多人或多子 agent 并行执行。

- `01`
- `02`
- `03`

目标：

- 冻结门禁、基线、边界与协议

### 2.2 第 2-3 周：波次 B

- `04`
- `05`
- `06`

目标：

- 建立统一路由主链路、Provider 治理、控制平面主流程

### 2.3 第 4-5 周：波次 C

- `07`
- `08`
- `09`
- `10`

目标：

- 完成多模态资源、数据、安全与观测闭环

### 2.4 第 6 周：波次 D

- `11`
- `12`
- `13`

目标：

- 完成交付、性能、最终发布与放行

### 2.5 第 1-5 周：五周平衡版

- 第 1 周：`00-03`
- 第 2 周：`04-05`
- 第 3 周：`06-08`
- 第 4 周：`09-10`
- 第 5 周：`11-13`

目标：

- 比四周极速版更稳
- 比六周稳妥版更快
- 适合作为默认推荐排期

## 4. 每周固定节奏

- 周一：冻结本周 step 输入与执行卡
- 周中：并行推进，处理 `B2/B3` 阻塞
- 周五：使用 `91` 做 step 复盘，使用 `93` 做波次检查

## 5. 快速完整执行建议

### 5.1 六周稳妥版

- 第 1 周：`00-03`
- 第 2-3 周：`04-06`
- 第 4-5 周：`07-10`
- 第 6 周：`11-13`

适用场景：

- 需要边做边校正文档与架构
- 需要保证较低返工率

### 5.1.1 五周平衡版

- 第 1 周：`00-03`
- 第 2 周：`04-05`
- 第 3 周：`06-08`
- 第 4 周：`09-10`
- 第 5 周：`11-13`

适用场景：

- 要求快，但不能接受四周极速版的高返工风险
- 有 `Owner + 5 条实现车道 + 1 条验证车道`
- 可以保证每天一次收口、每周一次波次验收

### 5.2 四周极速版

- 第 1 周：`00-03`
- 第 2 周：`04-06`
- 第 3 周：`07-10`
- 第 4 周：`11-13`

硬前提：

- 至少具备 `Owner + 4 条实现车道 + 1 条验证车道`
- `00-04` 无重大返工
- 高频集成与回写能力已成熟

建议的日历节奏：

- 周一：冻结波次输入、主写入范围和阻塞级别
- 周二到周三：实现车道并行推进
- 周四：验证车道集中跑测试、演练、复盘
- 周五：owner 收口、评分、回写架构并决定是否过波次

### 5.2.1 四周极速版角色拆分

- `Owner`：只负责裁决、集成、回写、波次放行
- `Lane-1`：协议 / 路由
- `Lane-2`：Provider / 控制平面
- `Lane-3`：数据 / 安全 / 观测
- `Lane-4`：交付 / 性能 / 发布
- `Verifier`：测试、演练、评分卡、放行审计

### 5.2.2 四周极速版成功条件

- `00-03` 在第 1 周末前全部收口
- `04` 必须在第 2 周前半段冻结，否则后续并行会全面返工
- `06` 与 `08` 必须在第 3 周中期前收口，否则第 4 周无法形成完整放行材料
- `11-12` 若未通过演练，不允许 `13` 给出正式放行

### 5.2.3 四周极速版降级策略

出现以下任一情况，必须立即从四周极速版降级为六周稳妥版：

- `00-03` 任一步在第 1 周末仍未收口
- `04` 未能在第 2 周前半段冻结主时序和双态边界
- 任一波次出现未解决的 `B3` 阻塞
- 验证车道无法在周四前完成本周核心验证
- `93` 波次验收未通过

降级后的执行规则：

- 暂停新并行扩散
- 先回修当前波次失败项
- 待 `91` 与 `93` 都通过后，再恢复下一波次推进

### 5.2.4 四周极速版并行上限

- 波次 A：最多 `4` 条实现车道 + `1` 条验证车道
- 波次 B：最多 `6` 条实现车道 + `1` 条验证车道
- 波次 C：最多 `5` 条实现车道 + `1` 条验证车道
- 波次 D：最多 `5` 条实现车道 + `1` 条验证车道

超过上限时，优先减少实现车道，不减少验证车道。

### 5.3 不建议的执行方式

- 一开始就把 `05-12` 全部并行摊开
- 没有验证车道的“纯实现冲刺”
- 没有 owner 裁决的多人同时改高风险边界文件
- 在 `93` 未验收通过前，提前开始下一波次实现

### 5.4 最快完整并行执行总建议

若目标是“最快且完整地跑完全部 step”，推荐固定执行法：

1. 默认采用五周平衡版，只有在 `00-04` 已高度稳定时才升级到四周极速版。
2. 每个波次固定保留四类角色：`Owner`、`实现车道`、`验证车道`、`文档回写车道`。
3. 高风险边界只允许一个主写 owner：协议字段、路由主时序、权限模型、发布结论、release notes 基线。
4. 每周只允许一个集成窗口；未进入集成窗口的分支不允许影响下一波次。
5. 每个 step 收尾必须同时完成 `91` 评分、`93` 波次检查、`97` 架构回写；缺一项即视为未完成。
6. `11-13` 必须单独设置 release 真值车道，专门核对 sibling SDK、`/docs/release`、GitHub 不可变依赖和最终发布说明。

## 6. 结论

本文件的价值在于把架构计划转换成节奏化执行，不让 step 体系停留在静态文档层。
