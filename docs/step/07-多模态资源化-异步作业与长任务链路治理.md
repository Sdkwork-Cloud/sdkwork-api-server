# Step 07 - 多模态资源化、异步作业与长任务链路治理

## 1. 目标与范围

本 step 的目标是把文本、图像、音频、视频、上传、向量、批处理、评测、实时会话与编程代理相关 API，统一纳入“资源对象 + 执行方式 + 结果物料化 + 任务治理”的平台模型，而不是继续按协议散落实现。

### 1.1 执行输入

- step 04 的统一路由主时序
- step 05 的 Provider 能力声明标准
- step 06 的控制平面与工作区对象体系
- `docs/架构/07-路由-缓存-流式通信-异步任务设计.md`
- `docs/架构/131-多模态请求生命周期与路由时序设计-2026-04-07.md`
- `docs/架构/140-数据生命周期与成本治理设计-2026-04-07.md`

### 1.2 本步非目标

- 不在本步内完成所有性能极限优化
- 不为每一种资源类型建立完全独立的第二套平台能力
- 不忽略长任务与短链路在可观测性和计费上的差异

### 1.3 最小输出

- 多模态资源对象模型
- sync / stream / async / batch 的统一执行模型
- 长任务状态与结果物料化模型
- 资源与任务治理工作区入口

## 3. 当前现状

当前架构已将平台目标扩展到文本、图像、音频、视频、上传、向量、批处理、评测、实时与编程代理相关 API，但工程层仍需继续统一：

- 资源类对象与任务类对象的关系
- 上传、文件、向量、批处理、评测结果的生命周期
- 流式短链路与长任务链路的差异化治理
- stateful / stateless 在资源持久化与任务证据方面的差异

## 4. 设计

### 4.1 资源对象模型

至少统一以下对象：

- file / upload
- vector store / file batch
- image / audio / video generation result
- eval / batch job
- realtime session / coding task result

### 4.2 执行方式模型

统一区分：

- `sync`
- `stream`
- `async`
- `batch`

并为每种执行方式定义：

- 请求接受点
- 状态推进点
- 结果完成点
- usage / billing 结算点
- 错误与取消语义

### 4.3 长任务治理

长任务必须支持：

- 幂等提交
- 状态查询
- 结果物料化
- 失败重试
- 取消
- 审计与追踪

### 4.4 资源与工作区联动

资源对象与任务对象必须可在控制平面、Admin、Portal 工作区中被查询、诊断和治理。

### 4.5 生命周期与成本治理

必须同步定义：

- 资源结果物料的保留期、归档、清理与删除边界
- 长任务中间态、最终态、失败态的存储与过期策略
- 多模态结果、批处理结果、评测结果的成本归因口径
- `stateful` 与 `stateless` 在资源持久化与成本证据上的差异

## 9. 风险与回滚

### 9.1 风险

- 若资源对象与任务对象分裂，后续查询、审计、结算会高度混乱
- 若流式与长任务共享错误的执行模型，会带来性能与稳定性风险

### 9.2 回滚

- 新资源对象模型优先保留旧查询兼容层
- 长任务治理先以额外状态层并行挂接，再切换主流程

## 10. 完成定义

满足以下条件视为完成：

- 多模态与长任务已统一纳入平台模型
- 资源、状态、结果、结算、审计主链路可解释且可验证
