# 93 - 波次里程碑与阶段总验收矩阵

## 1. 文档目标

本文件用于定义波次级里程碑和阶段总验收标准，避免每个 step 单独通过后，整体仍然不具备进入下一阶段的条件。

## 2. 波次定义

### 2.1 波次 A：基线与骨架

- `00`
- `01`
- `02`
- `03`

通过标准：

- 实施体系已冻结
- 当前事实与差距已冻结
- 模块边界与协议契约已冻结

### 2.2 波次 B：主链路与平台能力

- `04`
- `05`
- `06`

通过标准：

- 统一路由内核已成型
- Provider / 扩展运行时已形成统一标准
- 控制平面、Admin、Portal、商业化主链路已贯通
- 权限能力模型与控制面配置边界已收敛

### 2.3 波次 C：数据、安全、观测

- `07`
- `08`
- `09`
- `10`

通过标准：

- 多模态资源与长任务已统一治理
- 数据、缓存、配置、安全、审计、观测已成体系
- 数据生命周期、成本归因、配置变更安全已形成规则

### 2.4 波次 D：交付与放行

- `11`
- `12`
- `13`

通过标准：

- 安装部署发布回滚标准可执行
- 性能、容量、HA、DR 有证据
- 最终评分卡与发布结论可成立
- GitHub Release 依赖真值、Cell/Region 演练与 release 说明基线可成立

## 3. 阶段总验收矩阵

| 波次 | 必须回答的问题 | 必备证据 | 不通过时的处理 |
| --- | --- | --- | --- |
| A | 我们是否知道当前真实状态，并已冻结结构与协议边界 | 差距矩阵、边界矩阵、兼容矩阵、`8.6` 兑现清单 | 回修 `01-03` |
| B | 主链路和平台骨架是否真的可运行、可治理 | 路由时序、Provider 生命周期、控制平面工作流、`8.6` 兑现清单 | 回修 `04-06` |
| C | 数据、安全、观测是否已进入生产基线 | 真值分层、权限模型、审计与 SLO 证据、`8.6` 兑现清单 | 回修 `07-10` |
| D | 是否具备发布、回滚、性能与最终放行条件 | 演练报告、评分卡、最终放行结论、release 真值清单 | 回修 `11-13` |

## 4. 波次准入与放行门槛

### 4.1 通用准入门槛

任一波次进入前，至少满足：

- 前一波次已通过总验收
- 相关 step 使用 `91` 评分不低于 `90/100`
- 前一波次各 step 的 `8.6` 已全部兑现，或已被正式降级记录且不阻断后续
- 前一波次各 step 的 `97` 架构回写已完成
- 不存在未解决的 `B3` 阻塞
- `stateful standalone` 与 `stateless runtime` 的影响边界已说明

### 4.2 波次 D 放行门槛

进入最终放行前，至少满足：

- `docs/架构/138-*` 评分卡有当前事实分
- 安全维度不出现空白项
- 发布、回滚、性能、灾备有演练证据
- 兼容客户端与运维工具已做真实接入验证
- `/docs/release` 与 Git 历史可以回溯到最近一次成功发布基线
- sibling SDK 本地/远端一致性已通过

### 4.3 与 `138` 评分卡的对齐要求

建议每个波次在通过前，至少给出与 `docs/架构/138-*` 对应维度的阶段性评分：

- 波次 A：重点覆盖“产品与协议完备度”“架构与模块边界”两个维度的阶段分
- 波次 B：重点覆盖“路由与多模态能力”“产品与协议完备度”“架构与模块边界”三个维度的阶段分
- 波次 C：重点覆盖“安全与合规”“可观测性与运维”“测试与质量保障”三个维度的阶段分
- 波次 D：必须覆盖八个一级维度的完整当前事实分

阶段性评分的用途：

- 不是提前宣称最终成熟度
- 而是避免某一波次结束时只完成实现、不知道离正式放行还有多远

## 5. 波次否决条件

任一波次出现以下情况，不允许进入下一波次：

- 核心主责 step 未完成
- 关键架构边界仍存在争议
- 无法提供波次级证据
- 任一步 `8.6` 仍存在未裁决缺口
- 任一步未完成 `97` 回写或 `98` 交付包
- 存在未被接受的 `B3` 阻塞
- 没有分别给出 `stateful` 与 `stateless` 结论
- 无法给出与 `138` 对齐的阶段性评分和差距说明

## 6. 结论

单步完成不代表阶段完成。只有波次级里程碑通过，整体实施才算在正确轨道上。
