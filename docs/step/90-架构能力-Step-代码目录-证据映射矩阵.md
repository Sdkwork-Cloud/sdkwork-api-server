# 90 - 架构能力、Step、代码目录、证据映射矩阵

## 1. 文档目标

本文件用于回答四个问题：

- 某项架构能力到底在哪个 step 落地
- 会主写哪些目录与模块
- 应该产出哪些验证证据
- 没做到什么程度就不能宣称该能力闭环

## 2. 使用原则

- 先按能力找 step，再按 step 找代码目录和证据
- 一个能力可以跨多个 step，但必须有主责 step
- 没有证据的能力，不能算已完成

## 4. 证据等级

建议证据按以下等级管理：

- `E1`：文档证据
- `E2`：测试与脚本证据
- `E3`：运行与演练证据
- `E4`：评分卡与放行证据

每个核心能力至少同时具备 `E1 + E2`；涉及交付、性能、安全的能力必须具备 `E3`；最终发布能力必须具备 `E4`。

## 4.1 最低放行门槛

每个能力在主责 step 完成后，至少达到以下最低门槛才允许放行到下一波次：

- 协议、模块、对象、权限、交付等“边界型能力”至少具备 `E1 + E2`
- 路由、Provider、控制平面、多模态、数据等“运行型能力”至少具备 `E1 + E2 + 局部 E3`
- 安全、观测、交付、性能、灾备等“生产型能力”必须具备 `E1 + E2 + E3`
- 最终发布与行业领先宣称必须具备 `E1 + E2 + E3 + E4`

## 4.2 推荐 owner 对照

建议每个能力域都指定单一 owner：

- `00-03`：Architect / Protocol Owner
- `04-05`：Routing / Provider Owner
- `06`：Control Plane Owner
- `07-08`：Runtime Data Owner
- `09-10`：Security / Observability Owner
- `11-13`：Delivery / Release Owner

## 5. 使用方法

执行任何 step 前，先用本矩阵确认：

- 本轮到底要实现哪项能力
- 主写入范围是否明确
- 必备证据是否已规划
- 是否缺少前序 step 的输入
- 最低放行门槛是否已被写入执行卡
- 本步结束后是否真的能兑现对应架构能力

## 6. 结论

本文件的作用不是列目录，而是把“架构能力 -> 实施 step -> 代码改动 -> 证据闭环”串成同一条链。没有这条链，架构文档就无法真正落地。
