## 2. Current / Target / Deferred

- Current：
  - marketing 存储仍保留 canonical 与 legacy 双轨。
  - bootstrap 已有商业账户与账本数据基础，但 runtime 写路径仍存在 split-brain。
  - `gateway_commercial`、`app-usage`、request metering、account kernel 尚未完全闭成同一事实链。
  - legacy coupon 域/应用兼容层曾残留运行时依赖与兼容主路径。
- Target：
  - canonical marketing、commercial account、request metering 成为唯一主写主读路径。
  - dual-write、dual-read、diff、cutover、dependency cleanup 有明确阶段与回退点。
  - 运行时主路径不再把 legacy coupon 当真值，也不再把 legacy usage 当唯一计量真值。
- Deferred：
  - 更深层的离线回放系统
  - 跨环境自动回灌平台

## 3. 设计

- `S07` 只做迁移、灰度、切换、删除，不再发明新业务对象。
- 采用固定四阶段：
  1. schema / mapper 就绪
  2. shadow dual-write + diff
  3. read switch + cutover
  4. legacy dependency cleanup
- 所有 cutover 必须遵守“先比对、后切读、再删旧”的顺序；禁止边迁移边删回退通道。
- request metering、account ledger、coupon redemption 必须能串出同一交易解释链。

## 9. 风险与回退

- 风险：
  - dual-write 数据不一致会污染审计结论。
  - 读切换后可能暴露历史脏数据与映射缺口。
  - metering/account/coupon 三条链一旦分叉，账务解释会失真。
- 灰度：
  - 先 shadow write、再 diff、再 sample replay、最后切读；观察窗内保留回退开关。
- 回退触发条件：
  - diff 无法收敛。
  - 切读后出现订单、核销、账本结果不一致。
- 回退动作：
  - 回退 read-switch 到旧路径。
  - 保留新 schema 与 shadow write，继续补映射与回放，不立即删旧依赖。
