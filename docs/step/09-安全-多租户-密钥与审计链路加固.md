# Step 09 - 安全、多租户、密钥与审计链路加固

## 1. 目标与范围

本 step 的目标是把身份、权限、多租户隔离、API Key 生命周期、密钥后端、审计证据与供应链安全统一提升到生产级标准，确保平台在 SaaS、私有化和本地运行态下都具备清晰的安全边界。

### 1.1 执行输入

- step 06 的控制平面对象与权限边界
- step 08 的数据、缓存与配置治理结果
- `docs/架构/08-安全-多租户-SaaS-私有化-部署设计.md`
- `docs/架构/136-安全与合规评估标准-2026-04-07.md`
- `docs/架构/139-权限与能力模型设计-2026-04-07.md`
- `docs/架构/141-控制面配置治理与变更安全设计-2026-04-07.md`

### 1.2 本步非目标

- 不把“已支持密钥后端”直接等同于“已达到企业级安全”
- 不只做接口鉴权而忽略内部服务、扩展运行时与发布链路
- 不把审计当作可选能力

### 1.3 最小输出

- 身份与权限边界模型
- 多租户隔离规则
- 密钥与 API Key 生命周期治理
- 审计链路与供应链安全加固标准

## 3. 当前现状

当前仓库已经具备：

- 多种 secret backend
- identity / tenant / credential 相关领域模块
- Admin / Portal / Gateway 三个关键接口面

仍需继续加固的问题包括：

- 平台管理员、租户管理员、项目操作者、最终 API 调用方之间的权限边界
- 多租户数据、缓存、配置、使用量和账单隔离
- API Key、Provider credential、extension credential 的统一生命周期
- 审计证据、依赖供应链与发布链路安全

## 4. 设计

### 4.1 身份与权限模型

必须至少定义：

- 平台级角色
- 租户级角色
- 项目级角色
- 能力级授权与 feature capability
- 机器身份 / service identity
- 外部 API client identity

### 4.2 多租户隔离模型

必须覆盖：

- 数据隔离
- 配置隔离
- 缓存命名与隔离
- 使用量与账单隔离
- 运维查询与审计隔离

### 4.3 密钥与凭据生命周期

统一治理：

- 创建
- 加密存储
- 展示与脱敏
- 轮换
- 吊销
- 审计

### 4.4 审计与供应链安全

必须覆盖：

- 管理操作审计
- 路由与计费证据审计
- 配置变更审计
- Provider / extension 生命周期审计
- 依赖、构建、发布资产的供应链安全检查

### 4.5 配置变更安全与紧急授权

必须定义：

- 配置审批与双人复核边界
- break-glass 紧急授权与失效机制
- 管控面变更后的审计、回滚与追责链路

## 9. 风险与回滚

### 9.1 风险

- 若安全与租户边界后置到发布阶段，返工成本最高
- 若密钥治理与审计不完整，平台难以进入企业级生产场景

### 9.2 回滚

- 对权限和租户规则的高风险调整，先以灰度或双判定方式上线
- 密钥后端切换必须保留旧后端回退路径

## 10. 完成定义

满足以下条件视为完成：

- 安全、多租户、密钥与审计已形成统一生产基线
- 能够支撑正式部署与客户验收

## 11. 下一步准入条件

进入 step 10 前必须确认：

- 所有关键安全事件、权限事件和凭据事件都可以被观测和审计
- 不再存在无法追踪的高价值操作链路
