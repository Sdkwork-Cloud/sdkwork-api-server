# 97 - Step 完成后的架构回写与能力兑现清单

## 1. 文档目标

本文件用于约束每个 step 完成后必须做哪些架构回写，避免架构文档与代码现实再次脱节。

## 2. 每个 Step 必须回写的内容

- 当前事实是否发生变化
- 目标状态是否已被部分兑现
- 新增限制、风险或例外
- 对 `stateful` 与 `stateless` 的影响
- 对评分卡的影响
- 对应 step `8.6` 中列出的架构能力是否已全部兑现
- 未兑现部分为什么仍未关闭

若 `8.6` 任一能力未兑现，则该 step 只能保留“进行中”状态，不得回写为“已完成”。

## 3. 推荐回写路径

### 3.1 总纲类文档

- `docs/架构/README.md`
- `docs/架构/130-142`

### 3.2 基础说明类文档

- `docs/架构/01-09`

### 3.3 专项或进度类文档

- 对应拆分进度、专项治理或运行记录文档
- 若触达发布与放行：`docs/release/` 与相关 release 基线说明

### 3.4 高风险专题补充回写

若本 step 触达以下主题，必须同步回写对应专题文档：

- 权限、角色、能力开关：`docs/架构/139-*`
- 生命周期、归档、成本治理：`docs/架构/140-*`
- 配置审批、灰度、回滚、发布安全：`docs/架构/141-*`
- Cell / Region、HA、DR、回切：`docs/架构/142-*`

## 4. 回写检查清单

- 是否把目标状态误写成当前事实
- 是否把局部能力误写成全局能力
- 是否把 `stateful` 能力误写到 `stateless`
- 是否同步更新评分卡与验收标准
- 是否同步更新 `139-142` 涉及的权限、生命周期、配置变更、DR 结论
- 是否同步更新了“本步完成后必须兑现的架构能力”对应的结果
- 若触达 `11-13`，是否同步回写 `docs/release/`、release notes 基线与发布限制说明

## 4.1 推荐回写顺序

建议每个 step 收尾时固定按以下顺序回写：

1. 先写 `docs/review/step-XX-架构兑现-*.md`
2. 再更新对应专项架构文档
3. 再更新 `docs/架构/README.md` 或评分卡类文档
4. 最后在 `docs/review/step-XX-架构回写决议-*.md` 中冻结结论

## 5. 推荐回写产物

- `docs/review/step-XX-架构兑现-YYYY-MM-DD.md`
- `docs/review/step-XX-架构回写决议-YYYY-MM-DD.md`

## 6. 结论

架构不是一次性设计物，而是每个 step 都要回写兑现的产品说明书。本文件就是兑现清单。
