# 98 - Step 并行车道交付包与集成交接标准

## 1. 文档目标

本文件用于定义并行车道在交回总 owner 集成前，必须提交什么交付包，避免并行开发最后在集成阶段失控。

## 2. 每条车道必须提交的最小交付包

- 变更说明
- 主写入范围
- 已兑现的架构能力
- 对应 step `8.6` 已兑现项 / 未兑现项
- 测试结果
- 剩余风险
- 回滚说明
- 需要总 owner 裁决的问题
- 若涉及发布：依赖真值、release notes 基线与是否触达 GitHub 不可变依赖

## 3. 推荐交付包模板

```md
# 车道交付包

## 1. 车道名称

## 2. 本轮目标

## 3. 主写入范围

## 4. 已完成项与已兑现架构能力

## 4.1 对应 step 8.6 已兑现项 / 未兑现项

## 5. 测试结果与证据链接

## 6. 残余风险

## 7. 回滚方案

## 8. 架构回写状态

## 9. 待总 owner 裁决事项
```

## 4. 集成交接标准

交回总 owner 前必须确认：

- 不与其他车道主写入范围冲突
- 测试结果可复现
- 已明确说明兑现了哪一条 step `8.6` 架构能力
- 未兑现的 `8.6` 项已明确标注并升级，不允许静默遗留
- 关键限制已写明
- 没有把临时方案伪装成最终方案
- 已同步说明 `97` 架构回写是否完成

## 5. 验证车道特别规则

验证车道可以独立提交：

- 测试资产
- 演练报告
- 复盘文档
- 放行建议

但不应在没有协调的情况下大量改写业务逻辑。

## 6. 结论

并行执行的成败，不取决于并行本身，而取决于是否能把每条车道的结果稳定交接并收口。本文件就是并行交付的最小集成协议。
