# 95 - 架构能力闭环验收标准

## 1. 文档目标

本文件用于定义“什么叫真正闭环”。不是写了文档、也不是改了代码，而是能力从设计到实现到验证到交付都成立。

## 2. 通用闭环标准

任一能力要算闭环，至少满足：

- 有明确架构定义
- 有明确主责 step
- 有明确代码落点
- 有明确测试与验证
- 有明确 review 产物
- 有明确发布或运行结论

并且必须额外满足：

- 明确区分当前事实与目标状态
- 明确区分 `stateful standalone` 与 `stateless runtime`
- 明确给出未达标项与限制条件

## 3. 分能力闭环判定

### 3.1 协议能力闭环

- 兼容矩阵准确
- 契约测试通过
- 文档与实际行为一致

### 3.2 路由能力闭环

- 时序统一
- 双态边界清晰
- decision log 与 usage / billing 证据一致

### 3.3 Provider 能力闭环

- onboarding、灰度、回滚可执行
- 健康检查和密钥治理可验证

### 3.4 控制平面闭环

- 对象真值清晰
- Admin / Portal 主流程可用
- 商业化闭环已贯通
- 能力矩阵、权限矩阵与配置变更边界一致

### 3.5 数据与安全闭环

- 真值分层清晰
- 多租户隔离可证
- 密钥与审计链路完整

### 3.5.1 数据生命周期与成本治理闭环

- 保留期、归档、清理、删除规则可执行
- 成本归因、查询口径与账务证据一致
- 配置 rollout 与数据生命周期不互相越权

### 3.5.2 可观测性与运营闭环

- logs / metrics / traces / request-id 可贯通
- 技术证据、业务证据、变更证据可统一查询
- runbook、告警、诊断入口可直接支撑生产处置

### 3.6 交付闭环

- 可安装、可启动、可升级、可回滚
- 有演练报告与运行证据

### 3.6.1 发布依赖与配置变更安全闭环

- 工作区 sibling 相对路径依赖不会进入 Release 产物
- GitHub 不可变依赖、sibling SDK 真值一致性与 release 说明基线可验证
- 配置审批、回滚、break-glass 与发布链路一致

### 3.6.2 多Cell多Region与灾备闭环

- Cell / Region 级切换、回切、RTO、RPO 有阶段性演练证据
- 不同运行态的 DR 结论分别明确

### 3.7 发布闭环

- 评分卡真实
- 已知限制清晰
- 放行结论明确

## 4. 分层成熟度判定

建议结合 `docs/架构/138-*` 采用以下判断：

- `L1-L2`：可运行、可开发，但仍不具备完整生产证明
- `L3`：具备生产基线，发布、回滚、安全、观测有证据
- `L4`：企业级能力完整，支持稳定运营与私有化交付
- `L5`：行业领先，双运行态、协议面、路由、商业化、交付全部具备高质量证据

任何能力若只达到 `L1-L2`，不得在外部材料中宣称“企业级”或“行业领先”

## 5. 否决条件

出现以下情况时，不能算闭环：

- 只有文档，没有实现
- 只有实现，没有测试
- 只有测试，没有运行证据
- 只有“看起来能跑”，没有回滚和运维能力
- 用目标状态充当当前事实
- 没有双运行态分离结论
- 对应 step 的 `8.6` 架构能力仍有未兑现项
- Release 产物仍携带本地相对路径 SDK 依赖
- 无法证明最近成功发布到当前版本的 changelog 真值

## 6. 结论

本文件是所有“已完成”说法的最终裁判标准。达不到闭环标准，就只能算“进行中”。
