本 step 的目标是围绕平台对象、资源对象、任务对象、商业化对象和运行态证据，收敛 `durable truth / cache truth / query truth`，并建立开发、测试、生产、多环境与私有化场景下统一的数据、缓存与配置 rollout 治理模型。

- 数据真值分层规则

- 缓存分层与失效规则

## 3. 当前现状

当前仓库已经具备：

- 存储后端：`PostgreSQL`、`SQLite`、`libsql`、`MySQL`
- 缓存后端：`memory`、`Redis`
- 配置模块：`sdkwork-api-config`
- 数据与存储相关模块：`storage-*`、`cache-*`

但仍需明确：

- 哪些对象是 durable truth
- 哪些查询属于 query truth
- 哪些缓存只作为性能优化
- 配置变更如何滚动发布与回滚

## 4. 设计

### 4.1 真值分层

统一定义：

- `Durable Truth`：交易、配置、对象状态、审计、账务、任务结果等持久真值
- `Cache Truth`：热路径缓存、健康快照、可快速失效的加速数据
- `Query Truth`：面向查询、列表、报表、工作区显示的派生视图

### 4.2 存储后端策略

建议冻结：

- 开发 / 本地：`SQLite + memory`
- 生产 / 私有化主基线：`PostgreSQL + Redis`
- `libsql / MySQL` 作为兼容或扩展选项，必须有边界与验收标准

### 4.3 缓存策略

缓存必须明确：

- 哪些对象可缓存
- 缓存 TTL 与失效规则
- 命中与回源策略
- 与路由、健康、限流、目录查询的关系

### 4.4 配置 rollout 策略

配置变更必须支持：

- 分环境差异
- 灰度发布
- 回滚
- 审计
- 秘钥与普通配置分离
- 变更审批与 blast radius 控制

### 4.5 数据生命周期与成本治理

必须同步定义：

- 热、温、冷、归档数据分层
- 保留期、清理、脱敏与删除规则
- usage、billing、资源对象与审计对象的成本归因口径
- 数据导出、恢复与合规留存边界

## 9. 风险与回滚

### 9.1 风险

- 若缓存越权成为真值，会直接损害一致性
- 若配置 rollout 没有回滚策略，生产风险极高

### 9.2 回滚

- 所有 schema、缓存、配置改造都必须支持双写、灰度或保留旧配置入口

## 10. 完成定义

满足以下条件视为完成：

- 主生产基线可执行、可验证、可回滚
