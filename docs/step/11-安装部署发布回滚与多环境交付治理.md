# Step 11 - 安装部署发布回滚与多环境交付治理

## 1. 目标与范围

本 step 的目标是把 `build / install / start / stop / upgrade / rollback` 的全链路交付资产标准化，覆盖本地开发、单机 stateful、轻量 stateless、私有化交付与多环境推广场景。

### 1.1 执行输入

- step 08 的数据与配置治理
- step 09 的安全基线
- step 10 的观测与诊断能力
- `docs/架构/09-测试-安装-部署-发布与评估标准.md`
- `docs/架构/137-安装部署发布回滚设计-2026-04-07.md`
- `docs/架构/141-控制面配置治理与变更安全设计-2026-04-07.md`

### 1.2 本步非目标

- 不只验证本地可启动而忽略生产环境升级与回滚
- 不让脚本散落成为新的维护黑洞
- 不在没有 manifest 和环境说明的情况下对外发布

### 1.3 最小输出

- 多环境交付模型
- 安装、部署、启停、升级、回滚标准
- 发布清单与版本资产
- 私有化交付手册

- 私有化交付与平台正式发布的统一标准
- sibling SDK 本地相对路径联调与 Release 环境 GitHub 不可变依赖切换规则

## 4. 设计

### 4.1 多环境交付模型

统一区分：

- 本地开发工作区
- stateful standalone
- stateless runtime
- 一体化产品宿主
- 私有化部署

### 4.2 交付资产模型

每个发布版本必须有：

- 构建产物
- 配置模板
- 环境变量说明
- 依赖说明
- service descriptor 或宿主入口
- 发布 manifest
- GitHub Release 说明与 changelog 基线
- 回滚说明

### 4.3 升级与回滚模型

必须定义：

- 版本号与兼容范围
- schema / config / binary 升级顺序
- 失败回滚路径
- 数据与配置回退边界

### 4.4 私有化交付要求

私有化交付必须包含：

- 拓扑建议
- 最低资源配置
- 密钥与配置注入方式
- 升级、备份、恢复与支持边界

### 4.5 Release 依赖与版本真值治理

必须冻结：

- 工作区允许 `sibling relative path SDK` 用于本地联调，不得破坏当前相对路径设计
- Release 环境、安装产物与 GitHub Release 必须切换到 GitHub 不可变依赖或提交引用
- `sdkwork-core`、`sdkwork-ui`、`sdkwork-appbase`、`sdkwork-im-sdk` 的本地 HEAD、计划发布引用、远端提交或 Tag 必须一致
- `/docs/release` 不完整时，必须从最近一次成功发布基线回溯 Git 历史补全 changelog；未成功发布版本的变更必须并入当前成功发布说明

建议执行：

- 构建产物 smoke
- 安装与卸载 smoke
- start / stop / restart 测试
- 升级与回滚演练
- sibling SDK 本地/远端一致性校验
- GitHub Release 资产与 changelog dry-run
- 多环境配置矩阵测试

## 7. 结果验证

本 step 完成时必须满足：

- 任一发布版本都可被安装、启动、诊断、升级和回滚
- 不同环境的交付方式有清晰边界与操作手册
- 交付资产可被运维团队直接使用
- 发布资产不再引用本地 sibling 相对路径 SDK 依赖
- release notes 可回溯到最近一次成功发布与未成功发布变更汇总

## 9. 风险与回滚

### 9.1 风险

- 若没有统一交付模型，平台无法稳定进入客户环境
- 若升级回滚演练缺失，正式发布风险不可接受
- 若 sibling SDK 本地/远端不一致或 changelog 基线不清，发布真值会失真

### 9.2 回滚

- 所有交付改造都必须先在测试环境演练
- 新脚本切换前保留旧入口和旧文档

## 10. 完成定义

满足以下条件视为完成：

- 发布、安装、启停、升级、回滚的全链路交付标准已建立
- 多环境交付能力可用、可验证、可操作

## 11. 下一步准入条件

- 压测与灾备有稳定的部署与运维基础
- 性能与高可用演练不再依赖临时手工环境
