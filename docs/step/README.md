# SDKWork API Router Step 实施计划索引

## 1. 文档定位

本目录用于把 `docs/架构/` 中已经冻结的产品、架构、性能、安全、交付标准，转换成可执行、可检查、可并行编排、可验收的实施 step 体系。

它服务的对象不是“阅读架构的人”，而是“真正要在当前仓库里落地重构、升级、发布的人或 agent”。因此本目录必须回答以下问题：

- 当前 API Router 主线到底先做什么、后做什么。
- 哪些事项必须串行，哪些事项可以多子 agent 或多人并行。
- 每个 step 具体会落到哪些目录、模块、服务、脚本与文档。
- 每个 step 如何测试、如何验收、如何回滚、如何回写架构。

本目录的最高裁决依据为：

- `docs/架构/130-142`
- `docs/架构/01-09`

历史专项文档、阶段性审计文档、单点整改文档如与本目录冲突，以本目录与 `docs/架构/` 主基线为准。

## 2. 当前主线产品背景

当前 `sdkwork-api-router` 的主线定位已经明确为：

`面向企业与开发者的多模型、多协议、多模态 API 聚合、路由、治理与交付平台`

对标对象包括但不限于：

- `OpenRouter`
- `new-api`
- 具备企业控制平面、自助门户、统一路由与多 Provider 治理能力的行业头部聚合平台

当前仓库中已经可以直接验证的关键现实包括：

- 运行面服务：`gateway-service`、`admin-api-service`、`portal-api-service`、`router-web-service`、`router-product-service`
- 前端应用：`apps/sdkwork-router-admin`、`apps/sdkwork-router-portal`
- 双运行形态：`stateful standalone` 与 `stateless runtime`
- Provider / 扩展运行时类型：`builtin`、`connector`、`native-dynamic`
- 内置 Provider 基线：`openai`、`openrouter`、`ollama`
- 存储与缓存基线：`PostgreSQL / SQLite / libsql / MySQL`、`memory / Redis`
- 密钥后端基线：`database_encrypted`、`local_encrypted_file`、`os_keyring`

因此本目录不会把 `sdkwork-api-router` 当成“单一 OpenAI 兼容转发器”，而是按完整平台实施。

## 8. 执行规则

后续任何 step 必须同时满足以下规则：

- 必须区分 `当前事实` 与 `目标状态`，不得把目标写成已落地现实。
- 必须区分 `stateful standalone` 与 `stateless runtime`，不得把前者能力外推到后者。
- 必须区分 `step 级依赖` 与 `step 内并行车道`，不得跳过主脊柱顺序。
- Release 相关 step 必须区分 `工作区 sibling 相对路径依赖` 与 `Release 环境 GitHub 不可变依赖`；`sdkwork-core`、`sdkwork-ui`、`sdkwork-appbase`、`sdkwork-im-sdk` 任一仓库本地与远端不一致时视为 `B3` 阻塞。
- 每个 step 必须包含：设计、实施落地规划、测试计划、结果验证、检查点、风险与回滚、完成定义。
- 每个 step 完成后必须回写 `docs/架构/` 与 `docs/review/`，不能只改代码不留证据。

## 9. 使用方法

建议每个执行周期按以下顺序推进：

1. 先读本 `README`，确认当前位于哪个波次、哪个 step。
2. 阅读对应 step 文档，确认目标、非目标、主写入范围、检查点、推荐并行车道。
3. 对照 [`90-架构能力-Step-代码目录-证据映射矩阵.md`](./90-架构能力-Step-代码目录-证据映射矩阵.md) 校验能力映射。
4. 对照 [`92-Step输入输出与阻塞升级规则.md`](./92-Step输入输出与阻塞升级规则.md) 生成执行卡。
5. 实施后使用 [`91-Step质量审计清单与复盘模板.md`](./91-Step质量审计清单与复盘模板.md) 打分。
6. 波次完成后使用 [`93-波次里程碑与阶段总验收矩阵.md`](./93-波次里程碑与阶段总验收矩阵.md) 做总验收。
7. 最终按 [`97-Step完成后的架构回写与能力兑现清单.md`](./97-Step完成后的架构回写与能力兑现清单.md) 更新架构口径。

### 9.1 最快完整执行建议

如果目标是“尽快但不失控地完整执行完全部 step”，推荐采用以下固定打法：

1. 默认采用 `96` 中的“五周平衡版”，只有在 `00-04` 已高度稳定时才升级到四周极速版。
2. 先把 `00-04` 视为主脊柱连续完成，不允许跨 step 跳跃推进。
3. 从 `05` 开始，统一采用 `单一总 owner + 多实现车道 + 独立验证车道 + 文档回写车道` 的并行模型。
4. 每个波次只允许一个集成窗口，避免并行车道持续互相覆盖。
5. 每个 step 收尾必须同步完成 `91` 评分、`93` 波次检查、`97` 回写和下一 step 准入判断，否则不得视为完成。
6. `11-13` 必须单独保留 `release 真值车道`，核对 sibling SDK、`/docs/release`、GitHub 不可变依赖与最终发布说明。
7. `13` 之前不得对外宣称“已达到行业领先”或“已可生产”，除非 `93` 和 `95` 已同时通过。

### 9.2 推荐的多子 agent 并行方式

建议每个波次最少分成四类角色：

- `Owner`：冻结边界、处理阻塞、做最终集成
- `Worker-Lane`：按主写入范围并行改造
- `Verifier`：只负责测试、演练、评分与证据审计
- `Doc-Lane`：负责 `docs/review/` 与 `docs/架构/` 回写

并行时的硬约束：

- 同一高风险文件只能有一个主写车道
- 契约字段、权限模型、评分卡结论必须由单一 owner 最终裁决
- 任何车道发现 `B3` 阻塞，必须立即停止扩散并回到 `92` 升级流程

## 10. 完成标准

只有满足以下条件，才能宣称 `docs/step/` 体系完整可执行：

- `00-13` 的主线顺序、串行依赖与并行车道已经全部明确。
- 每个 step 都能映射到当前仓库真实目录，而不是抽象概念。
- 每个 step 都包含设计、实施、测试、验证、检查点与回滚方案。
- 每个 step 都已把 `8.6 架构能力兑现`、`91 评分`、`97 回写`、`98 交付包` 纳入完成定义。
- `90-98` 已形成能力映射、质量审计、阻塞升级、并行编排、阶段验收、架构回写的完整治理闭环。
- 历史专项文档与主线实施体系的边界已经清楚。
