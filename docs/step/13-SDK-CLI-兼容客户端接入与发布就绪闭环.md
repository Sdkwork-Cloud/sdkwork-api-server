## 3. 当前现状

当前项目已经具备较丰富的对外兼容能力和运行骨架，但最终发布就绪仍需统一收口：

- 兼容客户端接入是否经过真实验证
- CLI / 运维工具是否满足生产支持
- 文档、样例、评分卡与交付结论是否一致
- `stateful standalone` 与 `stateless runtime` 的发布结论是否分别清楚

## 4. 设计

### 4.1 对外接入面

最终对外必须覆盖：

- OpenAI 兼容客户端接入
- Anthropic / Gemini 兼容客户端接入
- curl / CLI / 自动化脚本接入
- Admin / Portal 用户接入说明

### 4.2 工具面

至少明确：

- `bin/router-ops.mjs` 的运维定位
- 常见诊断与巡检命令
- 发布后支持团队的工具入口

### 4.3 最终评分与放行

必须分别给出：

- `stateful standalone` 当前事实分
- `stateless runtime` 当前事实分
- 未达标项与发布限制
- 是否允许预发 / 正式生产 / 行业领先对外宣称

### 4.4 文档与样例要求

必须统一：

- 快速开始
- 协议兼容说明
- 多模态接入说明
- 认证、限流、配额、错误码说明
- 升级、回滚、运维说明

### 4.5 最终发布真值与 release 说明

必须最终确认：

- `/docs/release` 与 Git 历史能回溯到最近一次成功发布
- 未成功发布版本的 changelog 已并入当前 release 说明
- `sdkwork-core`、`sdkwork-ui`、`sdkwork-appbase` 与 `sdkwork-craw-chat-sdk`（发布 Git 根为 `apps/craw-chat`）的本地、远端、计划发布引用一致
- Release 产物未携带本地 sibling 相对路径依赖

### 4.6 架构能力兑现总表

最终放行必须逐项给出 `130-142` 的结论：

- 已兑现
- 部分兑现
- 未兑现但可发布
- 未兑现且阻断发布

## 9. 风险与回滚

### 9.1 风险

- 若没有最终兼容客户端验证，发布可信度不足
- 若没有明确放行结论，平台会陷入“看似完成、实则未完成”的灰区
- 若 `/docs/release`、依赖真值和架构能力结论不一致，最终发布将不可接受

### 9.2 回滚

- 对外发布前发现关键问题，应回退到前序 step 修复并重新验收
- 不允许通过修改文档措辞掩盖未达标事实

## 10. 完成定义

满足以下条件视为完成：

- 对外接入、运维、发布、评分与放行已完成统一闭环
- 平台可以给出可信的当前成熟度结论与后续改进方向

## 11. 下一步准入条件

本 step 为主线最终收口 step，完成后进入持续迭代与版本治理阶段。
