# Step 01 - 现状基线冻结与差距审计

## 1. 目标与范围

本 step 的目标是把当前 `sdkwork-api-router` 的真实现状、差距、风险与优先级冻结为单一基线，避免后续改造建立在错误假设上。

### 1.1 执行输入

- `docs/架构/README.md`
- `docs/架构/01-09`
- `docs/架构/130-142`
- 当前仓库的 `services/`、`apps/`、`crates/`、`bin/`、`scripts/dev/`
- 现有 `docs/step/` 历史专项计划

### 1.2 本步非目标

- 不直接实施 crate 拆分、协议改造或服务重构
- 不以“目标状态”覆盖“当前事实”
- 不直接消化全部审计问题

### 1.3 最小输出

- 当前事实清单
- 差距矩阵
- 高风险清单
- 优先级排序
- 事实与目标边界说明

## 3. 当前现状

从当前仓库可验证事实看，项目已经具备完整平台骨架，但仍然需要系统性审计才能指导改造：

- 已有五个核心运行面服务和两个前端应用
- 已有大量 `app / domain / storage / provider / extension / interface` crate
- 已同时存在 `stateful standalone` 与 `stateless runtime`
- 已具备多协议兼容与多模态覆盖能力
- 仍存在部分高风险模块、历史文档口径冲突、落地成熟度不均衡等问题

## 4. 设计

### 4.1 审计维度

基线审计必须按以下维度逐项执行：

- 产品与协议面
- 运行形态与拓扑
- workspace / crate / service / app 边界
- 路由内核与 Provider 生态
- 控制平面与商业化
- 数据、存储、缓存与配置
- 安全、审计、密钥与多租户
- 可观测性、SLO、部署、发布、回滚
- 权限与能力模型
- 数据生命周期与成本治理
- 控制面配置变更安全
- Cell / Region / HA / DR 演进成熟度

### 4.2 审计分层

所有审计结论必须同时区分：

- `当前事实`
- `架构目标`
- `已知差距`
- `本轮优先解决`
- `后续阶段处理`

### 4.3 运行形态分离审计

对 `stateful standalone` 与 `stateless runtime` 必须分别给出：

- 能力清单
- 差距清单
- 风险清单
- 验收口径

不得用 `stateful standalone` 的商业化与治理能力为 `stateless runtime` 背书。

### 4.4 结果格式

审计结果建议至少落为以下文档：

- `docs/review/step-01-现状清单-YYYY-MM-DD.md`
- `docs/review/step-01-差距矩阵-YYYY-MM-DD.md`
- `docs/review/step-01-高风险清单-YYYY-MM-DD.md`
- `docs/review/step-01-stateful-stateless分离结论-YYYY-MM-DD.md`

## 5. 实施落地规划

### 5.1 目录级基线盘点

逐项盘点以下目录并冻结事实：

- `services/`
- `apps/`
- `crates/`
- `bin/`
- `scripts/dev/`
- `docs/api/`
- `docs/架构/`

### 5.2 服务与运行路径盘点

明确以下内容：

- 服务职责、启动方式、组合关系
- 本地开发模式、脚本模式、一体化宿主模式
- 端口、入口与宿主差异

### 5.3 协议与能力盘点

明确以下内容：

- OpenAI / Anthropic / Gemini 兼容面当前真实覆盖范围
- 文本、图像、音频、视频、文件、向量、批处理、评测、实时与编程代理相关能力的当前状态
- 已落地能力与目标能力的边界

### 5.4 模块边界与高风险盘点

重点审计：

- 超大文件
- 依赖方向不清的 crate
- 服务中混杂业务、接口、存储、Provider 逻辑的模块
- 前端工作区中与后端契约边界不清的页面或组件

### 5.5 结果归档与优先级排序

最终形成：

- P0：必须先处理，否则无法进入主链路改造
- P1：必须在本轮重构中关闭
- P2：可并行推进
- P3：可在后续阶段继续打磨

## 6. 测试计划

本 step 重点不是功能测试，而是“基线真实性测试”，建议执行：

- 各核心服务启动 smoke
- `bin/start-dev.*`、`bin/start.*`、`bin/stop.*` 的可用性验证
- `scripts/dev/start-workspace.*`、`start-stack.mjs` 等入口核对
- 架构文档与仓库目录的交叉验证

## 7. 结果验证

本 step 完成时，必须能回答：

- 当前应用到底已经具备什么
- 哪些只是目标，不是现实
- 哪些差距必须先处理
- 后续 step 顺序为什么这样安排

## 9. 风险与回滚

### 9.1 风险

- 若事实盘点不准确，后续改造会建立在错误前提上
- 若不区分 `stateful` 与 `stateless`，会导致验收和交付口径失真

### 9.2 回滚

如审计结果与既有文档冲突，应先更新基线文档与审计结论，再决定是否调整后续 step。

## 10. 完成定义

满足以下条件视为完成：

- 当前基线已冻结
- 差距矩阵已形成
- 高风险项已按优先级排序
- 后续重构顺序已有客观依据
